# Sub-agent 回复暂存

## G：核心玩法质询与改良

状态：已返回，待主线程验收。

可重点采纳的方向：

- 核心循环：在会被神志污染的九州行旅图上，用车轴、粮草、神志做路线与事件的代价选择。
- 三资源要有行为差异：车轴影响路线可达，粮草影响队伍与休整压力，神志影响世界可信度与隐藏收益。
- 假动态横屏长图应是“可读的风险仪表盘”，不是装饰背景。
- 10 分钟 Demo 应包含 6-8 个地点、12-18 条路线、15-25 个事件、至少一次濒危补救和可复盘失败。

## T：横向旅途舞台技术评估

状态：已返回，待主线程验收。

可重点采纳的方向：

- 推荐 `React + SVG + CSS`，Canvas 只作为可选氛围层。
- 不建议第一版使用 Three.js、Pixi、Phaser、Godot 或复杂游戏引擎。
- 路线映射为可播放横向段，底层仍是节点网络。
- 1 天可做静态舞台，3 天可做可演示假动态，1 周可做 Web Demo 视觉骨架。

## Cmp：同类产品调研

状态：已返回，待主线程验收。

可重点采纳的方向：

- 最稳借鉴组合不是像某一款游戏，而是：`Out There / Oregon Trail` 的路线资源压力，`Roadwarden` 的文字状态界面，`Darkest Dungeon` 的横向推进戏剧感，`Sunless Sea` 的认知/恐惧资源。
- Web Demo 第一版应保持节点路线、三资源、短事件、结果日志、轻量横向旅途舞台。
- 横向旅途舞台只做结果表达：路线选择后车队剪影推进、背景滚动、天色和神志变化，然后进入文字事件。
- 神志可以影响可读性，但资源数字、按钮位置和结果日志必须稳定。
- Demo 应强调短局可复盘，失败原因要能被玩家讲清楚。
- 竞品借鉴必须避免宣传表述依附，例如不要写“山海经版 Out There”或“Darkest Dungeon 式山海经”。
