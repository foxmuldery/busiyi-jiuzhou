# 《不思异：九州》产品架构与实现评估

> 暂定名：《不思异：九州》  
> 类型：山海经题材远行生存叙事 Roguelike  
> 核心参考：节点探索、资源压力、随机事件、失败重开、未知语言与世界理解  
> 核心差异：从太空迷航转译为中国古代神秘地理远行，从科幻生存转译为山海经式认知污染与九州游记

## 1. 产品一句话

《不思异：九州》是一款以《山海经》为灵感的远行生存叙事游戏。玩家率领一支古代行旅队伍，从中原出发，穿越九州、大荒、昆仑、归墟等神秘地理区域，在车轴、粮草、神志三项核心资源的压力下，记录异兽、辨认巫文、与异族交流，并逐渐发现这个世界并非凡人能够完整理解。

游戏的核心体验不是战斗，而是：

**一个凡人队伍用有限的器物、粮草和心智，强行穿过一个越来越不像人间的世界。**

## 2. 核心定位

### 2.1 目标体验

- 远行感：玩家始终在路上，下一站既是希望也是风险。
- 生存感：每一次移动和选择都会消耗关键资源。
- 神秘感：世界不是被讲明白的，而是被玩家一点点误读、理解和怀疑。
- 恐怖感：神志越低，文字、地图和选项越不可靠。
- 重开欲：失败不是简单结束，而是一段有记忆点的荒外游记。

### 2.2 不做什么

- 不做传统打怪升级 RPG。
- 不把《山海经》做成普通怪物图鉴。
- 不做大开放世界。
- 不在早期堆复杂背包、复杂战斗、角色养成。
- 不复制参考游戏的 UI、文案、剧情、视觉表达和具体流程。

### 2.3 产品关键词

山海经、九州、远行、生存、节点地图、文字事件、异兽、巫文、异族语言、认知污染、克苏鲁式未知、古籍感、失败叙事。

## 3. 核心玩法循环

玩家每一局的基本循环如下：

1. 查看当前地点、资源、队伍状态和可前往路线。
2. 选择下一个地点。
3. 消耗车轴与粮草进行移动。
4. 抵达地点后触发事件。
5. 根据事件选择行动：绕行、祭祀、交流、采集、修复、冒险、阅读碑文等。
6. 获得资源、器物、语言理解、地图线索，或损失车轴、粮草、神志、随从。
7. 神志状态影响后续文本、地图和选项可信度。
8. 继续深入，直到抵达结局、死亡、困死、疯癫或归返。

核心循环可以概括为：

```text
选路 -> 消耗 -> 遇事 -> 抉择 -> 变化 -> 更深入
```

### 3.1 真实产品本质

这个项目可以降维理解为：

**文字选择游戏 + 地点节点图 + 资源状态机 + 横向旅途舞台。**

地图导航不需要一开始做成复杂开放世界。它的核心作用只有三件事：

1. 告诉玩家现在在哪里。
2. 告诉玩家能去哪里。
3. 告诉玩家去那里要付出什么代价。

因此，最小实现可以非常轻：

```text
当前地点 -> 可选路线 -> 消耗资源 -> 进入新地点 -> 触发事件 -> 选择结果 -> 更新状态
```

地图不是为了让玩家“走路”，而是为了把文字选择变成空间选择。

表现层可以借鉴横屏旅途游戏：车队或辇车沿一卷山海图从左向右推进，前景、后景分层滚动，出现日出日落、白云漂移、山影、河渡、墟市、祭所和禁地。但这不是横版动作游戏，不需要碰撞、物理、跳跃、战斗操作或实时关卡逻辑。底层仍然是节点、路线、事件和资源。

例如同样前往“赤水”，路线设计可以直接制造策略压力：

```text
北路：车轴 -8，粮草 -2，低风险
南路：车轴 -3，粮草 -5，可能遇见羽民
夜路：车轴 -1，神志 -12，可能看见隐藏路线
```

第一版可以采用“横向山海行程图”：节点仍然离散排列，只是视觉上像一段正在展开的远行画卷。车队图标停在当前节点，玩家选择下一段路线后，画面移动到下一个地点，再触发文字事件。

更准确地说，它应该是“假动态横屏长图”：不是每个地点只换一张静态图，也不是完整横版关卡，而是一段可以演示短途行进的旅途舞台。前景的车队、路面、草木与后景的山影、云气、日月、异象分层移动，让玩家感觉自己正在穿越一小段九州道路。这个方向可借鉴《暗黑地牢》式横向推进的戏剧呈现，但不复制其美术、UI、战斗结构或具体镜头。

## 4. 三项核心消耗元素

### 4.0 命名口径

正式规则名采用：

```text
车轴、粮草、神志
```

正式 UI 短名采用：

```text
车、粮、神
```

完整系统含义为：

| UI 短名 | 规则名 | 含义 |
|---|---|---|
| 车 | 车轴 / 车力 | 还能不能继续远行 |
| 粮 | 粮草 | 队伍还能不能撑下去 |
| 神 | 神志 | 所见世界还可不可信 |

文档和规则说明中使用“车轴、粮草、神志”解释机制；游戏界面、路线代价、资源栏优先显示“车、粮、神”，例如：

```text
夜渡赤水：车 -1 / 粮 -1 / 神 -12
```

### 4.1 车轴（UI 短名：车）

对应原型中的“燃料”。

车轴代表远行工具的通行力，包括车轴、车轮、舟板、绳索、桨、辇架、牲口状态等。它不是普通耐久，而是玩家继续前往下一个节点的能力。

设计作用：

- 每次移动都会消耗车轴。
- 山地、流沙、沼泽、暴雨、强渡、异兽袭击会额外消耗。
- 车轴归零时，玩家不能继续远行，只能在当前地点等待、修复、拆器、采集或触发困境事件。
- 高级器物可以降低特定地貌的车轴消耗。

玩家心理：

**车轴低时，玩家怕的是走不出去。**

### 4.2 粮草（UI 短名：粮）

对应原型中的“氧气”。

粮草代表队伍生存的基础倒计时，包括粮食、盐、干肉、草料、药、薪柴等。它不是单纯的食物数量，而是“队伍还能支撑几天”的综合补给。

设计作用：

- 每到一个地点、每经过一日都会消耗粮草。
- 绕路、休整、等待、穿越严酷地貌会增加消耗。
- 粮草归零时进入饥荒状态：随从逃散、疾病、事件选项减少、神志持续下降。
- 某些事件可以通过采集、交易、狩猎、祭祀或牺牲换粮草。

玩家心理：

**粮草低时，玩家怕的是撑不过今天。**

### 4.3 神志（UI 短名：神）

对应原型中的“耐久”，但从物理耐久转译为认知完整性。

神志代表玩家和队伍在神怪世界中的认知稳定。见异兽、读巫文、进入神域、夜宿鬼国、听见神谕、遭遇不可名状的地理异常，都会损耗神志。

设计作用：

- 神志越低，地点描述越恐怖、越主观、越疯狂。
- 神志越低，选项越模糊，结果提示越不可靠。
- 神志低时会出现幻路、幻人、重复地点、错误地图。
- 神志低不是纯惩罚，也可能解锁隐藏真相、神文、异兽真名和特殊路线。
- 神志归零时不是普通死亡，而是认知崩坏：玩家可能迷失、成为荒外之人，或进入特殊结局。

玩家心理：

**神志低时，玩家怕的是自己看到的已经不是真的。**

## 5. 神志文本系统

这是本项目最重要的差异化机制。

同一个地点、事件、选项，在不同神志区间下显示不同文本。不是简单换皮，而是让游戏逐渐背叛玩家的感官。

### 5.1 三层文本状态

| 神志状态 | 文本气质 | 玩家感受 |
|---|---|---|
| 高神志 | 地理志 | 冷静、古雅、可判断 |
| 中神志 | 异闻录 | 征兆增多、主观感增强 |
| 低神志 | 疯人书 | 语义断裂、世界开始说话 |

### 5.2 示例

高神志：

> 前方有泽，水色赤而无波。泽畔多白骨，骨上生细草。随从说此地不可夜渡。

中神志：

> 赤泽无风，却有水声从车底传来。白骨上的草向你低伏，像在听你腹中饥响。随从不再说话，只看着天色。

低神志：

> 水不是水。骨不是骨。车轴转了一圈，又转回昨日。草在你牙缝里生长。有人借你的喉咙说：夜渡，夜渡，夜渡。

### 5.3 机制价值

- 把数值状态变成叙事体验。
- 让恐怖感从系统内部自然长出来。
- 提供高风险高收益：低神志可能看到假的路，也可能看到真正的路。
- 增强重玩价值：同一事件在不同状态下会像另一个故事。

## 6. 世界与地图架构

### 6.1 地图形式

采用节点网络地图，不采用大开放世界。

每个节点代表一个地点，节点之间是路线。路线本身也有地貌、消耗和风险。

地图的视觉表现可以横向化。玩家看到的是从左到右展开的九州旅途，但系统内部仍然保存为节点网络。这样既保留策略选择，又获得更强的远行感。

示意：

```text
中原驿
  |
雷泽 ---- 青丘 ---- 赤水 ---- 昆仑丘
  |          |         |
黑齿国 ---- 羽民泽 ---- 大荒北 ---- 归墟边
```

### 6.2 地点类型

- 人间边境：中原驿、旧都、废关、渡口。
- 山川险地：雷泽、流沙、弱水、赤水、不周山遗址。
- 异族之国：青丘、羽民国、黑齿国、长臂国、贯胸国。
- 神话地标：昆仑丘、汤谷扶桑、归墟、建木、幽都。
- 异常地点：重复驿站、第二个昆仑、无名归路、梦中泽。

### 6.3 地点数据

每个地点建议包含：

- 地点 ID
- 地点名称
- 所属区域
- 地貌标签
- 危险等级
- 可产出资源
- 可触发事件池
- 关联语言或文明
- 隐藏路线条件
- 神志影响权重

示例：

```json
{
  "id": "red_marsh",
  "name": "赤水",
  "region": "大荒西",
  "terrain": ["泽", "异水", "禁地"],
  "danger": 4,
  "resources": {
    "grain": -2,
    "axle": -4,
    "sanity": -6
  },
  "eventPool": ["red_marsh_warning", "red_marsh_night_crossing"],
  "languages": ["羽民语", "巫咸古辞"],
  "hiddenRoutes": ["sunken_route"]
}
```

## 7. 事件系统

事件系统是内容主引擎。

### 7.1 事件组成

每个事件由以下部分组成：

- 触发条件
- 文本变体
- 可见选项
- 选项条件
- 消耗与收益
- 后续状态变化
- 解锁内容

示例：

```json
{
  "id": "red_marsh_001",
  "location": "red_marsh",
  "conditions": {
    "sanityMax": 60
  },
  "texts": {
    "clear": "前方有泽，水色赤而无波。泽畔多白骨。",
    "uneasy": "赤泽无风，却有水声从车底传来。",
    "mad": "水不是水。车轴转了一圈，又转回昨日。"
  },
  "choices": [
    {
      "id": "detour",
      "label": "绕行三日",
      "cost": {
        "grain": -6,
        "axle": -3
      },
      "effects": {
        "routeSafety": 1
      }
    },
    {
      "id": "night_cross",
      "label": "夜渡",
      "cost": {
        "axle": -8,
        "sanity": -12
      },
      "effects": {
        "unlockRoutes": ["sunken_route"],
        "flags": ["heard_red_water"]
      }
    }
  ]
}
```

### 7.2 事件类型

- 地貌事件：流沙、泽地、暴雨、雪山、海雾。
- 异兽事件：遇见、追踪、献祭、避让、误食、取骨。
- 异族事件：交易、误解、语言学习、禁忌、结盟。
- 神祇事件：神谕、梦境、祭祀、惩罚、赐物。
- 遗迹事件：碑文、古车、断桥、祭坛、废城。
- 队伍事件：随从争执、疾病、逃散、信仰动摇、疯狂。
- 幻觉事件：假归路、重复地点、死者来信、地图自改。

### 7.3 事件设计原则

- 每个事件都应该有明确资源意义。
- 每个事件都要有叙事记忆点。
- 选择不应该只是正确/错误，而是不同代价。
- 低神志事件要危险，但不能只是随机坑玩家。
- 同一事件最好能通过器物、语言、过往线索改变解法。

## 8. 器物系统

器物替代科幻飞船设备，是玩家构筑路线和解法的主要方式。

### 8.1 器物定位

器物不是单纯加属性，而是改变玩家与世界互动的方式。

### 8.2 示例器物

| 器物 | 作用 |
|---|---|
| 辟邪铃 | 降低妖魅类事件的神志损耗 |
| 禹步履 | 涉水路线车轴消耗减少 |
| 鲛人灯 | 海域和水下事件显示隐藏选项 |
| 河图残片 | 在地图上显示一条真实隐藏路线 |
| 息壤囊 | 可修复车轴，也可能触发天罚 |
| 夸父杖 | 长距离移动消耗降低，但粮草消耗增加 |
| 青鸟羽 | 可跳过一段危险路线，但会损耗神志 |
| 巫咸骨简 | 提升巫文理解，增加神谕事件风险 |

### 8.3 器物槽位

MVP 阶段建议设置 3 到 5 个器物槽位，避免背包复杂化。

器物分类：

- 行路器：影响车轴和路线。
- 生存器：影响粮草和队伍状态。
- 通神器：影响神志、巫文和神祇事件。
- 异兽器：影响异兽遭遇。
- 地图器：影响隐藏路线和地点可见性。

## 9. 异族语言与巫文系统

这是对“未知文明语言”机制的山海经转译。

### 9.1 语言类型

- 羽民语
- 黑齿语
- 青丘誓辞
- 巫咸古辞
- 昆仑神文
- 归墟潮文

### 9.2 机制方式

初见异族或古碑时，玩家只能看到残缺信息。多次交流、获得器物、完成事件后，理解度提升，文本逐渐清晰。

未理解：

> 羽民长者指向西方：“赤……水……不可……日落……”

部分理解：

> 羽民长者警告你：赤水在日落后会发生变化。

完全理解：

> 羽民长者警告你：赤水在日落后会倒流，舟行必覆，唯有献白羽者可借逆流入地下河。

### 9.3 设计价值

- 增强未知感。
- 让重复遭遇有成长。
- 让玩家用“理解世界”替代传统升级。
- 为隐藏路线和多结局提供自然入口。

## 10. 结局系统

结局不应只有成功和死亡，而应该像不同版本的远行传说。

### 10.1 主要结局方向

| 结局 | 条件方向 | 气质 |
|---|---|---|
| 归中原 | 带回足够地图线索并成功返程 | 人间胜利 |
| 登昆仑 | 找到昆仑路线并通过神祇考验 | 神话揭示 |
| 入归墟 | 低神志或特殊器物打开归墟路线 | 宇宙恐怖 |
| 补天 | 收集五色石相关线索与材料 | 英雄神话 |
| 化外 | 放弃王命，与荒外诸国结盟 | 价值转向 |
| 失心 | 神志崩坏但完成记录 | 疯人归来 |
| 困荒 | 车轴耗尽且无法修复 | 生存失败 |
| 饥亡 | 粮草尽且队伍崩散 | 生存失败 |

### 10.2 结局原则

- 失败也要有文本记忆点。
- 胜利不一定解释全部真相。
- 低神志结局不一定是坏结局，但一定要有代价。
- 不同结局可以解锁新开局背景、器物或地图残片。

## 11. MVP 范围

第一版不要做完整九州，而要做一个可验证核心乐趣的纵切片。

### 11.1 MVP 内容规模

- 1 张节点地图，表现为横向山海旅途舞台
- 20 个地点
- 80 到 120 个事件
- 3 个核心资源：车轴、粮草、神志，UI 短名为车、粮、神
- 10 个器物
- 3 层神志文本
- 2 种语言或巫文
- 3 个主要结局
- 单局时长 20 到 40 分钟

### 11.2 MVP 必须验证的问题

1. 玩家是否能在 10 分钟内理解车轴、粮草、神志的区别。
2. 玩家第一次神志降低时，是否明显感觉世界变了。
3. 玩家失败后是否愿意立刻重开。
4. 玩家是否会主动复述自己遇到的事件。
5. 玩家是否觉得“我还没看懂这个世界”。

### 11.3 MVP 不做内容

- 不做复杂战斗。
- 不做大量角色立绘。
- 不做完整九州。
- 不做多人。
- 不做大型任务链。
- 不做复杂经济系统。

## 12. 技术实现建议

### 12.1 推荐路线

建议先用 Web 技术做可玩原型，再根据验证结果决定是否迁移到 Godot 或继续 Web 打包。

| 路线 | 适合度 | 说明 |
|---|---:|---|
| Web / React | 高 | 数据驱动快，适合事件编辑和快速迭代 |
| Godot | 高 | 适合独立游戏发行，2D UI 和数据系统够用 |
| Unity | 中 | 成熟但偏重，文字节点游戏早期成本偏高 |
| Ren'Py | 中 | 文本强，但地图、资源、Roguelike 系统要扩展 |
| 原生 App | 低 | 早期没有必要 |

### 12.2 轻量实现路线

最稳的原型路线是：

```text
自研核心游戏逻辑
+ React / TypeScript
+ JSON 事件数据
+ 节点逻辑驱动的横向旅途舞台
+ 本地存档
+ 可选 inkjs 处理复杂分支
```

第一版不必急着上完整游戏引擎，也不必先接复杂叙事工具。核心代码可以只围绕四个对象：

- `Location`：地点。
- `Route`：路线和消耗。
- `Event`：事件文本、选项和结果。
- `GameState`：资源、器物、标记、语言进度、历史路线。

这样做的好处是：

- 技术复杂度低。
- 数据结构清楚。
- 后期容易做事件编辑器。
- 不被第三方叙事工具的表达方式锁死。
- 商业风险更低，因为核心系统是自研。

### 12.3 可复用开源组件建议

目前找到的开源项目更适合作为工具层，而不是直接作为游戏底座。

| 组件/项目 | 建议 | 用途 |
|---|---|---|
| React / TypeScript | 推荐 | 主界面和状态管理 |
| xyflow / React Flow | 可选推荐 | 山海图节点地图、路线连线、缩放拖动 |
| inkjs | 可选推荐 | 复杂分支文本、变量、选择逻辑 |
| Yarn Spinner | 备选 | 如果后续转 Unity/Godot，可考虑做对话系统 |
| rot.js | 备选 | 随机数、调度、Roguelike 工具；不是核心需求 |
| A Dark Room | 只研究，不直接改 | 学习文字生存节奏和极简资源反馈 |
| Twine | 只适合纸面/早期原型 | 快速试文字，但正式版本扩展受限 |
| ChoiceScript | 不建议 | 商业许可证限制明显 |

推荐原则：

**用宽松许可证组件解决显示和工具问题，用自研逻辑解决核心玩法。**

### 12.5 页面结构

MVP 可以只有一个主界面：

- 上方或左侧：横向山海行程图
- 中间：事件文本与选项
- 右侧：资源、器物、语言理解、队伍状态
- 底部：行旅日志

横向行程图可包含轻量氛围层：远景山影、云气、昼夜色调、前景草木或车辙。第一版只做视觉反馈，不引入碰撞系统、物理系统或实时移动玩法。

如果做移动端，优先用分页结构：

- 地图
- 当前事件
- 行囊状态
- 日志

### 12.6 存档

MVP 用本地存档即可：

- 当前局存档
- 历史结局记录
- 已见事件记录
- 已解锁图鉴或词条

正式版再考虑云存档。

## 13. 数据驱动内容生产

这个项目的长期瓶颈是内容，不是代码。因此从一开始就应该数据驱动。

### 13.1 内容表

至少需要以下数据表：

- 地点表
- 路线表
- 事件表
- 选项表
- 器物表
- 语言表
- 结局表
- 词条表

### 13.2 事件编辑器

第二阶段建议做一个简单事件编辑器，哪怕只是内部工具。

编辑器能力：

- 创建事件
- 填写三层神志文本
- 设置触发条件
- 设置选项与结果
- 预览不同神志状态下的显示
- 检查事件是否缺少文本或结果

### 13.3 内容写作规则

- 每个地点至少有 3 到 5 个事件。
- 每个关键事件至少写高神志和低神志两个版本。
- 低神志文本可以疯狂，但要保留可玩信息。
- 古籍感不要变成文言文阅读障碍。
- 恐怖感来自异样细节，不靠堆血腥。

## 14. 美术与声音方向

### 14.1 美术方向

建议基调：

- 帛书地图
- 竹简文字
- 龟甲裂纹
- 朱砂标记
- 古地图线条
- 暗色墨迹
- 局部异色发光

避免：

- 纯仙侠亮色
- 普通卡牌怪物图鉴
- 过度写实恐怖
- 大量蓝紫科幻感
- 直接模仿参考游戏界面

### 14.2 UI 气质

UI 应该像“正在被污染的古代行旅记录”。

高神志时，界面整洁、线条稳定、文字清楚。  
低神志时，地图墨线轻微错位，部分地点名称变形，日志出现重复句或异文。

### 14.3 声音方向

- 低频风声
- 木轮声
- 远处鼓声
- 水下人声
- 竹简摩擦
- 铜铃
- 祭祀吟唱

声音不宜太满，要保留荒野的空。

## 15. 难度评估

整体难度：中等偏高。

主要难点不在基础技术，而在内容生产、数值平衡和叙事系统稳定性。

| 模块 | 难度 | 风险 |
|---|---:|---|
| 节点地图 | 低 | 容易实现，但需要路线节奏 |
| 横向旅途舞台 | 低到中 | 只做分层滚动和昼夜云影时可控，不能升级成横版动作系统 |
| 三资源系统 | 低 | 数值平衡需要反复测试 |
| 事件系统 | 中 | 需要可扩展、可维护 |
| 神志文本系统 | 中高 | 内容量会翻倍，质量要求高 |
| 异族语言系统 | 中 | 设计不好会变成装饰 |
| 器物系统 | 中 | 容易变成普通加减数值 |
| 美术 UI | 中 | 风格必须强，但成本要控 |
| 长线内容 | 高 | 最吃创作力 |
| 商业完成度 | 高 | 需要大量打磨体验节奏 |

## 16. 商业评估

### 16.1 优势

- 题材差异化强：山海经 + 生存 Roguelike + 认知污染。
- 美术成本可控：不依赖大量动画和 3D 资产。
- 适合独立游戏体量。
- 事件系统便于长期扩展。
- 容易形成玩家分享：每局都是一段独特游记。
- 适合 Steam、TapTap、移动端测试。

### 16.2 风险

- 文字量较大，可能劝退部分玩家。
- 如果随机性过强，会被认为不公平。
- 如果神志系统只做成惩罚，会失去策略深度。
- 如果山海经只停留在怪物名词，会显得浅。
- UI 如果不够顺，会像网页原型而不是游戏。
- 文案质量直接决定上限。

### 16.3 目标用户

- 喜欢独立叙事游戏的玩家。
- 喜欢 Roguelike 生存压力的玩家。
- 喜欢《山海经》、神话、古籍、民俗、克苏鲁式未知的玩家。
- 喜欢阅读事件和做选择的玩家。

不优先服务：

- 期待强战斗操作的玩家。
- 期待大量角色养成的玩家。
- 只想快速刷数值奖励的玩家。

## 17. 版权与商业风险边界

### 17.1 可以借鉴的层面

一般来说，抽象玩法机制和类型结构不是独家版权，例如：

- 三资源压力。
- 节点探索。
- 随机事件。
- 失败重开。
- 生存路线选择。
- 未知语言逐渐理解。
- 单局旅程式结构。

这些属于玩法思想和类型方法，商业上通常可以借鉴。

### 17.2 不应该复制的层面

必须避免复制参考游戏的具体表达：

- UI 布局和视觉风格。
- 图标、字体、动效。
- 具体事件文案。
- 具体剧情、角色、科技名。
- 商店描述和宣传语。
- 代码、音效、美术素材。
- 高度相似的流程和结局结构。

### 17.3 本项目的避险方式

- 把太空星图转译为山海图和九州节点。
- 把飞船设备转译为古器、车舆、祭器、巫物。
- 把外星文明转译为异族、巫文和神祇。
- 把科幻孤独转译为古代神秘地理和认知污染。
- 建立独立的文本系统、视觉系统和结局系统。

产品原则：

**借生存压力结构，不借具体表达。**

### 17.4 开源代码商业风险判断

开源项目的商业风险主要来自许可证和相似表达两个方面。

| 项目 | 风险 | 判断 |
|---|---:|---|
| xyflow / React Flow | 低 | MIT，可商用，保留许可证即可 |
| inkjs | 低 | MIT，可商用，适合 Web 叙事系统 |
| Yarn Spinner | 低 | MIT，可商用，更适合 Unity/Godot 对话 |
| rot.js | 低 | BSD-3-Clause，可商用，适合工具层 |
| A Dark Room | 中 | MPL-2.0 可商用，但有文件级开源义务；更重要的是直接改容易变成换皮 |
| Twine | 中 | 可用于原型，但工具和故事格式许可证需要分别确认；正式产品不建议强依赖 |
| ChoiceScript | 高 | 许可证限制商业用途，不建议用于商业项目 |
| 许可证不明的小仓库 | 高 | 不作为商业基础 |

商业版建议：

- 不直接 fork 完整开源游戏做换皮。
- 不复制任何开源游戏的文案、UI、节奏表达和素材。
- 只使用许可证明确、适合商业用途的库。
- 在项目中保留第三方库许可证清单。
- 核心玩法代码、事件文本、山海经世界观表达全部自研。

最终技术组合建议：

```text
自研事件引擎
+ 自研三资源系统
+ 自研随机与平衡系统
+ MIT/BSD 工具库
+ 原创文本和美术
```

## 19. 成功指标

### 19.1 原型阶段指标

- 70% 以上测试玩家能说清车轴、粮草、神志的区别。
- 玩家第一次神志文本变化时能察觉到体验变化。
- 单局失败后，至少一半玩家愿意重开。
- 玩家能复述至少一个事件。
- 玩家能提出“我想知道某个地点/异族/神祇后面是什么”。

### 19.2 Demo 阶段指标

- 平均游玩时长超过 25 分钟。
- 单人平均重开次数超过 2 次。
- 玩家对低神志状态有明确讨论。
- 玩家不认为失败完全随机。
- 玩家能感受到山海经题材不是贴皮。

### 19.3 商业阶段指标

- 愿望单转化。
- Demo 完成率。
- 玩家评价关键词中出现“神秘”“想再来一局”“文本有味道”“山海经”“恐怖”。
- 玩家主动分享自己的死亡或结局故事。

## 20. 最大设计风险与解决方案

### 20.1 风险：三资源太像普通数值

解决：

- 车轴只管移动能力。
- 粮草只管生存倒计时。
- 神志改变文本、选项和地图。

### 20.2 风险：低神志让玩家觉得被坑

解决：

- 低神志也要给隐藏收益。
- 提供可判断线索，不做纯随机惩罚。
- 让玩家能通过器物和经验降低风险。

### 20.3 风险：山海经变成素材堆砌

解决：

- 每个地点必须有地理规则和生存意义。
- 异兽不只是怪物，而是事件、禁忌、资源和路线条件。
- 文本保持古籍感，但不堆难懂名词。

### 20.4 风险：内容量拖垮开发

解决：

- 先做 20 个地点的纵切片。
- 所有事件数据化。
- 尽早做事件编辑器和缺漏检查。
- 把事件分层：普通事件、关键事件、结局事件。

### 20.5 风险：随机生成造成死局

核心原则：

**随机生成世界，不能随机生成死局。**

解决：

- 地图生成先保证一条或几条理论可通关主路。
- 每 3 到 5 步必须出现一个喘息点，例如补给粮草、修车、交易、休整或低风险事件。
- 事件池分级抽取，不做纯随机抽取。
- 玩家资源过低时，系统暗中提高补救事件权重。
- 资源归零时先进入濒死状态，而不是立刻失败。
- 关键路线不能只依赖单次随机事件。

推荐地图生成顺序：

```text
生成保底主路 -> 放置补给点 -> 放置风险点 -> 放置关键线索 -> 生成支路 -> 加入隐藏路线
```

保底主路不是让玩家必胜，而是保证玩家如果判断正确、资源管理合理，就有机会抵达结局。

### 20.6 风险：随机性让玩家觉得不公平

解决：

- 维护隐藏的 `badLuckMeter`，连续坏结果后提高补救事件概率。
- 避免连续多个节点只扣资源不给选择。
- 每次重大失败前给玩家最后抉择。
- 死亡后明确显示主要失败原因。
- 让失败可复盘，例如“过早夜渡”“粮草耗过高”“低神志误入幻路”。

资源濒死状态示例：

车轴归零：

- 拆器物修车。
- 舍弃行囊减负。
- 徒步前往最近地点，神志大幅下降。
- 等待三日，触发救援或灾厄事件。

粮草归零：

- 吃掉祭品。
- 宰杀牲口。
- 向异族求粮草。
- 用神志换取梦中粮草。
- 舍弃随从或器物换取前进机会。

神志归零：

- 服丹砂强行清醒。
- 让巫祝代你记事。
- 接受幻路，进入特殊线。
- 放弃地图，原路逃返。

### 20.7 平衡验证方法

不要只靠手感调平衡，应该做自动模拟器。

模拟器至少跑五类玩家：

- 保守玩家：优先补资源。
- 冒险玩家：优先探索新地点。
- 贪心玩家：优先拿器物和隐藏奖励。
- 归家玩家：资源低就回撤。
- 随机玩家：乱选。

每轮平衡测试跑 1000 到 10000 局，记录：

- 平均存活步数。
- 第一次死亡原因。
- 通关率。
- 三资源归零次数。
- 死局出现率。
- 连续坏事件次数。
- 玩家是否连续 3 次以上只能选择坏选项。

建议目标值：

| 玩家阶段 | 合理通关率 |
|---|---:|
| 新手前 3 局 | 5%-15% |
| 理解规则后 | 25%-40% |
| 熟练玩家 | 50%-70% |
| 真结局/隐藏结局 | 5%-15% |

理想失败感应该是：

> 我知道我哪里贪了，下次可以换路线。

而不是：

> 这局系统没给活路。

## 21. 下一步建议

最合理的下一步不是立刻做完整游戏，而是做一个小而尖的可玩原型。

建议下一份文档拆成：

1. 核心规则说明书。
2. 第一张地图设计。
3. 20 个地点列表。
4. 30 个首批事件样稿。
5. Web Demo 技术任务拆分。
6. 随机地图与事件抽取规则。
7. 自动平衡模拟器指标。

其中最重要的是先写 30 个事件样稿。只要这些事件有味道，这个项目就站得住；如果事件不好，系统再完整也不会成立。
