## 2. 数据引用验收

必须检查：

1. `locations` 每个地点都有 `id / name / region / terrain / event / map / detail / supplies`。
2. `events` 覆盖每个地点的 `event`。
3. `routes` 的 `from / to` 都指向真实地点。
4. `route.midEvent` 指向真实 `routeEvents`。
5. `revealPlan` 的每条 `routes / fogRoutes / fogLocations` 都真实存在。
6. `fogLabels` 只指向真实地点。
7. 新增地点至少有一个入口路线；非终点地点至少有一个出口路线或结局选择。
8. `supplies` 的 `id` 在同地点内不重复，且每条都有 `label / hint / effect / result`。

## 3. 玩法闭环验收

必须能完成：

1. 重开后，开局只显示当前地点、可走路线和少量雾中轮廓，不全图暴露。
2. 从中原驿至少有两条初始选择。
3. 路线卡显示去向、消耗、风险、半途事件和目的地资源倾向。
4. 选择路线后进入横向行走；若有半途事件，先停下处理。
5. 抵达新地点后，地点志更新、补给区更新、地图显影更新。
6. 已走路线显示为车辙；未显影路线不可点击。
7. 至少存在一条从中原驿抵达九州裂隙的可行测试路径。
8. 车轴、粮草、神志任一归零先进入濒死补救，不应直接无解释失败。

## 4. 补给验收

首版必须达到：

1. 各地补给差异明显，不能每城都是粮草和车轴平均补。
2. 补给点击后资源真实变化。
3. 已用补给明确显示“已搜集”或等价状态。
4. 路线卡的目的地资源预览不应把已用补给误导为无限补给。

如 C 线程实现“每次抵达最多 1 次补给”，额外验收：

1. 同一地点同次抵达只能使用 1 个补给。
2. 玩家能看懂为什么其它补给暂不可用。
3. 重开后状态清空。

## 5. UI 验收

必须保留：

1. 横屏游戏框架。
2. 旅途 / 九州图切换。
3. 右侧行囊九宫格。
4. 音乐入口。
5. 主事件处理后进入“打开九州图”的下一步。

重点看小横屏：

1. 844x390 下资源条和主按钮不重叠。
2. 932x430 下路线卡文字不撑破容器。
3. 九州图节点不密到无法点击。

## 6. 风险验收

必须避免：

1. 深色星图、圆形星球节点、发光航线、科幻扫描。
2. 西部牛仔、枪、酒馆、马车木牌等直接西部符号。
3. Darkest Dungeon 式哥特英雄队列、职业头像、压力条、火把系统。
4. 现代《山海经》译文、百科段落、商业图鉴设定的直接改写。
