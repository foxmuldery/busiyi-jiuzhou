# D_BSI-D-004 双视图 UI 体验验收清单

> 子线程：D 体验与视觉结构  
> 项目：《不思异：九州》Web Demo  
> 日期：2026-06-14  
> 输入：`主协调_BSI-UX-004_九州图与路遇交接单.md`，当前 `GitHub资产区/03_WebDemo/prototype/` 静态原型  
> 边界：只做体验验收与 UI 收口建议，不修改 C 线程原型代码

## 1. 当前结构判断

当前双视图方向成立：`九州图页`负责战略选择，`地点页`负责停靠、横向行旅舞台、地点志、补给、事件与日志。这个拆法比把所有内容压进同一页更清楚，也更接近本轮目标里的两种体验来源：

- Out There 式战略感：玩家在出发前比较路线、消耗、风险、目的地资源和终点距离。
- 西部横向旅行体感：玩家在地点页看到车队沿路推进、半途停下、抵达城镇、处理停靠事件。

但当前原型的信息层级仍有一个风险：顶部资源、旅程总览、横向舞台、地点志、补给、事件、行囊、日志和九州图节点都已经存在，如果不收口，玩家可能会把它理解为“两个内容页都能做所有事”，而不是“九州图决定去哪，地点页处理路上发生了什么”。

第一版验收应以清楚为先，不以视觉丰富度为先。只要玩家能稳定回答五个问题，就算视觉收口通过：

1. 我现在在哪里。
2. 我离九州裂隙还有多远。
3. 下一站有什么资源。
4. 走这条路线会消耗什么。
5. 为什么车队半路停下。

## 2. 九州图页信息层级

九州图页的第一任务不是展示世界观，而是让玩家做一次有代价的路线选择。

推荐阅读顺序：

```text
当前位置 -> 终点方向/剩余距离 -> 可达下一站 -> 路线消耗与风险 -> 目的地可观察资源 -> 点击确认前进
```

当前原型已有可用基础：

- 顶部旅程总览显示 `西行 x / y` 与 `距九州裂隙约 x 段路`。
- 九州图节点有 current / reachable / locked / visited / future 状态。
- 路线卡显示出发地、目的地、消耗、风险、半途路遇、目的地可观察补给。
- 地图线条区分已走、可达、锁定、未知、半途 pending。

收口建议：

1. `九州图`页中，当前节点必须是第一视觉焦点，不能被可达节点、未来节点或背景网格抢走。
2. `距九州裂隙约 x 段路`必须在地图页和顶部总览同时出现，且文字一致。
3. 右侧`下一站`路线卡必须比地图节点承载更多决策信息。地图节点可以点，但路线卡才是完整解释。
4. 每张路线卡必须固定显示四行：去哪、消耗、风险/路遇、目的地可观察资源。
5. 锁定路线可以显示，但不能像可点击路线。锁定原因必须短到一眼能懂，例如“需旧令线索 / 需低神志幻路 / 需读出裂隙名”。
6. 未来地点可以有存在感，但不要提前暴露过多补给细节，避免玩家把未发现节点当作当前可选项。
7. 玩家点击可达节点或路线卡后，应立刻回到地点页进入横向行程反馈；不要留在地图页让玩家以为没有响应。

## 3. 地点页信息层级

地点页的第一任务是让玩家感觉“车队已经停在一个地方或一段路上”，并处理当前状态。

推荐阅读顺序：

```text
当前地点/半途状态 -> 横向舞台确认行旅体感 -> 当前事件或路遇 -> 选择结果 -> 地点志/补给 -> 回九州图继续选路
```

当前原型已有可用基础：

- 顶部标题显示当前地点，区域和日数常驻。
- 横向舞台有 from / route / to 三段节点标签，车队有 traveling / interrupted 状态。
- 地点页事件面板同时承载地点志、补给、事件文本和选择。
- 半途路遇通过 `pendingRoute` 接管事件面板，并禁止搜集城镇补给。
- 地点事件处理后出现“请切到九州图选择下一段旅程”的提示。

收口建议：

1. 地点页默认主焦点应是`当前事件/路遇`，地点志和补给是停靠辅助，不要比事件标题更抢。
2. 半途路遇状态必须显著区别于抵达城镇状态。标题、标签、舞台车队位置、补给禁用提示要共同说明“还没到下一站”。
3. 横向舞台只承担体感和状态确认，不承担复杂点击。玩家不应被诱导去点舞台背景。
4. `from -> route -> to` 三段标签应和路线卡一致，特别是半途时要显示“从哪里来、正走哪条路、目标去哪”。
5. 补给必须明确“一次性”。已搜集状态要保留在原位置，避免玩家以为按钮消失是 bug。
6. 地点事件处理后，主按钮区应只保留下一步提示，不要继续展示已无效选择。
7. 行囊、音频、动态层属于低优先级状态，不能挤压事件文本和行动按钮。

## 4. 双视图切换收口

双视图的玩家心理模型应固定为：

```text
地点页：我在这里发生了什么。
九州图：我接下来往哪里走。
```

验收时重点看是否形成这个闭环：

```text
地点页处理完地点事件/补给
-> 切到九州图
-> 比较下一站
-> 选择路线
-> 地点页播放/显示行程
-> 半途可能停下
-> 处理路遇
-> 抵达地点
-> 读地点志、搜补给、处理地点事件
```

需要避免的体验误解：

- 玩家以为`地点`和`九州图`只是两个同等的信息页，而不是流程的两段。
- 玩家以为地图页能处理事件，地点页能完整选路。
- 玩家以为半途路遇已经抵达了目的地，于是困惑为什么补给不可用。
- 玩家看到横向舞台移动后，不知道车队是否已经抵达。
- 玩家在地点事件已处理后，不知道下一步必须回到九州图。

## 5. C 线程可验证体验标准

以下标准应由 C 线程在实现完成后逐条验证，最好用 3-5 分钟核心流程跑通。

### 5.1 位置与终点

- 打开新局后，玩家在 3 秒内能看到当前地点是`中原驿`。
- 玩家不用打开日志，也能看到当前区域、日数、三资源。
- 玩家切到九州图后，能一眼识别当前节点，不需要通过路线卡反推。
- 玩家在任意地点都能看到离`九州裂隙`约几段路。
- 抵达九州裂隙时，距离文案从“距九州裂隙约 x 段路”变为明确的“已抵达九州裂隙”。

### 5.2 下一站与资源

- 每条可走路线都显示目的地名称。
- 每条可走路线都显示车轴、粮草、神志的消耗，即使某项为 0 也要让玩家能理解“这项不消耗”。
- 每条可走路线都显示目的地可观察补给，至少能回答“偏补车轴 / 偏补粮草 / 偏补神志 / 地利不明”。
- 玩家能在点击前判断“这条路会不会把某项资源推入危险区”。
- 已锁定路线必须说明锁定原因，不允许只显示“不可用”。

### 5.3 路线选择与战略感

- 九州图中至少同时呈现两条可比较路线时，玩家能比较风险、消耗、补给倾向，而不是只看路线名。
- 可达路线、锁定路线、已走路线、未知路线在视觉上有稳定区别。
- 地图节点点击和路线卡点击应触发同一套移动逻辑，不能产生不同结果。
- 选择路线后，资源变化、日数变化和日志记录必须同步出现。
- 如果路线有半途事件，路线卡应提前提示“半途路遇：事件名”，第一版可以是确定触发，不做概率伪装。

### 5.4 横向旅途体感

- 选择路线后，地点页必须给出出发反馈：车队推进、舞台状态变化、或明确的行程状态文字，三者至少有其一明显成立。
- 横向舞台的车队位置能区分“在地点停靠”“正在行进”“半途停下”三种状态。
- 舞台背景、云气、日夜、污染层可以增强气氛，但不能遮挡资源、事件文本、按钮和日志。
- 舞台内的 from / route / to 标签必须与当前 route 数据一致。
- 动态关闭时，玩家仍能通过文字和状态判断自己正在路上或停在半途。

### 5.5 半途路遇

- 触发路遇时，玩家必须看到“半途停下”的原因，而不只是突然出现一个新事件。
- 路遇标题、路线名、事件标签要同时出现，例如“半途黑云 / 车辙入泽 · 路遇”。
- 半途时城镇补给不可用，但提示必须解释为“车队停在半途，无法搜集城镇补给”。
- 路遇选择必须显示代价或收益，并在选择后写入日志。
- 处理路遇后，系统必须自动或明确引导抵达目的地，不能让玩家卡在半途状态。

### 5.6 地点停靠

- 抵达新地点后，地点标题、区域、地点志、见闻计数都要更新。
- 地点志负责世界观和停靠感，不承担路线决策。
- 补给按钮显示资源效果、一次性规则和已搜集状态。
- 地点事件已处理后，不再显示可重复点击的旧选择。
- 玩家处理完地点事件后，界面要明确提示回到九州图继续选路。

### 5.7 日志与复盘

- 每次路线选择、半途停下、路遇选择、抵达、补给、地点事件都应写入日志。
- 日志文字要能解释“刚才为什么资源变了”。
- 日志不应成为唯一信息来源；关键状态必须在主界面直接可见。
- 最近日志数量保持短，不要挤压当前事件。
- 失败、困厄、补救与重开要在日志里留下可复盘原因。

## 6. 视觉收口建议

### 6.1 九州图页

- 地图背景保持低对比，不要做成复杂山水画。第一版更适合“星图节点 + 卷轴质感”的混合，而不是写实大地图。
- 节点卡片尺寸要稳定，避免地点名、资源行换行后互相覆盖。
- 当前节点建议使用最强边框或光晕；可达节点次强；锁定和未来节点降权。
- 路线线条不要过粗，路线卡才是主要决策容器。
- 右侧路线列表要能独立完成决策；即使地图视觉失败，玩家仍能玩。

### 6.2 地点页

- 横向舞台应像“正在走的路”，不是风景封面。车队和路面要比远景更重要。
- 事件文本字号可以大于地点志，行动按钮紧跟事件文本。
- 地点志、补给和事件都在同一面板时，要用明确区块标题区分，不要让补给按钮夹在叙事正文中间造成阅读断裂。
- 半途状态可给事件面板加轻微强调，但不要用全屏警报感，避免误解成失败。
- 低神志污染只覆盖舞台，不覆盖 UI 面板。

### 6.3 移动端

- 移动端优先顺序应为：资源 -> 当前地点/距离 -> 当前事件或路线卡 -> 行动按钮。
- 九州图在移动端可以降级为路线列表 + 小地图，不必完整展示所有节点。
- 地点页舞台在移动端只做氛围缩略，不要压缩事件文本。
- 日志移动端默认只露最近一条或折叠。
- 所有路线卡和选择按钮的文字必须完整可读，不能靠 hover 说明。

## 7. 最小验收路径

C 线程实现完成后，建议主协调按以下路径验收：

1. 新开局停在`中原驿`，确认当前位置、三资源、距离、可往下一站都可见。
2. 切到`九州图`，比较`循旧王道西行`与`绕向废关`，确认消耗、风险、路遇、目的地补给可见。
3. 选择`循旧王道西行`，确认资源扣减、日数增加、地点页行程反馈出现。
4. 触发`轮声入骨`，确认玩家知道这是半途路遇，且补给不可用有解释。
5. 处理路遇后抵达`故王道`，确认地点志、见闻计数、补给和地点事件更新。
6. 搜集一个补给，确认资源变化、按钮变为已搜集、日志记录。
7. 处理地点事件，确认事件不再可重复点，并提示回到九州图。
8. 再次切到九州图，确认当前位置从`中原驿`变为`故王道`，距离和可达路线同步更新。

## 8. 最重要的五条验收标准

1. 玩家在任意时刻都能说出自己是在城镇停靠、正在行进，还是半途路遇。
2. 九州图上的每次路线选择都能在点击前看到目的地、消耗、风险、可观察补给和终点距离。
3. 地点页的横向舞台只增强“路上体感”，不抢事件文本、按钮、资源和日志的可读性。
4. 半途停下必须有明确原因、明确代价选择、明确返回抵达流程，不能像随机弹窗。
5. 地点事件、补给和路线选择必须形成闭环：处理完地点 -> 回九州图选路 -> 路上事件 -> 抵达新地点。

## 9. 需要用户决策的问题

1. 九州图最终气质要偏“地图卷轴”，还是偏“星图节点”？第一版建议先偏星图节点，因为路线决策更清楚。
2. 半途路遇第一版是否所有带 `midEvent` 的路线首次必定触发？D 线程建议必定触发，方便测试和体验验收。
3. 路遇是否允许正收益？如果允许，UI 需要把路遇从“惩罚”改成“风险机会”，路线卡文案也要同步。
4. 地点页抵达前是否需要一段短行程动画？D 线程建议保留短反馈，但不做长动画，避免拖慢文字玩法节奏。
5. 地图页是否允许直接点击节点前进，还是必须点击右侧路线卡确认？D 线程建议桌面端两者都可，移动端优先路线卡确认。
