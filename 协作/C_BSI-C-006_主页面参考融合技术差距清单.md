# C_BSI-C-006 主页面参考融合技术差距清单

项目：不思异：九州  
对应主协调任务：BSI-UX-008  
线程角色：C 线程 / Web Demo 技术原型与工具链工程师  
日期：2026-06-14  
阶段：第一阶段技术差距清单，不改代码  

## 一、当前判断

当前原型已经完成了 BSI-C-005 的一部分技术落地：`index.html` 已有横屏游戏框、`game-hud`、旅途 / 九州图两种模式、右侧抽屉、最近日志入口；`styles.css` 已有 16:9 游戏框、横屏小尺寸媒体查询、舞台分层、车队移动、九州图路线图；`app.js` 已有模式切换、抽屉状态、路线卡、地图节点、最近日志、行囊九宫格、音乐和死局保底。

所以 BSI-UX-008 后续不应重写玩法，也不应把已有结构推倒重来。真正差距是：第一眼仍偏“横屏网页 UI 壳”，还没有完全变成“车队正在路上 + 打开九州图做路线资源风险判断”的主页面体验。

## 二、距离 UX-008 的主要差距

### 1. 默认旅途页仍有网页面板感

- HUD 已经压缩，但顶部仍由 `topbar`、`resource-strip`、`journey-overview` 三组盒子拼成，边框和底色让它像网页仪表盘。
- 底部行动区仍是一个 `event-panel` 网格，地点志、补给、事件文本、选择按钮的面板边界都比较明确，游戏行动栏感还不够。
- `location-lore`、`supply-section`、`choice-list` 都在争主视觉，舞台虽然大，但下方面板仍像后台表单。
- 当前小横屏媒体查询会隐藏地点志和最近日志，解决了空间问题，但也可能让“当前位置 / 近期反馈 / 下一步”变得过于硬切。

### 2. “车队正在路上”的默认体感还不够

- 舞台分层、云、昼夜、污染层、车队素材都存在，但默认静止状态仍像一张横幅舞台。
- 车队移动主要发生在 `move(routeId)` 后的短动画；玩家刚打开页面时，还不够像“正在途中等待决策”。
- `fromNode`、`routeNode`、`toNode` 已显示三段路线，但表达更像标签，尚未变成路标 / 路况 / 下一站的画面内信息。
- 半途路遇已有 `pendingRoute` 与 `.interrupted` 状态，但需要在画面上更明确地区分“未到下一站，车队停在路中”。

### 3. 九州图已有路线数据，但战略决策感还可加强

- `renderRoutes()` 已显示目的地、消耗、风险、半途路遇、目的地补给和锁定原因，这是最关键的基础。
- `renderMap()` 已绘制节点与路线线段，但地图节点还是矩形信息卡，视觉上接近网页地图工具，不像九州舆图上的地点 / 道路。
- 路线卡信息完整，但还缺少更强的比较感：例如“出发后资源会变成多少”“哪项资源将接近危险线”“锁定路线为什么值得惦记”。
- 小横屏下路线卡横滑可用，但节点只保留地点名，地图本身的风险 / 资源暗示会变弱，需要由路线卡补足。

### 4. 两个参考方向还没有完全融合

- 西部横向旅行参考应承担“横向路途舞台、远景、层移、停靠、路遇体感”，当前具备骨架，但 UI 面板仍压住舞台。
- Out There 参考应承担“路线前判断、资源压力、风险比较、下一站预期”，当前具备数据，但地图视觉和路线卡层级还没有足够战略化。
- 表达边界要继续守住：不能像西部片换皮，也不能像太空星图换皮；最终应该是“山海舆图 + 古代车队行旅”。

## 三、只改 CSS 与需要少量 HTML/JS 的边界

### 1. 可以优先只改 CSS 的部分

- HUD 减重：降低 `game-hud` 三组容器的边框、底色、阴影，资源从卡片继续压成更轻的常驻状态条。
- 舞台加权：提高 `journey-stage` 在画幅里的视觉占比，减少舞台外卡片边框，让画面成为第一视觉。
- 行动区收口：把 `event-panel` 从“卡片”改成更像游戏底部行动栏；弱化 `location-lore` 的盒子感，减少分割线和大面积背景块。
- 补给和选项按钮游戏化：保留可点击面积，但改成路签 / 令牌 / 行动按钮的轻量表达，不再像网页按钮列表。
- 车队路上感：调整车队初始位置、路面前景、云雾和远山层移强度；用现有 `.traveling` / `.interrupted` 状态强化左到右推进。
- 九州图舆图化：把 `map-node` 从大矩形卡片压成点位 + 短标签，把路线线段改成墨线 / 车辙 / 未知虚线，路线详情交给 `routeList`。
- 小横屏适配：继续在 `max-height: 520px` 横屏媒体查询里压缩 HUD、行动区和路线卡，重点避免文字重叠和按钮过小。

### 2. 需要少量 HTML 的部分

- 如 D 线程方案要求“舞台内路标 / 路况条 / 车队状态短句”，可在 `journey-stage` 内增加极少量无玩法状态的新 DOM，不能复制现有关键 ID。
- 如需把“九州图”入口从页签变成更重要的战略按钮，可调整 `journey-overview` 内按钮分组，但保留 `townViewButton`、`mapViewButton`。
- 如需底部行动区分为“文本区 + 行动区 + 补给区”的更明确结构，可轻微调整 `event-panel` 内部顺序，但保留 `eventText`、`choiceList`、`supplyList`。
- 抽屉结构已经可用，后续不建议再迁移 `itemList`、`logList`、`audioToggleButton`、`motionToggle`，除非 D 线程明确要求。

### 3. 需要少量 JS 的部分

- 路线卡增加资源预判：在 `renderRoutes()` 中把当前资源 + 路线消耗计算成“抵达前预估”，并标红接近 35/15 的资源。
- 事件完成文案轻调：把“打开九州图模式选择下一段旅程”改得更像游戏内行动提示。
- 舞台状态短句：若新增舞台内状态 DOM，可由 `renderStage()` 或 `renderOverview()` 写入“正在前往 / 半途停下 / 下一站”。
- 地图节点信息精简：如 CSS 无法完全解决拥挤，可在 `renderMap()` 内给节点添加短标签或风险等级类名，路线详情仍以 `routeList` 为准。
- 不建议改动 `move()`、`completeRoute()`、`choose()`、`resolveRouteEvent()`、`checkResourceFailure()` 等玩法逻辑。

## 四、默认旅途页如何更像“车队正在路上”

- 让舞台成为页面主语：HUD 和底部行动区都退成轻 UI，`journey-stage` 负责第一眼。
- 把路线三段信息变成画面内路标：起点、当前路况、下一站不要像普通标签，而要像路旁标记或车辙上的提示。
- 默认状态保留轻微行进感：云、风沙、远山、前景路面保持慢速层移，即使没有点击前进，也像车队停在一段路上。
- 半途状态明确停车：`pendingRoute` 时车队停在画面中段，行动区只处理路遇，补给区提示“半途无法搜集城镇补给”。
- 下方文字像行动栏：事件正文、补给、选择按钮都贴近“当前路况 / 当前决定”，减少文章卡片和后台表格感。
- 最近反馈只给一句：保留 `latestLogPeek` 的价值，但不让完整日志常驻抢主画面。

## 五、九州图模式如何形成路线 / 资源 / 风险决策

目标不是复制 Out There 的星图，而是借用其决策语法：出发前比较目的地、消耗、风险、可观察补给和锁定条件。

建议表达：

- 地图层：九州舆图、山海图、旧路墨线、未知虚线、当前车队点、可达下一站、锁定远方。
- 路线卡层：每条路线必须显示“去哪、消耗三资源、风险、半途路遇、目的地补给、锁定原因”。
- 资源预判：路线卡增加“出发后：车轴 / 粮草 / 神志”或“危险资源提示”，让玩家能判断是否会陷入危机。
- 风险差异：低 / 中 / 高风险不只是一行字，CSS 可用不同边线、印记或轻色块表达，但不要复制任何竞品图标。
- 探索感：未发现地点保留未知状态；锁定路线不要隐藏，应该显示“尚未显明”的原因，制造下一轮探索目标。
- 决策重心：地图承担探索方向，路线卡承担实际比较；不要把地图做得复杂而路线信息变弱。

## 六、后续实现必须保留的功能

- 路线：`routes`、`renderRoutes()`、`renderMap()`、`move(routeId)`。
- 半途：`midEvent`、`state.pendingRoute`、`resolveRouteEvent()`、`.interrupted` 舞台状态。
- 补给：`locations[].supplies`、`renderSupplies()`、`useSupply(supplyId)`、一次性补给状态。
- 事件：`events`、`renderEvent()`、`choose(index)`、地点事件已处理后的前进提示。
- 行囊九宫格：`buildInventorySlots()`、`renderInventoryGrid()`、`#itemList.inventory-grid`。
- 日志：`state.log`、`#logList`、`#latestLogPeek`，完整日志继续放抽屉。
- 音乐：`audioAssets`、`audioToggleButton`、`motionToggle`、`updateAudioLayers()`，用户点击解锁不能被藏死。
- 死局保底：`crisisEvents`、`checkResourceFailure()`、`resolveCrisis()`、`enterHardFailure()`、`forceCrisis()` 测试入口。

## 八、测试清单

### 1. 静态与数据检查

- `node --check GitHub资产区/03_WebDemo/prototype/app.js`
- `node --check GitHub资产区/03_WebDemo/prototype/data.js`
- 浏览器执行 `window.BSI_PROTOTYPE.validate()`，确认 `ok: true`。
- 执行 `getView()`、`setView("town")`、`setView("map")`，确认旅途 / 九州图切换正常。

### 2. 核心流程 smoke

- 默认打开就是旅途舞台，能看到地点、日数、三资源、距裂隙、当前事件和下一步。
- 可使用补给，资源变化和日志写入正常。
- 可处理地点事件，事件已处理后能引导打开九州图。
- 九州图显示地图节点、路线线段和路线卡。
- 路线卡显示消耗、风险、半途路遇、目的地补给、锁定条件。
- 点击可达路线后回到旅途舞台，车队行进动画正常。
- 半途路遇能中断、选择、继续抵达。
- 行囊九宫格、日志、设置抽屉可打开和关闭，不压住主流程。
- 音乐按钮能启用 / 关闭，动态开关能关闭动效。
- `window.BSI_PROTOTYPE.forceCrisis("axle")`、`forceCrisis("grain")`、`forceCrisis("sanity")` 能触发濒死补救。

### 3. 画幅验收

- 1366 x 768：舞台是第一视觉，HUD 和底部行动区不重。
- 1280 x 720：路线、补给、事件按钮可读可点。
- 932 x 430：小横屏仍能看见舞台和九州图，不只剩文字列表。
- 844 x 390：无文字重叠，按钮不低于基本触控面积，抽屉能关闭。

### 4. 参考边界检查

- 不出现西部片气质、马车西部片化、竞品字体 / 配色 / 图标 / 音效模仿。
- 不出现太空星图、飞船、星球节点、Out There 式资源图标或文本结构。
- 视觉语言回到《山海经》、九州舆图、古道、车队、异象和裂隙。

## 九、主要风险

- 小横屏高度极紧，若 D 线程要求保留更多首屏文字，舞台会被挤压。
- 现有色彩和面板底色偏厚，若只减边框不重排层级，仍可能像网页皮肤。
- 路线卡信息已经很多，新增资源预判时必须控制长度，否则小横屏横滑卡会难读。
- 地图节点从卡片压成点位后，地图本身会更清爽，但必须保证可点击目标和路线卡信息补足。
- 当前仓库有较多他人改动和未跟踪素材，后续代码阶段必须只碰 Web Demo 原型相关文件。

## 十、是否有必须用户决策的问题

本阶段没有必须立即打断用户的问题。技术上可按以下默认收口：

- 默认旅途页偏“车队正在山海路上”的横向舞台。
- 九州图偏“山海舆图上的路线 / 资源 / 风险决策”。
- 行囊、日志、设置继续作为辅助抽屉，不抢主页面。
- 后续等主线程验收 D 线程方案后，再按 D 的信息架构做最小代码改造。
