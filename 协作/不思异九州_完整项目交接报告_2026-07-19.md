## 2. 一页结论

《不思异：九州》是一款以《山海经》气质、古代采风使命和神秘九州远行为核心的横屏生存叙事游戏。玩家在节点地图上选择路线，在横向长卷式旅途舞台中前进，在地点和半途事件中处理异象、民疾、官场与资源危机，并用不同写法把见闻收入《九州总志》。

当前已经存在一个可运行、可在线试玩、可走完整路径、可进入结局、可复制反馈包的 P0 Web 纵切片。它不是完整商业版，也不是只停留在文档里的概念稿。

当前最重要的三项资源为：

| 资源 | 表层含义 | 玩家问题 | 系统职责 |
| --- | --- | --- | --- |
| 车轴 | 车轮、辎车、绳索、舟板与整体通行能力 | 还能不能走到下一站 | 路线成本、地形损耗、断轴危机 |
| 粮草 | 粮食、草料、药材、薪柴与队伍续航 | 还能不能撑过今天 | 移动消耗、补给、救人、交易、饥荒危机 |
| 神志 | 对地图、文字和所见世界的认知稳定 | 看到的还是真的吗 | 文本污染、隐藏真相、异兆路线、崩溃危机 |

## 3. 产品定位

### 3.1 一句话介绍

一个奉命记录九州的古代车队，在有限的车轴、粮草和神志中穿过越来越不像人间的世界；玩家写下的每一笔，既可能救下一地，也可能成为它的判词。

### 3.2 类型

- 横屏手机优先的 Web 游戏原型。
- 节点地图探索。
- 资源管理。
- 文字选择事件。
- 轻 Roguelike 与多结局重开。
- 假动态横向旅途。
- 山海经气质的政治志怪叙事。
- 采风记录与后果系统。

### 3.3 它不是什么

- 不是横版动作游戏。
- 不是开放世界自由移动。
- 不是传统打怪升级 RPG。
- 不是单纯的怪物图鉴。
- 不是高操作强度的战斗游戏。
- 不是依靠随机生成文本临场编剧情的 AI 游戏。
- 当前版本也不是正式商业发行包。

### 3.4 参考作品与原创边界

产品曾研究《Out There》、西部远行题材横屏游戏和《Darkest Dungeon》的表现方式。

可借鉴的是抽象层：

- 节点路线与未知信息。
- 三资源压力。
- 地点停靠与一次性补给。
- 横向旅途的舞台感。
- 事件选择与结果落账。

不能复制的是具体表达：

- 不复制对方界面布局、图标、文案、数值表、地图、角色、世界观、音乐、美术与事件。
- 不使用对方专有名词。
- 不按截图做近似复刻。
- 不把对方的资源曲线或事件结构逐项翻译。

玩法规则通常不由著作权直接垄断，但具体表达、视听素材、代码、文本、商标和整体近似仍可能产生风险。正式商业发行前应由专业法律顾问做一次产品级审查；现有 E 线程文档只属于内部风险研究，不构成法律意见。

## 4. 玩家身份与新主线

### 4.1 原 P0 身份

原 P0 使用较抽象的“王朝采风使”。玩家奉命进入九州，记录风土、民谣、异闻和证词。

### 4.2 新剧本主干

新设定把抽象身份落成了固定人物和政治任务：

- 陆离：风宪司风宪郎，书生型主角，正直克制，患肺疾。
- 宛渠：鲛人遗民，擅武与器械，隐姓埋名，背负对韩凌君的复仇。
- 烈帝：派陆离编修《九州总志》，暗中希望获得民间疾苦和韩党暴政的证据。
- 韩凌君与韩党：把持朝野，在九州各地形成压迫与利益网络。
- 表层任务：采九州风物，修总志。
- 隐藏任务：寻找民疾、罪证和王朝真实状况。

建议把世界观称为“永昌朝”或“架空汉制王朝”，不要在对外文案中继续并列使用“公元前156年”和“汉宣帝”。原设定存在明显史实冲突，必须由主创最终拍板。

### 4.3 双主角关系

下一阶段建议采用固定双主角：

- 玩家主要以陆离视角做记录、判断和写作。
- 宛渠承担同行者、现实行动、器械能力、异族身份和复仇线。
- 二人的关系不是单纯战斗伙伴，而是“记录真相的人”与“被历史吞噬的人”之间的互证。

当前代码尚未实现角色关系变量。P0 不应立刻增加新的 HUD 数值；可先用事件旗标记录宛渠线索、信任和关键选择，P1 再决定是否加入隐藏统计。

## 5. 核心玩法循环

### 5.1 当前循环

```text
抵达地点
-> 阅读地点志和诗引
-> 处理地点事件
-> 选择采风写法
-> 执行一次性补给
-> 打开九州图
-> 查看路线成本、补给倾向和半途风险
-> 二次确认后启程
-> 横向舞台推进
-> 处理半途路遇
-> 抵达下一地点
-> 新路线显影
-> 继续或进入结局
```

### 5.2 三类遭遇

| 类型 | 触发时机 | 作用 |
| --- | --- | --- |
| 地点事件 | 抵达地点后 | 交代风俗、人物、异象、政治冲突和采风题材 |
| 半途路遇 | 两地之间 | 制造旅途不确定性、资源交换和短决策 |
| 危机补救 | 某项资源触底 | 避免坏运直接形成无解死局，给最后一次有代价的补救 |

### 5.3 死局防护

当前机制不在第一次资源触底时直接判死：

- 车轴触底进入断轴补救。
- 粮草触底进入饥荒补救。
- 神志触底进入认知危机。
- 连续补救过多、同类危机反复或坏运过高，才进入“旅途断绝”硬失败。

这套机制是 P0 稳定性的关键，不应为了“更残酷”随意删除。后续应通过成本、预警、可控风险和补给稀缺制造压力，而不是依赖突然无解。

### 5.4 随机稳定策略

当前采用“受控随机”而不是完全随机：

- 地点和路线的基础结构固定。
- 事件从符合地形、阶段和状态的池中抽取。
- 每个节点有明确补给倾向。
- 关键路线上保留可生存路径。
- 资源触底有危机补救。
- 自动模拟持续检查前 3 步死局、硬失败、告警率和资源分位数。

新增事件、路线或补给后必须重跑 1000 局平衡模拟，否则不能只凭主观判断宣布平衡。

## 6. 采风系统

### 6.1 核心意义

采风不是图鉴收集的装饰层，而是本项目与普通资源生存游戏的主要差异：

- 玩家不只处理事件，还决定如何记录事件。
- 记录会进入采风册、日志和本局复盘。
- 同一件事可以被写成祥瑞、讽谏、哀悼或异兆。
- 写法会影响朝廷如何理解一个地方。
- 新剧本中，写法将进一步影响韩党罪证、民疾证据和三结局。

### 6.2 四种写法

| 写法 | 叙事功能 | 新主线政治含义 | 结局倾向 |
| --- | --- | --- | --- |
| 颂 | 记物产、秩序、祥瑞 | 盛世粉饰，可能遮蔽民怨 | 偏坏结局 B，但少量使用不应直接受罚 |
| 讽 | 含蓄记录官府矛盾 | 韩党罪证 | 偏好结局 A |
| 哀 | 记录饥渴、劳役、流亡 | 民疾证据 | 强偏好结局 A |
| 谶 | 记录无法解释的异兆 | 神志真相、妖书风险 | 偏隐藏线 C，也可能因证据不足落入 B |

### 6.3 后续底层标签

P1 建议在 UI 保持四按钮不变，后台增加：

- `民疾证据`
- `韩党罪证`
- `盛世粉饰`
- `异兆谶言`
- `隐匿 / 销毁`
- `宛渠身世线索`

这些标签不应变成顶部资源。玩家应通过采风册、案牍反馈、角色反应和结局复盘理解后果，而不是再增加一排数字。

## 7. 地图、旅途与地点结构

### 7.1 九州战略图

九州图负责战略选择：

- 只显示已发现节点和可见路线。
- 抵达新地点后迷雾散开。
- 后续可走路线逐步显现。
- 路线卡显示目的地、资源成本、补给倾向和半途风险。
- 第一次点击为预览，第二次点击才启程，防止触屏误操作。

### 7.2 横向旅途舞台

横向舞台是“中地图”和表现层：

- 背景缓慢向右移动，制造远行感。
- 远山、云雾、前景、神志污染层分层移动。
- 车队不是动作主角，必要时可以弱化或隐藏。
- 中途可以停止并弹出路遇。
- 不做碰撞、不做平台动作、不做连续物理关卡。

它的技术成本远低于真正横版游戏，但沉浸感明显高于纯点线图。现有实现方向应保留。

### 7.3 地点页

地点页应承担：

- 地点名和短标签。
- 一张地点图。
- 地点志。
- 一句或两句诗引，作者和篇名另行。
- 专属事件。
- 一次性差异化补给。
- 采风题材。
- 下一站入口。

长说明不应常驻在主画面下方。当前版本已经把遭遇详情、地点志、图鉴和结果放入居中弹窗或抽屉，适合横屏触屏。

## 8. 神志系统

神志不是普通生命值，而是叙事滤镜和风险收益系统。

建议维持三层文本：

| 神志层 | 文本气质 | 信息可靠性 |
| --- | --- | --- |
| 高 | 地理志、官修记录 | 相对可靠，可判断 |
| 中 | 异闻录、征兆增多 | 事实与错觉混合 |
| 低 | 疯人书、句法断裂 | 不可靠，但可能看见隐藏真相 |

神志低时可改变：

- 地点志。
- 事件正文。
- 诗引的呈现。
- 地图和路线提示。
- 音频污染层。
- 画面叠层。
- 特殊选择和隐藏线索。

不能把低神志只做成“文字更难读”。玩家应获得少量独有信息或路线价值，否则它只是一种令人厌烦的视觉惩罚。

## 14. 技术架构

### 14.1 技术路线

当前是无框架静态 Web 原型：

- HTML。
- CSS。
- 原生 JavaScript。
- 浏览器 `localStorage` 存档。
- Node.js 脚本用于数据、资产、平衡和部署检查。
- 无必需打包器。
- 无后端账号系统。
- 无数据库。

- 打开快，部署简单。
- 适合 P0 机制验证。

## 20. 协作制度与专业分工

项目采用主线程统筹、专业线程交付、主线程验收的方式。

| 线程 | 职责 |
| --- | --- |
| A 机制与平衡 | 三资源、随机、保底、失败、难度、模拟 |
| B 世界观与叙事 | 地点、事件、神志文本、人物、六章、结局 |
| C 技术原型 | Web 代码、数据接入、状态机、检查与部署包 |
| D 视觉与交互 | UI、地图、事件图、资产切片、动效和提示语 |
| E 同类与商业风险 | 借鉴边界、素材授权、字体音频和合规 |
| F 音乐与音效 | 音乐、环境声、短音、接入和听感验收 |

标准流程：

```text
提出任务
-> 主线程写任务书
-> 专业线程提交方案或产物
-> 主线程检查边界、文件和可执行性
-> 技术线程接入
-> 机器检查
-> 真人验证
-> 主线程验收并更新台账
```

重要规则：

- 不允许专业线程直接改动不属于自己的系统。
- 世界观文档不能自动等于已实现功能。
- 生图提示语不能自动等于可商用资产。
- 机器通过不能替代真人手感。
- 未经主线程验收的文件保留“候选”状态。
- 每项任务以任务 ID、文件路径和回执为准，不依赖聊天记忆。

## 21. 内容协作者交接

《山野逸事》等旧剧集设定可成为内容来源，但原编剧不需要手工拆数百张游戏卡。

推荐流程：

1. 内容协作者提交原文档、设定、典故和未解决问题。
2. AI 先阅读并提出一批结构化反问。
3. 内容协作者回答人物动机、主题、尺度和不可改边界。
4. AI 拆成地点、事件、人物、风俗、物件和采风题材候选。
5. B 线程做叙事改编。
6. A 线程评估玩法价值。
7. C 线程只接入已批准内容。

这样能让原作者发挥最有价值的部分：

- 判断主题。
- 解释人物。
- 决定不可改设定。
- 补足故事因果。
- 审核改编是否失去原意。

批量格式整理和素材卡拆解交给 AI。

## 22. 商业与授权风险

### 22.1 玩法借鉴

风险控制重点：

- 保留抽象机制，重做具体表达。
- 不复刻《Out There》的界面、地图视觉和文案。
- 不直接使用竞品截图作游戏资产。
- 不照搬其路线、资源、数值和事件结构。
- 对外介绍强调古代采风、九州总志、双主角和神志文本等独立表达。

### 22.2 生成素材

正式发行前每项资产应有：

- 来源。
- 生成平台或作者。
- 生成日期。
- 输入参考。
- 是否含第三方角色或作品。
- 可商用判断。
- 人工复核状态。
- 最终入库版本。

### 22.3 现实文化锚点

新剧本中曾用阿拉伯、因纽特、琉球等现实文化作为内部参考。正式文本建议只转译：

- 地貌。
- 生产方式。
- 器物。
- 居住形态。
- 交通与祭祀逻辑。

不要把现实族群直接当作奇观或异族标签。

### 22.4 内容尺度

鲛人屠杀、献童、剥取、饥荒与身体异化等内容可能影响：

- 年龄分级。
- 平台审核。
- 宣传素材。
- 美术表现。
- 玩家接受度。

P0 建议以暗示和结果为主。正式版是否明写，应由主创、发行和法律顾问共同决定。

## 28. 最终交接结论

当前项目已经越过“只有想法”的阶段：

- 有清晰的产品定位。
- 有可玩的完整 P0 循环。
- 有线上试玩。
- 有路线、事件、补给、采风、神志、结局和反馈闭环。
- 有较完整的视觉、音频和协作资产。
- 有自动 QA、平衡模拟和部署检查。
- 有新的正式剧本主干和第一章改编方案。

但它仍未进入完整商业制作：

- 真人反馈仍不足。
- 音频和部分图像仍是内部临时状态。
- 新剧本尚未写入游戏数据。
- 六章内容、正式三结局、角色关系和商业工程化仍待开发。

接手团队最应避免的错误，是把“已有大量文件”误判为“所有内容已经完成”，或者把“新设定已经写成文档”误判为“已经进入游戏”。

最合理的下一步是：

```text
先完成主创拍板
-> 只制作第一章沙墟纵切片
-> 保留现有 P0 作为回归基线
-> 机器验证
-> 真人试玩
-> 再决定是否展开六章
```

这条路径能最大限度保留已有成果，同时把新剧本真正变成可玩的游戏，而不是继续累积彼此脱节的文档和素材。
