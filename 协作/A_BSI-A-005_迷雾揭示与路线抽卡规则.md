# A_BSI-A-005 迷雾揭示与路线抽卡规则

> 子线程：A，核心规则与数值平衡  
> 任务：BSI-A-005，对应主协调任务 BSI-UX-009  
> 范围：只写规则文档，不修改 WebDemo 原型代码  
> 依据：`主协调_BSI-UX-009_羊皮纸九州图与迷雾揭示交接单.md`、`A_BSI-A-004_路线决策与路遇规则.md`、当前 `03_WebDemo/prototype/data.js`

## 1. 设计目标

本轮把九州图从“完整路线图”改成“逐步显影的羊皮纸山海舆图”。玩家不应一开始看到完整世界，而是在每次抵达新城后，看见当前城、来路、可走前路、少量雾中远影。

核心口径：

```text
抵达新城 -> 迷雾散开 -> 城市显现 -> 可走路线显现 -> 雾中远方给出诱饵和压力
```

这套规则服务三件事：

1. 给玩家 Out There 式路线决策压力：每条路都是资源、风险、信息收益的取舍。
2. 用“雾中显影”承载随机性和抽卡感：不是全图铺开，而是每到一处抽出下一组前路。
3. 保留当前 WebDemo 的 `locations / routes / routeEvents / supplies` 骨架，第一版先用可控随机和预制池模拟，不做完整程序生成世界。

## 2. 地图节点揭示状态

第一版建议节点至少有 6 种状态，C 线程可先用样式类或派生状态实现。

| 状态 | 含义 | 规则来源 | 玩家可见信息 | 可交互 |
|---|---|---|---|---|
| 当前 | 车队所在城市 | `state.currentLocation` | 清晰地名、区域、地貌、车队标记、当前可补给/事件 | 可点开详情 |
| 已见 | 已到达或已经完整显现的城市 | `state.discoveredLocations`，不含当前 | 清晰地名、区域、地貌、已搜集补给状态 | 可查看，是否可回撤由路线决定 |
| 可达 | 当前城市后续可走节点 | 当前地点出发且未锁定的 `routes` | 目的地名、路线卡、消耗、风险、可观察补给倾向 | 可选择路线 |
| 可达但锁定 | 从当前城市看得见，但条件不足 | 当前地点出发，存在 `requireAny` 且未满足 | 目的地可见或半可见、锁定原因、缺口提示 | 不可走 |
| 雾中 | 远方未完整揭示节点 | 一跳之外、预告节点、抽卡候选 | 模糊节点、方向、地貌影子、可能资源倾向，不给完整路线 | 不可走 |
| 未入图 | 玩家暂时完全不知道 | 不在当前可见/雾中集合 | 不显示 | 不可交互 |

状态优先级：

```text
当前 > 已见 > 可达 > 可达但锁定 > 雾中 > 未入图
```

同一节点如果同时满足多个条件，以优先级最高的状态显示。例如玩家站在“赤水外滩”时，它永远是“当前”，即使它也在 `discoveredLocations` 中。

### 2.1 路线状态

路线也需要对应显隐状态，避免“节点雾中但路线全露”的割裂。

| 路线状态 | 含义 | 显示方式 | 规则 |
|---|---|---|---|
| 已走 | 本局已经走过 | 淡墨车辙、旧朱砂线 | `state.traveledRoutes` 或后续新增记录 |
| 可走 | 当前地点出发且条件满足 | 朱砂线 / 金墨线，路线卡完整 | 可点击确认 |
| 行进中 | 已选路线，半途路遇未处理 | 路线高亮，车队停在线中段 | `state.pendingRoute` 存在 |
| 雾中 | 远方可能有路 | 断续墨线、方向影、不可读目的地 | 只作预告，不可点击 |
| 锁定 | 路线存在但条件不足 | 封印线、断线、红墨锁记 | 显示 `lockedHint` |
| 未显现 | 还没轮到玩家知道 | 不画 | 不进路线卡 |

第一版不要把关键锁定路线完全藏掉。像 `red_to_rift` 这类主线终点路线，应该在玩家抵达赤水外滩后灰显，让玩家知道“那里有路，但缺条件”。

## 3. 路线显现规则

### 3.1 开局显现

开局只显示“中原驿”和第一组可决策前路。

建议开局规则：

1. `central_post` 为当前节点。
2. 从 `central_post` 出发的路线显现为可走：`central_to_road`、`central_to_pass`。
3. 两个目的地 `old_king_road`、`abandoned_pass` 显示为可达。
4. 再远一层只显示雾中影子，不显示完整路线网：
   - 可提示“西北泽气”“日中边市”“更西有红水”一类方向或地貌。
   - 不提前展示全部通往 `kyushu_rift` 的明确路径。
5. 路线卡必须保留 A-004 的比较信息：消耗、风险、半途、目的地可观察补给、锁定原因。

开局不建议随机隐藏两条起始路线。第一步是教学，必须稳定。

### 3.2 抵达新城后的显现

玩家完成路线移动并抵达目的地后，按以下顺序结算：

1. 旧当前城市变为已见。
2. 本次路线变为已走。
3. 目的地变为当前。
4. 目的地的地点事件、补给和当前资源状态刷新。
5. 从新当前城市出发的候选路线显现。
6. 候选路线目的地变为可达或可达但锁定。
7. 一跳之外新增 1-3 个雾中节点或雾中道路影子。

这一步要产生“雾散、城显、路显”的反馈。规则上，玩家看到的是从当前城市向外扩一圈，不是全图一次性解锁。

### 3.3 半途状态

若路线有 `midEvent` 或抽到路遇，路线不应直接完成。

半途规则：

1. 选择路线后先结算路线消耗。
2. 若触发路遇，进入 `pendingRoute`。
3. 地图上旧城市仍为已见或当前锚点，车队标记停在路线中段。
4. 目标城市可以保持“将至 / 雾边 / 可达”状态，但不要提前打开目标地点事件和补给。
5. 玩家处理完半途路遇后，才真正抵达并触发新城显现。

这样可以避免玩家绕过路遇，也能让半途停车在地图上有存在感。

### 3.4 锁定路线显示

锁定路线分两类处理。

| 锁定类型 | 例子 | 显示规则 |
|---|---|---|
| 主线关键锁定 | `red_to_rift` | 抵达起点后必须灰显，显示终点轮廓和 `lockedHint` |
| 支线/高收益锁定 | 未来的秘市、禁地、异族小径 | 可先显示封印线或雾中影，不一定给完整地名 |

主线关键锁定必须给替代入口。当前 `red_to_rift` 的三种入口很好：

```text
followed_bones：地点事件清醒路径
rift_name：断碑事件清醒路径
sanityMax 34：低神志危险路径
```

后续主线锁定建议继续采用“至少两条清醒路径 + 一条危险认知路径”。

### 3.5 回撤路线

第一版允许有限回撤，但不要做成刷资源。

建议：

1. 已见城市之间如果存在路线，可显示为回撤或绕行。
2. 回撤路线的风险应低于继续深入，但收益也低。
3. 已搜集补给不会刷新，路线卡必须显示“此地补给已空”。
4. 回撤主要用于防死局，不是主要推进玩法。

## 4. 抽卡式地点 / 路线 / 事件池

第一版不需要程序生成新世界，可以把现有地点和路线当作一副“预制牌组”，每次抵达新城时按规则抽出下一组可见牌。

### 4.1 抽卡单位

| 池 | 抽出的是什么 | 当前数据可用来源 | 第一版用途 |
|---|---|---|---|
| 地点池 | 下一个可达城市、雾中远影 | `locations` | 决定地图上显出哪些节点 |
| 路线池 | 从当前城市出发的路线卡 | `routes` | 决定玩家本轮可选路线 |
| 路遇池 | 半途事件 | `routeEvents` + `route.midEvent` | 决定路线中段是否打断 |
| 补给倾向池 | 目的地可观察资源 | `locations[].supplies` | 决定路线卡上的资源诱因 |
| 锁定/线索池 | 可见但不可走的高价值路线 | `requireAny / flags / languages / conditions` | 给玩家中期目标 |

### 4.2 地点池权重

地点抽卡不是全随机，而是受 `map.step`、地貌、当前资源和主线进度约束。

| 因素 | 权重建议 | 说明 |
|---|---:|---|
| 从当前地点有直接路线 | 必选 | 当前可走路线的目的地必须显现 |
| `map.step = 当前 step + 1` | +40 | 优先显现向西推进的节点 |
| 同 step 支线或回撤 | +15 | 用于补给、避险、绕行 |
| `map.step >= 当前 step + 2` | 只进雾中 | 可给远影，不给完整路线 |
| 目的地有当前危险资源的补给 | +20 | 低车轴时更容易提示修车点 |
| 目的地地貌连续高风险 | -15 | 避免连续把玩家推向高耗地貌 |
| 已搜集完且无事件收益 | -10 | 不把空地点伪装成好选择 |

当前地点可大致归类：

| 地点 | 适合定位 | 资源倾向 |
|---|---|---|
| 中原驿 | 起点 / 教学补整 | 车轴、粮草 |
| 故王道 | 低风险推进 / 车轴补救 | 车轴 |
| 废关 | 边境支线 / 修车高收益 | 车轴，损神志 |
| 雷泽浅畔 | 泽地推进 / 粮草补给 | 粮草，损车轴或神志 |
| 黑齿市 | 交易支线 / 粮草与神志 | 粮草、神志，损车轴或粮草 |
| 赤水外滩 | 主线锁定前沿 | 粮草少量，神志压力 |
| 巫咸断碑 | 解锁与安神 | 神志、裂隙名 |
| 九州裂隙 | 终点 / 结局前整理 | 神志，结局选择 |

### 4.3 路线池权重

从当前城市抽路线时，先放入所有 `from = currentLocation` 的预制路线，再做权重排序和保底修正。

基础权重：

| 路线风险 | 基础权重 | 第一版显示建议 |
|---|---:|---|
| 低 | 40 | 至少保留一条低或中风险路线 |
| 中 | 35 | 作为主要取舍路线 |
| 高 | 20 | 作为诱惑、捷径、主线压力 |
| 锁定 | 15 | 可灰显，不占“可走路线”保底名额 |

资源修正：

| 当前资源状态 | 对路线池的影响 |
|---|---|
| 车轴 < 30 | 降低高车轴消耗路线权重；提高通往修车地点路线权重 |
| 粮草 < 35 | 降低长途和高粮耗路线权重；提高通往泽地采集、市场交易路线权重 |
| 神志 < 45 | 降低高神志损耗路线权重；提高通往断碑、安神补给路线权重 |
| 任一资源进入危急线 | 不允许只显示高消耗路线；必须显出补救或低成本出口 |

路线消耗边界沿用 A-004：

| 风险 | 车轴 | 粮草 | 神志 |
|---|---:|---:|---:|
| 低 | -3 到 -5 | -3 到 -5 | 0 到 -2 |
| 中 | -4 到 -9 | -5 到 -8 | -2 到 -7 |
| 高 | -7 到 -12 | -8 到 -11 | -8 到 -14 |

第一版不要新增比 `red_to_rift` 更重的普通路线。

### 4.4 路遇池权重

沿用 A-004：配置了 `midEvent` 的路线首次必触发；之后或无 `midEvent` 的路线走轻随机。

| 风险 | 路遇触发率 | 权重口径 |
|---|---:|---|
| 低 | 15% | 轻损耗、线索、小补给 |
| 中 | 35% | 地貌匹配、交易、风险/收益二选一 |
| 高 | 60% | 高压力、关键线索、高收益高代价 |

现有 3 个路遇的地貌匹配：

| 路遇 | 优先地貌 | 资源主题 |
|---|---|---|
| `wheel_omen` | `road` | 车轴、赶路代价 |
| `roadside_shrine` | `market` / `road` | 粮草换神志、绕路伤轴 |
| `black_cloud` | `water` | 泽地压力、车轴/神志取舍 |

`rift` 地貌第一版不建议抽普通路遇，优先让地点事件和结局事件承担压力，避免结局前节奏被随机打散。

### 4.5 补给倾向池

补给不单独“发牌”，而是影响地点和路线被抽中的倾向，并显示在路线卡上。

| 当前状态 | 倾向 |
|---|---|
| 车轴低 | 更容易显出废关、故王道等修车点；路线卡强调“可修车 / 旧铁 / 门木” |
| 粮草低 | 更容易显出雷泽、黑齿市；路线卡强调“可采食 / 可交易” |
| 神志低 | 更容易显出巫咸断碑、黑齿市静神药粉；路线卡强调“可安神 / 可正名” |
| 资源健康 | 按主线推进和风险多样性显现，不刻意补救 |

补给必须受 `usedSupplies` 限制。已搜集地点仍可作为道路节点，但不能继续被当作有效补给来源加权。

## 5. 死局防护和保底规则

保底目标不是让玩家稳赢，而是保证玩家还看得见至少一个可理解选择。

### 5.1 显图前检查

每次渲染九州图前做一次检查：

| 检查项 | 通过标准 | 不通过时 |
|---|---|---|
| 有可推进路线 | 当前地点至少 1 条可走路线可推进到新地点或关键回撤点 | 插入低收益回撤线或困境事件 |
| 有可承受路线 | 至少 1 条可走路线不会立即让任一资源 <= 0 | 提高补救路线权重或触发危机补救 |
| 锁定有替代 | 主线锁定至少有 2 条清醒解法或 1 条危险解法 | 显示缺口并显出最近线索地点 |
| 没有连续压死 | 上一次高风险后，本次不只显示高风险 | 强制加入低/中风险路线或补给倾向 |
| 补给不误导 | 已搜集地点不作为补给保底 | 标注“已空”，另找补救 |

### 5.2 至少一条可推进路线

第一版硬规则：

```text
只要游戏状态仍是 playing，当前地点必须至少有一条可走路线，或一个可处理的地点/危机事件能让路线重新可走。
```

若当前地点所有路线都锁定，优先处理：

1. 显示最近的线索地点或回撤路线。
2. 若资源太低，触发对应 `crisisEvents`。
3. 若是主线锁定，允许灰显目标并给明确缺口。
4. 不直接把玩家停在空地图上。

### 5.3 避免连续高消耗

连续压力规则：

| 情况 | 限制 |
|---|---|
| 上一条路线为高风险 | 下一次显现不得全部为高风险 |
| 连续 2 次坏事件 | 下一次路遇不得抽高损耗结果 |
| 连续 3 步无正向收益 | 下一次事件或地点必须包含资源、flag、language、解锁线索之一 |
| 刚触发危机补救后的 2 步 | 禁止抽同一资源的高损耗路遇 |

坏事件和正向收益定义沿用 A-004：

```text
坏事件：没有获得资源、旗标、语言、路线线索或结局进展，
且总资源净损耗 >= 8，或任一资源单项净损耗 >= 6。

正向收益：资源净增、获得 flag、获得 language、解锁路线、
获得明确目的地提示、完成补救。
```

### 5.4 资源危险时的补救倾向

当前阈值沿用 A-004。

| 资源 | 低值预警 | 危急线 | 补救倾向 |
|---|---:|---:|---|
| 车轴 | 30 | 12 | 修车点、低车轴消耗路、断轴补救 |
| 粮草 | 35 | 15 | 采集点、交易点、粮草补救 |
| 神志 | 45 | 20 | 断碑、正名、静神、神志补救 |

当资源低于预警线时，抽卡权重向补救倾斜；低于危急线时，普通随机让位给危机补救或低成本出口。

注意：补救不等于免费送满。可以给资源，但应带代价、坏运、旗标或后续风险。

### 5.5 锁定路线保底

锁定路线必须避免“只差一个隐藏条件但玩家不知道去哪找”。

每条主线锁定路线至少满足以下之一：

1. 路线卡显示明确 `lockedHint`。
2. 地图上显出一个雾中或可达的线索地点。
3. 当前或已见地点事件中有可获得对应 flag/language 的选择。
4. 低神志危险路径可见，但不作为唯一解。

`red_to_rift` 当前满足此规则：赤水外滩可获得 `followed_bones`，巫咸断碑可获得 `rift_name`，低神志可看到幻路。

## 6. 与现有数据结构的兼容建议

当前 WebDemo 已经有足够骨架，不建议推翻。

### 6.1 `locations`

可继续使用：

- `id / name / region / terrain`
- `event`
- `map.x / map.y / map.step`
- `supplies`

建议第一版由 `map.step` 和路线关系派生雾中层级：

```text
当前 step：当前 / 已见 / 可达
下一 step：可达或雾中
更远 step：只作为雾中远影，除非被路线直接显现
```

后续如需更细，可给地点加可选字段：

| 字段 | 用途 | 是否首版必须 |
|---|---|---|
| `fogLabel` | 雾中显示的短文案，如“泽气”“断碑影” | 可选 |
| `revealHint` | 显影时的短反馈 | 可选 |
| `resourceProfile` | `["grain", "sanity"]` 这类补给倾向 | 可选，可从 `supplies` 派生 |
| `mainStep` | 主线推进层级 | 可选，当前可用 `map.step` |

### 6.2 `routes`

可继续使用：

- `from / to`
- `risk / terrain / cost`
- `hint`
- `midEvent`
- `requireAny / lockedHint`

建议新增字段都保持可选：

| 字段 | 用途 | 是否首版必须 |
|---|---|---|
| `revealMode` | `normal / alwaysHint / hiddenUntilNear` | 可选 |
| `poolTags` | 如 `["main", "repair", "shortcut"]` | 可选 |
| `drawWeight` | 手调权重 | 可选 |
| `safeFallback` | 是否可作为保底路线 | 可选 |
| `lockImportance` | `main / side`，决定锁定路线是否灰显完整目的地 | 可选 |

第一版可以先完全不加字段，由当前地点的出边路线派生可走路线。

### 6.3 `routeEvents`

现有 `routeEvents` 可继续使用。建议后续可选补充：

| 字段 | 用途 |
|---|---|
| `terrainTags` | 适合抽到的地貌 |
| `riskTags` | 适合低/中/高风险路线 |
| `resourceTags` | 影响车轴/粮草/神志 |
| `positiveTag` | 是否算正向收益，用于保底计数 |

第一版仍以 `route.midEvent` 为准，随机事件只做轻量补充。

### 6.4 `supplies`

现有地点补给足够使用。规则上只需确保：

1. `usedSupplies` 会影响路线卡观察信息。
2. 已搜集地点不再作为补给保底来源。
3. 补给效果参与资源危险判断。

可选新增：

| 字段 | 用途 |
|---|---|
| `resourceTags` | 如 `["axle"]`，便于低资源加权 |
| `supplyTier` | `small / large / risky`，便于显示强弱 |

这些都可以先从 `effect` 自动判断，不急着手填。

## 7. 给 C 线程的数据字段建议

尽量少加字段。首版建议把大部分状态派生出来。

### 7.1 状态字段

当前已有：

- `currentLocation`
- `discoveredLocations`
- `usedSupplies`
- `routeEventResults`
- `pendingRoute`
- `badLuckMeter`
- `failureStats`

建议新增最小状态：

| 字段 | 类型 | 用途 |
|---|---|---|
| `hintedLocations` | `string[]` | 雾中节点，不等于已发现 |
| `revealedRoutes` | `string[]` | 已显现过的路线，避免到达后重新洗牌 |
| `hintedRoutes` | `string[]` | 只画雾中道路影，不给完整路线卡 |
| `traveledRoutes` | `string[]` | 已走路线，用于旧车辙和回撤判断 |
| `routeDrawHistory` | object | 记录每个地点抽出的路线，保证同一地点回看稳定 |
| `safetyCounters` | object | 记录连续高风险、坏事件、无正向收益步数 |

如果想更轻，也可以先不持久化 `revealedRoutes / hintedRoutes`，每次根据 `currentLocation + discoveredLocations + hintedLocations` 派生。

### 7.2 推荐派生函数

C 线程可以用概念上的派生函数，不必真的按此命名。

| 派生结果 | 输入 | 输出 |
|---|---|---|
| 节点显示状态 | `currentLocation / discoveredLocations / hintedLocations / routes` | 当前、已见、可达、锁定、雾中、未入图 |
| 路线显示状态 | `currentLocation / revealedRoutes / hintedRoutes / traveledRoutes / requireAny` | 可走、锁定、雾中、已走、行进中 |
| 可承受路线 | 当前资源 + route.cost | 过滤会立即归零的路线 |
| 补救倾向 | 当前资源 + usedSupplies + locations.supplies | 提高对应地点/路线权重 |
| 抽卡修正 | safetyCounters + badLuckMeter | 防连续高压 |

### 7.3 可控随机字段

第一版可以使用“地点显现计划”模拟抽卡，不必程序生成。

建议数据形态：

```text
revealPlan[locationId] = {
  routes: [固定显现路线],
  fogLocations: [雾中节点],
  fogRoutes: [雾中道路影],
  priorityHint: "资源低时优先显出的路线或地点"
}
```

这不是要求马上改代码，而是给 C 线程一个最小实现方向。没有 `revealPlan` 时，就用现有 `routes.from = currentLocation` 自动派生。

## 8. 第一版实施口径

第一版不做完整程序生成，采用“预制池 + 可控随机 + 保底修正”。

### 8.1 推荐流程

```text
进入地图
-> 根据当前地点取出预制候选路线
-> 判断锁定条件
-> 根据资源和安全计数调整显示顺序
-> 至少保留一条可推进路线
-> 显示可达节点、锁定节点、雾中远影
-> 玩家选择路线
-> 可能触发半途路遇
-> 抵达后更新已见和下一组雾中显影
```

### 8.2 当前数据可直接形成的显现节奏

| 当前地点 | 可显现路线 | 雾中远影建议 |
|---|---|---|
| 中原驿 | 故王道、废关 | 雷泽泽气、黑齿市影 |
| 故王道 | 雷泽浅畔、废关 | 赤水外滩红水影 |
| 废关 | 黑齿市 | 巫咸断碑影、赤水方向 |
| 雷泽浅畔 | 赤水外滩 | 九州裂隙远雾、断碑影 |
| 黑齿市 | 巫咸断碑、赤水外滩 | 九州裂隙远雾 |
| 赤水外滩 | 九州裂隙锁定、巫咸断碑 | 裂隙破痕 |
| 巫咸断碑 | 九州裂隙 | 裂隙破痕 |
| 九州裂隙 | 结局事件 | 不再抽普通路线 |

### 8.3 本轮不建议做的事

1. 不要一次性展示完整路线网。
2. 不要让随机抽卡覆盖起点教学。
3. 不要让锁定主线完全隐藏。
4. 不要在资源危急时继续抽高损耗路线压死玩家。
5. 不要为抽卡新增大量难维护字段；先用现有数据加少量状态即可。

## 9. 需要用户决策的问题

本轮没有必须立刻让用户拍板的问题。A 线程建议按以下默认值推进：

1. 主线锁定路线灰显，不完全隐藏。
2. 开局路线固定，不随机。
3. 抵达新城后抽出 2 条左右可比较路线，另给 1-3 个雾中远影。
4. 资源危险时抽卡向补救倾斜，但不免费补满。
5. 第一版使用当前 `data.js` 预制地点和路线模拟抽卡，不做完整程序生成。

后续如果主协调要扩大随机性，再让用户决定“每局地图布局是否可变”。首版先验证迷雾揭示、路线卡决策和死局防护即可。

## 10. A 线程结论

BSI-A-005 的规则收口如下：

```text
地图不是全开放，而是按当前城市向外显影。
路线不是纯随机，而是从预制池抽出并受资源、主线和保底修正。
锁定不是惩罚玩家猜谜，而是明确展示目标和缺口。
保底不是降低难度，而是保证玩家始终至少有一个可理解选择。
```

当前 WebDemo 的数据结构已经能支撑第一版。C 线程只需在 `discoveredLocations` 之外增加少量“雾中”和“路线显现”状态，先把羊皮纸九州图的显隐逻辑做稳，再考虑更复杂的地图生成。

## 11. BSI-UX-010 补充接收

已接收 `主协调_BSI-UX-010_全线程持续推进指令.md`。A 线程不等待 B/D/C/E/F 的进一步产物，先按规则与数值边界继续给出稳定口径。

给 C 线程的最低接收清单：

2. 节点状态先派生：当前、已见、可达、可达但锁定、雾中、未入图。
