## 1. 技术目标

第一阶段 Web Demo 的目标不是做正式游戏工程，而是做一个能验证核心循环的网页版测试原型：

```text
选路 -> 消耗资源 -> 触发事件 -> 做选择 -> 更新状态 -> 写入日志 -> 继续远行
```

技术优先级：

1. 数据驱动：地点、路线、事件、器物尽量从结构化数据读取。
2. 逻辑清楚：三资源、事件条件、选项结果、神志文本切换要能被看懂和测试。
3. 轻量可改：先服务策划和内容验证，不追求复杂工程框架。
4. 可复盘：每次选择写入日志，失败时能看出主要原因。
5. 可扩展：后续能接入内容校验、自动平衡模拟器、简单事件编辑器。

## 2. 推荐 Web Demo 技术栈

### 2.1 第一阶段推荐

| 模块 | 建议 | 原因 |
|---|---|---|
| 前端框架 | React | 适合快速搭建状态驱动 UI，团队协作成本低 |
| 类型系统 | TypeScript | 地点、路线、事件字段较多，类型能减少内容错误 |
| 构建工具 | Vite | 轻量，启动快，适合原型 |
| 数据格式 | JSON | 内容人员和工具链都容易读写 |
| 状态管理 | React 内置状态 | 第一版状态规模小，不需要 Redux/Zustand |
| 存档 | localStorage | 当前只需要本地单机验证 |
| 地图/旅途表现 | React + CSS/SVG 横向山海旅途舞台 | 第一版优先采用假动态横屏长图，而不是传统节点图作为唯一默认 |
| 旅途舞台层 | CSS 为主，SVG 辅助，Canvas 后置 | 只做前景后景、云影、昼夜色调、车队推进等氛围表现，不承载碰撞或实时操作 |
| 校验/模拟 | TypeScript 脚本或 Node 脚本 | 后续直接复用同一套数据结构 |

### 2.2 暂不建议第一阶段引入

| 工具 | 暂缓原因 |
|---|---|
| React Flow / xyflow | 第一版不需要传统节点编辑/拖拽地图；如果后续做内部地图编辑器再评估 |
| inkjs | 适合复杂叙事分支，但第一版事件结构自研更透明 |
| Redux / MobX / Zustand | 当前状态规模不大，额外概念会增加原型成本 |
| UI 组件库 | 容易把时间花在外观组件上，第一阶段应优先跑通玩法 |
| 游戏引擎 | 当前本质是文字选择 + 节点/路线数据 + 资源状态机，Web 足够 |
| Phaser 等 2D 游戏引擎 | 当前不需要碰撞、物理、实时操作和关卡对象生命周期，引入后复杂度大于收益 |

## 4. 核心数据结构

以下是字段草案，不是最终实现代码。字段命名建议保持英文，内容值可以使用中文。

### 4.1 Location

地点是地图节点，也是事件触发和路线连接的基础。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 地点唯一 ID，例如 `central_post` |
| name | string | 显示名称，例如 `中原驿` |
| region | string | 所属区域，例如 `人间边境`、`大荒西` |
| terrain | string[] | 地貌标签，例如 `泽`、`山`、`异族国` |
| danger | number | 危险等级，建议 1-5 |
| eventPool | string[] | 可触发事件 ID 列表 |
| visibleByDefault | boolean | 开局是否可见 |
| unlockConditions | Condition[] | 解锁条件 |
| sanityWeight | number | 神志污染权重，用于事件和文本变化 |

### 4.2 Route

路线连接两个地点，并承担移动消耗与风险提示。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 路线唯一 ID |
| from | string | 起点 Location ID |
| to | string | 终点 Location ID |
| name | string | 路线名称，例如 `北路`、`夜渡` |
| cost | ResourceDelta | 移动消耗 |
| terrain | string[] | 路线地貌 |
| riskLevel | number | 风险等级，建议 1-5 |
| eventPool | string[] | 路线途中可能触发的事件 |
| conditions | Condition[] | 可见或可走条件 |
| hidden | boolean | 是否隐藏路线 |

### 4.3 Event

事件是内容主引擎，负责文本、选择、结果和解锁。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 事件唯一 ID |
| title | string | 内部标题或简短显示标题 |
| location | string | 绑定 Location ID，可为空表示通用事件 |
| trigger | TriggerRule | 触发规则 |
| texts | SanityTexts | 高/中/低神志文本 |
| choices | Choice[] | 可选行动 |
| weight | number | 抽取权重 |
| once | boolean | 是否一局只触发一次 |
| tags | string[] | 事件标签，例如 `异兽`、`补给`、`幻觉` |

文本建议至少支持三档：

| 字段 | 说明 |
|---|---|
| clear | 高神志文本，冷静、古雅、信息可靠 |
| uneasy | 中神志文本，征兆增加、主观感增强 |
| mad | 低神志文本，语义断裂、但仍保留可玩线索 |

### 4.4 Choice

选项代表玩家在事件中的行动。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 选项唯一 ID |
| label | string | 玩家看到的选项文案 |
| conditions | Condition[] | 显示或可用条件 |
| cost | ResourceDelta | 立即消耗 |
| effects | ChoiceEffect | 结果变化 |
| resultText | string | 选择后的反馈文本 |
| riskHint | string | 可选，给玩家的风险提示 |

ChoiceEffect 建议覆盖：

| 字段 | 说明 |
|---|---|
| resources | 改变车轴、粮、神志 |
| addFlags | 增加状态标记 |
| removeFlags | 移除状态标记 |
| unlockRoutes | 解锁隐藏路线 |
| addItems | 获得器物 |
| removeItems | 失去器物 |
| languageProgress | 改变语言理解度 |
| triggerEvent | 指向后续事件 |
| ending | 进入结局 |

### 4.5 GameState

GameState 是当前局的完整状态，也是存档和模拟器的输入。

| 字段 | 类型 | 说明 |
|---|---|---|
| runId | string | 当前局 ID |
| day | number | 行旅天数或步数 |
| currentLocation | string | 当前地点 ID |
| resources | Resources | 当前车轴、粮、神志 |
| maxResources | Resources | 资源上限 |
| knownLocations | string[] | 已知地点 |
| unlockedRoutes | string[] | 已解锁路线 |
| inventory | string[] | 当前器物 ID |
| flags | string[] | 事件和剧情标记 |
| languageProgress | Record<string, number> | 各语言理解度，建议 0-100 |
| seenEvents | string[] | 已见事件 |
| routeHistory | string[] | 已走路线 |
| log | LogEntry[] | 行旅日志 |
| status | string | `playing`、`dead`、`ended`、`stranded` 等 |
| endingId | string | 已进入结局时记录结局 ID |

### 4.6 Item

器物不做复杂背包，第一阶段只作为改变规则的小型模块。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 器物唯一 ID |
| name | string | 器物名称 |
| type | string | `travel`、`survival`、`spirit`、`beast`、`map` |
| description | string | 玩家可读说明 |
| effects | ItemEffect[] | 对路线、事件、资源或文本的影响 |
| charges | number | 可选，使用次数 |
| tags | string[] | 关联标签 |

第一阶段建议只做 3 类器物效果：

1. 降低特定地貌路线消耗。
2. 降低特定事件的神志损耗。
3. 解锁一个隐藏选项或隐藏路线。

## 5. 第一版 UI 组成

第一版 UI 应是可玩的测试工作台，不做营销页。主表现方向优先采用“假动态横屏长图 / 横向山海旅途舞台”：屏幕上看到的是一小段可演示路程，前景和后景分层，云、昼夜、风景和车队有轻量动态；底层逻辑仍然是节点、路线、事件和资源。

### 5.1 桌面布局

```text
┌──────────────────────────────────────────────┐
│ 顶部：当前地点 / 天数 / 当前状态              │
├──────────────────────────────────────────────┤
│ 横向山海旅途舞台：远山 / 云影 / 昼夜 / 车队推进 │
├──────────────┬────────────────┬──────────────┤
│ 路线区        │ 事件区          │ 资源区        │
│              │                │              │
│ 当前段        │ 当前事件文本      │ 车轴          │
│ 可选下一段    │ 选项按钮          │ 粮            │
│ 消耗/风险     │ 结果反馈          │ 神志          │
│              │                │ 器物/语言      │
├──────────────┴────────────────┴──────────────┤
│ 日志区：最近路线、事件、资源变化、失败原因      │
└──────────────────────────────────────────────┘
```

`JourneyStage` 是第一版优先视觉方案，不是可选装饰。传统节点图可以作为开发期辅助信息，或在路线区以简化列表出现。

### 5.2 横向山海旅途舞台

```text
┌──────────────────────────────────────────────┐
│ 分段长图：中原驿 ---- 雷泽 ---- 青丘 ---- 赤水 │
│ 远景：山势/云/日月色调                         │
│ 中景：泽水/林线/渡口/荒原                       │
│ 前景：草木/车辙/风沙/车队剪影                   │
└──────────────────────────────────────────────┘
```

第一版推荐实现方式：

| 层级 | 推荐技术 | 说明 |
|---|---|---|
| React 状态层 | React + TypeScript | 根据 currentLocation、selectedRoute、transitionState 控制舞台状态 |
| 布局和动画 | CSS transform / transition / keyframes | 负责车队从当前段移动到下一段、背景轻微滚动、云漂移 |
| 地貌形状 | SVG | 画远山、泽水、渡口、树线、车队剪影，易于响应式缩放 |
| 色调变化 | CSS 变量 + 叠层渐变 | 根据 day、sanity、terrain 改变昼夜、雾气、污染感 |
| Canvas | 后置可选 | 只有在需要大量粒子、连续绘制或复杂天气时再考虑 |

结论：第一版最适合用 **React + CSS + SVG**。Canvas 暂不作为默认，因为第一版的需求是分段推进和轻量氛围，不需要逐帧游戏循环。

### 5.3 节点路线如何映射为横向分段

底层仍然保留 Location / Route 数据结构，但显示时把路线映射成横向舞台分段：

```text
Location A -> Route AB -> Location B
显示为：
[当前地标 A] ---- [路线段 AB：山/泽/荒原表现] ---- [下一地标 B]
```

映射规则建议：

| 数据 | 舞台表现 |
|---|---|
| Location.name | 舞台段起点/终点地标名称 |
| Location.region | 背景大色调和远景轮廓 |
| Route.terrain | 路面、前景、风景元素 |
| Route.cost | 路线区显示消耗，不由舞台动画计算 |
| Route.riskLevel | 舞台色调、云速、风沙强弱等氛围提示 |
| GameState.day | 昼夜或天光变化 |
| GameState.resources.sanity | 低神志时可让远景错位、颜色偏移、地名变形 |

推进流程：

1. 玩家在路线区选择一条 Route。
2. 系统先用 routeSystem 判断是否可走，并扣除资源。
3. `JourneyStage` 进入 `traveling` 状态。
4. 车队剪影从当前段左侧/中段移动到下一段。
5. 前后景做轻微反向滚动，制造行进感。
6. 动画结束后更新 currentLocation。
7. eventSystem 触发文字事件，事件区出现文本和选项。

舞台动画只是结果表达，不决定移动是否合法。

### 5.4 路线区

第一版仍需要路线区，避免玩家只看风景而不知道策略信息：

- 当前所在地点。
- 可选择的下一段路线。
- 每条路线消耗多少车轴、粮、神志。
- 路线风险等级。
- 路线地貌标签。
- 隐藏路线解锁后出现。

显示方式建议：

- MVP：路线卡片 + 横向舞台当前段高亮。
- 第二步：舞台下方加小型进度线，表示已走过的地点。
- 第三步：如需要编辑完整地图，再评估内部节点图或 React Flow。

### 5.5 轻量旅途舞台层边界

本项目可以借鉴“旅途横向背景滚动”的氛围表达，但必须把它定义为轻量旅途舞台层，而不是横版场景玩法。

明确不做：

- 不做碰撞系统。
- 不做物理系统。
- 不做跳跃、攻击、闪避等实时操作。
- 不让玩家直接操控角色在场景里移动。
- 不把路线判断、事件触发、资源扣减交给舞台层。

游戏逻辑仍然只来自：

- 节点和路线。
- 事件和选项。
- 车轴、粮、神志。
- flag、器物、语言理解、结局条件。

舞台层只负责让“正在远行”的感觉更直观。推荐实现：

| 表现 | 推荐技术 | 说明 |
|---|---|---|
| 前景和后景分层 | CSS transform / SVG group | 远山慢、近草快，形成轻微视差 |
| 白云漂移 | CSS animation | 低成本，适合第一版 |
| 日出日落 | CSS 变量 / 渐变叠层 | 根据 day、路线类型或事件阶段切换色调 |
| 路上风景变化 | SVG / 简单图片切片 | 按 terrain 标签切换山、泽、荒原、渡口等轮廓 |
| 车队剪影 | SVG / CSS | 只做状态表达，不做可控角色 |
| 段落推进 | CSS transition | route 选择后触发一次短动画，再进入事件 |

第一版建议做：

```text
车队从当前段移动到下一段 + 背景轻微滚动 + 触发文字事件
```

理由：

1. 成本低，不会改变核心架构。
2. 能补足“远行感”，但不会误导成横版动作游戏。
3. 可以直接由当前地点、路线地貌、天数、神志状态驱动画面色调。
4. 后续如果视觉方向成立，再替换为更精致的 SVG 或 Canvas 舞台。

技术判断：

- CSS 足够处理云、昼夜、简单视差和车队位移。
- SVG 适合画山海图轮廓、车队剪影和地貌符号。
- Canvas 适合后续做更连续的背景滚动，但第一版不是必需。
- Phaser 等游戏引擎暂不建议引入；除非后续确定要大量精灵、粒子、复杂摄像机、统一动画时间轴，并且这些收益明显大于工程复杂度。

### 5.6 事件区

事件区是 Demo 核心。

需要显示：

- 当前地点名。
- 当前事件标题。
- 根据神志区间选择的事件文本。
- 可用选项。
- 被条件锁住的选项可以暂不显示，或显示为不可用并说明缺少条件。
- 选择后的短反馈。

神志文本切换规则建议：

| 神志 | 文本档 |
|---:|---|
| 70-100 | clear |
| 35-69 | uneasy |
| 0-34 | mad |

### 5.7 资源区

资源区必须让玩家快速区分三种压力：

- 车轴：还能不能继续走。
- 粮：还能不能撑过今天。
- 神志：看到的世界是否可靠。

显示建议：

- 三个数值条。
- 当前器物列表。
- 语言理解度。
- 当前危险状态，例如 `饥荒`、`车坏`、`失心边缘`。

### 5.8 日志区

日志用于复盘，不追求长篇叙事。

每条日志建议包含：

- 天数或步数。
- 地点。
- 行动。
- 资源变化。
- 关键 flag 或结局触发。

失败时，日志末尾需要能归因，例如：

- 粮尽：连续绕路且未补给。
- 车轴尽：多次强渡或山路损耗过高。
- 神志崩坏：频繁阅读巫文或夜行。

## 6. 本地存档方式

第一阶段使用 `localStorage` 即可。

建议存三类数据：

| 存档项 | 内容 |
|---|---|
| currentRun | 当前局 GameState |
| metaProgress | 已见事件、已达成结局、已解锁词条 |
| settings | 文本速度、是否显示风险提示等轻量设置 |

存档策略：

1. 每次移动后保存。
2. 每次事件选择后保存。
3. 达成结局或失败后保存到历史记录。
4. 新开局时清空 currentRun，但保留 metaProgress。

风险与处理：

- localStorage 容量有限，但第一阶段数据量足够。
- 存档结构变更时需要版本号，例如 `saveVersion`。
- 读档失败时应回到新开局，不应让页面空白。

## 7. 自动平衡模拟器如何连接数据结构

模拟器不需要单独发明一套规则，应该直接调用和 Demo 相同的数据结构与核心规则。

### 7.1 输入

模拟器读取：

- locations.json
- routes.json
- events.json
- items.json
- 初始 GameState
- 玩家策略配置

### 7.2 复用规则

模拟器应复用：

- routeSystem：选择路线、计算移动消耗。
- resourceSystem：扣资源、判断濒死或失败。
- eventSystem：根据地点、flag、神志、权重抽事件。
- sanityTextSystem：模拟阶段可不渲染文本，但要记录文本档位。

### 7.3 第一版模拟玩家

第一版只需要 3 类策略：

| 策略 | 行为 |
|---|---|
| random | 随机选路线、随机选项 |
| cautious | 优先低风险、低消耗、补资源 |
| explorer | 优先未知地点、隐藏路线、高收益选项 |

后续再扩展为产品文档中的 5 类玩家：保守、冒险、贪心、归家、随机。

### 7.4 输出指标

第一版记录：

- 平均存活步数。
- 失败原因分布。
- 三资源归零次数。
- 通关率。
- 进入结局比例。
- 连续坏事件次数。
- 是否出现无合法路线但未进入结局的死局。

### 7.5 与内容校验的关系

数据校验先保证“结构正确”，模拟器再验证“体验是否合理”。

校验例子：

- route.from / route.to 必须存在。
- event.location 必须存在。
- choice.effects.unlockRoutes 必须指向存在路线。
- 每个普通地点至少有一条可达路线或明确结局状态。
- 每个事件至少有一个可用 Choice。

## 8. 开源依赖建议和许可证风险

### 8.1 推荐低风险依赖

| 依赖 | 用途 | 许可证风险 |
|---|---|---|
| React | UI | 低，MIT |
| React DOM | UI 渲染 | 低，MIT |
| TypeScript | 类型系统 | 低，Apache-2.0 |
| Vite | 构建工具 | 低，MIT |

### 8.2 可选依赖

| 依赖 | 用途 | 建议 | 风险 |
|---|---|---|---|
| React Flow / xyflow | 内部地图编辑器或传统节点图 | 第一版不引入 | 低，MIT，但会把默认方向带回节点图 |
| inkjs | 复杂文本分支 | 暂不引入 | 低，MIT，但会改变内容格式 |
| rot.js | 随机与 Roguelike 工具 | 当前不需要 | 低，BSD-3-Clause |
| Phaser | 2D 游戏引擎 | 第一阶段不引入 | 许可证通常可控，但复杂度和玩法误判风险较高 |

### 8.3 禁止或不建议

| 对象 | 结论 | 原因 |
|---|---|---|
| 许可证不明的小型游戏库 | 禁止 | 商业风险不可控 |
| 直接 fork 完整开源游戏 | 禁止 | 容易变成换皮，也可能带来开源义务 |
| ChoiceScript | 不建议 | 商业许可证限制明显 |
| A Dark Room 代码或表达 | 不直接使用 | MPL-2.0 有文件级开源义务，且表达相似风险较高 |
| 把旅途舞台做成横版动作玩法 | 不建议 | 会引入碰撞、物理、实时操作，偏离节点/路线/事件/资源核心 |
| 现代译本、百科文案、插画素材 | 禁止复制 | 资料版权和表达相似风险高 |

原则：

```text
核心玩法逻辑自研；显示、构建、工具层可用 MIT/BSD/Apache 等宽松许可证依赖。
```

## 9. 第一阶段实现顺序

在主线程批准“搭建工程”后，建议按以下顺序执行。

### 第 1 步：建立最小工程骨架

- React + TypeScript + Vite。
- 不引入 UI 库。
- 只放一个主界面。
- 默认加入 `JourneyStage` 横向山海旅途舞台。
- 舞台只做假动态横屏长图、分层背景、车队剪影、云和昼夜色调。

完成标准：

- 页面能打开。
- 显示五个区域：旅途舞台、路线区、事件区、资源区、日志区。
- 旅途舞台层不影响路线选择、事件触发和资源计算。

### 第 2 步：定义类型和样例数据

- 先做 5 个地点。
- 6-8 条路线。
- 6-10 个事件。
- 2-3 个器物。

完成标准：

- 数据能被读取并显示。
- 当前地点和可走路线能出现。

### 第 3 步：实现核心循环

- 选择路线。
- 扣资源。
- 进入短暂 `traveling` 状态。
- 驱动车队从当前段移动到下一段。
- 动画结束后更新当前地点。
- 触发事件。
- 执行选项结果。
- 写入日志。

完成标准：

- 玩家可以连续走 5-10 步。
- 资源变化清楚。
- 事件选择能改变状态。
- 每次移动能看到一次轻量横向推进表现，但不需要实时操控。

### 第 4 步：接入神志文本系统

- 根据神志显示 clear / uneasy / mad。
- 低神志可以显示隐藏选项或隐藏路线。
- 低神志时允许旅途舞台出现轻微错位、色偏、地名扰动，但不影响真实路线判定。

完成标准：

- 同一事件在不同神志下文本明显不同。
- 玩家能感到世界状态变化。

### 第 5 步：接入本地存档

- 保存当前局。
- 刷新恢复。
- 新开局。
- 记录已见事件和结局。

完成标准：

- 刷新页面不丢当前局。
- 新开局不会污染旧的历史记录。

### 第 6 步：接入数据校验

- 检查地点、路线、事件引用。
- 检查缺失文本和缺失选项。
- 检查无出口死局。

完成标准：

- 内容错误能在开发阶段被发现，而不是运行时页面崩溃。

### 第 7 步：接入自动平衡模拟器

- 先跑 random / cautious / explorer 三类玩家。
- 每类 100-1000 局。
- 输出存活步数、失败原因、通关率、死局率。

完成标准：

- 能发现明显数值问题。
- 能证明“随机生成世界，不能随机生成死局”的检查路径。

## 10. 第一阶段验收建议

技术侧完成后，不以“页面好看”为主要验收标准，而以玩法验证为主：

1. 玩家能在 10 分钟内理解车轴、粮、神志的区别。
2. 玩家能清楚知道当前在哪、能去哪、要付出什么。
3. 至少一次神志降低后，文本或选项明显变化。
4. 失败后日志能解释主要失败原因。
5. 数据错误能通过校验工具提前暴露。
6. 横向山海旅途舞台能表达一小段路程推进，但没有碰撞、物理或实时操作误导。

## 11. 需要用户决策的问题

1. 第一版 Demo 是否只做固定地图，还是需要加入轻量随机路线？建议先固定地图。
2. 第一版地点规模采用 5-8 个测试点，还是直接按 20 个地点 MVP 目标建表？建议先 5-8 个。
3. 横向旅途舞台第一版更偏“水墨剪影”还是“帛书长卷”？建议先用水墨剪影，制作成本更低。
4. 失败机制第一版是资源归零即失败，还是先做濒死选择？建议至少保留一个简单濒死选择。
5. 第一版是否需要移动端适配？建议先桌面可玩，移动端用简单纵向堆叠兜底。
