## 3. 阶段计划

### 阶段 P0：当前试玩版收口

目标：让网页版 demo 不再只是“能跑”，而是能给人完整试玩 5 到 8 分钟。

预计剩余工作量：1 到 2 个小迭代。

必须完成：

- 完整试玩路径复核：从起点到九州裂隙至少跑通一条主路径。844x390 真实点击路径已通过。
- 小横屏和桌面 UI 复核：重点看路线卡、地点志、补给区、选择区是否一屏可读。844x390 和 1365x768 已通过；试玩入口也已按 844x390 与 1365x768 复验为一屏启动器。
- 文本入口收口：默认短文本，详细地点志和事件详情进入弹窗或抽屉；遭遇层已拆成短摘要和详情全文。
- 状态反馈收口：资源涨跌、告急、补给完成、半途路遇、见闻入志、地图显影和新结局收录要能被玩家明确感知；顶部“下一步”提示已覆盖旅途页、补给前和地图页，且落账后不再残留“等落账”；章节进度带已显示整局 6 段进度；行动执行中旧选择列表会收起，避免误点和误以为卡住；结局页已能给出本局摘要、复盘和重开。
- 音乐与音效入口可见：第一版允许使用待复核素材，但必须明确标记临时状态；入口页已显示“声音状态”，音乐 6 首待听、环境/神志/短音仍按临时处理；P0 音频听感验收工作台已可逐条试听并复制摘要，后续补人工主观结论。
- 旅途环境层可感知：当前已接入 `AMB-TRAVEL-001-SYN` 程序占位，状态为 `demo-temporary`，后续需由真实或高质量生成素材替换。
- QA 脚本继续守住：路线可达、资源可活、补给可预览、素材引用、音频 hook 都必须过。
- 批量平衡模拟继续守住：默认、谨慎、冒险和失误玩家四类策略都不能出现早期随机死局；当前 1000 局报告门槛 8/8。

验收标准：

- 新玩家不看说明，也能知道先处理地点事件，再补给，再选择下一站；当前已由 HUD 下一步提示缓解，并建立 P0 真人试玩验收工作台用于复核。
- 所有可点击选择一次性显示完，不出现关键选择被滚动条藏住。
- 一局不会因为随机过坏直接死局。
- 路线选择能看到路费、风险、路遇预感和目的地补给。

### 阶段 P1：第一章内容扩展

目标：把当前测试骨架扩成更像“第一章”的内容量。

预计剩余工作量：5 到 8 个小迭代。

必须完成：

- 地点志扩写：每个地点都有“是什么、为什么危险、有什么风俗/异象”；当前已先保证所有地点有抵达句和基础地点志。
- 路遇事件扩写：固定路遇和随机路遇分层，避免每条路都像同一种文本。
- 低神志版本：神志低时地点志、路遇、日志逐步变得不可靠和恐怖。
- 成就感反馈：抵达关键地点、发现异象、保住资源、走出险路时给轻量反馈。
- 补给差异深化：每城资源倾向更明确，有些偏车轴，有些偏粮草，有些偏神志。
- 《山野逸事》试运行：`B_BSI-B-035` 已建立第一批内容试运行任务包，`BSI-B-036` 已补上内容策划 AI 问答工作台和一键复制 Prompt；下一步等内容协作者提交 3 到 5 个旧设定、故事桥段或典故来源，AI 初读并追问，主线程挑 1 到 2 条转成 Web Demo 事件。

验收标准：

- 玩家每到一个新地点都能感到“这是山海经九州的一处地方”，不是普通节点。
- 至少有 3 类明显不同的地点体验：墟市、祭所、禁地/异象地。
- 神志低时文本变化足够明显，但不影响核心操作理解。

### 阶段 P2：数值与随机稳定

目标：把“好玩但不死局”的平衡从经验判断变成可测数据。

预计剩余工作量：3 到 6 个小迭代。

必须完成：

- 维护自动模拟器：批量跑 100 到 1000 局，统计死亡原因、通关率、告警线、补救次数和失败阶段；当前状态报告已写入 P0/P1 门槛表。
- 随机事件权重表：区分固定路遇、随机路遇、补救事件和稀有异象。
- 保底规则明确：低资源时提高补救机会，但不让玩家感觉被强行救活。
- 资源曲线调整：车轴、粮草、神志各自承担不同压力，不互相重复。

验收标准：

- 正常策略下主路径通关率稳定。
- 错误选择会带来压力，但不会前几步随机死局；当前 `novice` 策略 1000 局成功率 100%，`early_0_3` 为 0。
- 三资源都重要，且每种资源都有对应的恐惧感和策略意义。

### 阶段 P3：视觉与音频正式化

目标：让第一版不再依赖混乱的临时素材，而是有可追踪、可替换、可发布的资产体系。

预计剩余工作量：5 到 10 个小迭代，取决于生图/视频质量。

必须完成：

- 背景图分类：地图、地点插图、横向舞台背景、前景遮挡、污染层、车队层分开。
- 车队素材替换：首版已从 `CAR-005` 替换为 `CAR-013` 透明车队层；当前默认改为风景主导行程舞台，车队只作弱状态标记，正式版前再决定是否恢复为主行动层。
- 视频策略确定：只在必要位置使用短循环或神志异象，不让视频拖累性能。
- 音频包确定：`MUS-CORE-001/003/004` 已分别接入古道、水泽、废墟/裂隙临时轮换；`MUS-CORE-005/006` 已接入地点级临时轮换；后续继续确定音乐、环境声、地图短音效、资源告急、神志污染分层。
- 授权台账：每个素材标记来源、用途、状态和是否可正式使用。

验收标准：

- 背景不带车队；P0 默认风景主导，车队只作独立弱状态层，避免贴地不准时破坏沉浸感。
- 音频可关闭，默认不惊扰，但能增强告急和异象。
- 素材来源清楚，临时素材不会误标成正式可商用。

### 阶段 P4：工程化与版本发布

目标：把静态原型变成可维护、可同步、可发布的试玩项目。

预计剩余工作量：4 到 8 个小迭代。

必须完成：

- 是否迁移到 React + TypeScript + Vite：静态原型稳定后再迁移。
- 数据拆分：地点、路线、事件、补给、素材、音频从大文件拆出。
- 存档与重开：至少支持本地重开、继续和简单复盘。
- 版本包：形成 v0.1 试玩包，能本地打开或部署到网页。
- GitHub 同步：确认远端仓库、分支、提交和发布记录。

验收标准：

- 非开发者可以打开一个链接试玩。
- 每次版本变化都有可读更新说明。
- 后续新增地点、事件、素材不需要大改代码。

### 阶段 P5：发布前风险复核

目标：在对外试玩前避免明显商业和授权风险。

预计剩余工作量：2 到 4 个小迭代。

必须完成：

- 竞品相似度复核：避免复刻 Out There 或其他同类游戏的具体表达。
- 素材授权复核：AI 生成、外部素材、音乐音效全部标状态。
- 文案风险复核：商店文案、宣传语、截图说明不提“借鉴某游戏玩法”。
- 开源依赖复核：如后续引入库，确认 license 和商用边界。

验收标准：

- 玩法借鉴停留在抽象机制层，不复制 UI、文案、数值表、图像风格和具体关卡。
- 素材可追溯。
- 对外版本不会把临时素材当正式资产发布。
