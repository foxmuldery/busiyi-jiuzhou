# D_BSI-D-005 移动游戏化界面方案

> 子线程：D 视觉体验与交互氛围设计  
> 对应主协调任务：BSI-UX-006  
> 项目：《不思异：九州》Web Demo  
> 日期：2026-06-14  
> 边界：本文件只做体验方案与验收标准，不修改 `index.html`、`styles.css`、`app.js`

## 0. 结论

当前 Demo 的玩法结构已经能支撑“地点 -> 九州图 -> 选路 -> 半途路遇 -> 抵达地点”的闭环，但第一眼太像网页工作台。下一步不要重写玩法，也不要引入新框架；优先把现有内容收进一个竖屏手机游戏壳：

```text
顶部 HUD：日数 / 当前地点 / 三资源 / 距九州裂隙
中部主画面：地点舞台或九州路线图
底部操作区：当前事件、补给、路线卡、确认选择
底部导航：地点 / 九州图 / 行囊 / 日志
```

桌面浏览器只作为手机预览器，主体宽度固定在 390-430px 左右居中显示。不要再把桌面宽屏当成主体验。

## 1. 当前“网页感 / 桌面工作台感”的来源

### 1.1 结构层面

1. `.app-shell` 当前宽度是 `min(1440px, calc(100vw - 32px))`，天然会铺满桌面浏览器，第一眼像网页应用。
2. `topbar`、`resource-strip`、`journey-overview`、`town-grid`、`log-panel` 依次铺开，像一套仪表盘，而不是手机游戏的一屏一事。
3. `resource-strip` 用三张大资源卡横排展示，信息很清楚，但占了手机游戏 HUD 应该更轻的位置。
4. `journey-overview` 同时承担页面切换、进度条、当前位置、可往下一站提示，像网页页签和数据面板。
5. `town-grid` 把事件面板和 `status-panel` 并排；行囊、语言、困厄、音乐、动态层都常驻右侧，像调试/状态侧栏。
6. `log-panel` 常驻在地点页下方，容易让主界面变成记录列表。
7. `map-layout` 是地图和路线列表左右两栏，适合桌面分析，不像手机上的“上方小地图 + 下方可选路线卡”。
8. 顶部 `project-label` 写着 `Web Demo Prototype`，会直接提示玩家“这是网页原型”，应从游戏主界面移除或折叠到关于/设置。

### 1.2 视觉层面

1. `.panel`、`.resource-card`、`.route-card`、`.choice-card` 的边框、阴影、分区很多，叠加后更像后台卡片系统。
2. 大量区块标题同时出现：地点志、补给、行囊、语言、困厄、音乐、行旅日志、九州图、下一站。玩家会先读结构，而不是进入旅程。
3. 地点舞台已有游戏感，但下面马上接信息密集的面板网格，削弱了“手机游戏主画面”的第一印象。
4. 地图节点卡片在大画布上散布，节点本身携带风险、消耗、补给文字，像策略编辑器；移动端应把地图降为氛围/定位，把路线卡作为决策主体。

### 1.3 交互层面

1. 现在只有 `地点 / 九州图` 两个页签，行囊和日志不是页面，而是地点页的常驻内容。
2. 音乐、动态层、视频素材状态这些设置类内容出现在主玩法路径里，会让 Demo 显得像开发调试台。
3. 玩家处理完事件后，需要理解“切到九州图继续”，但页面切换控件在上方，不像手机游戏常见的底部导航。
4. 路线选择后虽然已有 `move()` 和半途逻辑，但视觉上没有形成“底部点路线 -> 中部行旅反馈 -> 底部出现路遇选择”的手机流程。

## 2. 竖屏手机游戏主结构

### 2.1 页面骨架

建议 C 线程建立或等价模拟一个手机画幅容器：

```text
.game-phone-shell
  .game-hud
  .game-main
    当前 view-panel
  .game-action-dock
    当前页主要操作
  .game-bottom-nav
```

如果不想大改 HTML，也可以先把现有 `.app-shell` 直接改成手机壳：

- 桌面：`.app-shell { width: min(430px, 100vw); min-height: 100svh; margin: 0 auto; }`
- 390/430 竖屏：`.app-shell { width: 100vw; min-height: 100svh; padding: 0; }`
- `body` 只负责居中和预览背景，不再承载宽屏布局。

### 2.2 顶部 HUD

HUD 只放玩家必须随时知道的信息：

```text
左：第 X 日 / 当前地点
中：车轴 80 · 粮草 90 · 神志 85
右：距九州裂隙 4 段
```

落地建议：

1. 复用现有 `#dayLabel`、`#locationTitle`、`#axleValue`、`#grainValue`、`#sanityValue`、`#distanceLabel`。
2. 把三张 `.resource-card` 改成 HUD 小胶囊，不再显示大卡片式 `small` 提示；危险时用颜色和轻微闪边表达。
3. `#regionLabel` 可以保留为当前地点副信息，但不要和日数、资源抢一行；空间不足时只在地点页标题区显示。
4. `#newRunButton` 不应常驻 HUD，可移到日志页、设置抽屉，或在失败/结局状态才出现。
5. HUD 高度建议控制在 64-84px 内。390x844 下不能挤掉主画面。

### 2.3 中部主画面

中部只展示当前页的主体验：

1. 地点页：横向行旅舞台 + 当前地点/半途状态。
2. 九州图页：小地图 / 路线节点图，负责让玩家确认当前位置和方向。
3. 行囊页：九宫格、语言、困厄，不插入地点事件。
4. 日志页：最近记录与复盘，不抢主玩法。

地点舞台建议高度：

- 390x844：约 220-260px。
- 430x932：约 250-300px。
- 不要用 `33vw` 控制手机高度；手机端应按视口高度或固定范围控制。

### 2.4 底部操作区

底部操作区是当前页面的主要行动，不是额外说明区：

1. 地点页停靠状态：显示当前事件文本、选择按钮、补给入口或补给卡。
2. 地点页半途状态：显示路遇标题、路遇文本、路遇选择；补给只显示禁用原因。
3. 九州图页：显示可达路线卡，每张卡包含目的地、消耗、风险、半途路遇、目的地可观察补给。
4. 行囊页：显示物品格、语言/线索、困厄统计；一般没有主要行动。
5. 日志页：显示日志、重开入口、可选的清理/回看入口。

底部操作区要求：

- 按钮最小高度 48px，路线卡和选择卡建议 56-72px。
- 390px 宽度下所有文字完整可读，不能依赖 hover。
- 重要按钮放在大拇指自然触达区，避免主操作出现在页面最上方。

### 2.5 底部导航

底部导航固定 4 个入口：

```text
地点 / 九州图 / 行囊 / 日志
```

职责：

1. 替代当前上方 `.view-tabs` 的主导航位置。
2. 始终固定在底部，或者随 `.app-shell` 底部停靠。
3. 当前页高亮，半途路遇时“地点”可加轻提示点。
4. 日志有新记录时“日志”可加小点，但不要弹出完整日志。

第一版不需要图标库。可以先用文字按钮完成，但样式必须像游戏底栏，不像网页 Tab。

## 3. 四个页面的职责边界

| 页面 | 第一任务 | 应展示 | 不应展示 |
| --- | --- | --- | --- |
| 地点页 | 处理当前停靠 / 半途发生的事 | 地点舞台、当前地点名或半途状态、地点志摘要、补给、当前事件、选择按钮 | 完整行囊九宫格、完整日志、音乐/动态调试信息、完整九州路线列表 |
| 九州图页 | 决定下一段往哪里走 | 当前节点、小地图、距离终点、可达路线卡、锁定原因、目的地可观察补给 | 地点事件选择、城镇补给按钮、完整行囊、完整日志 |
| 行囊页 | 查看随身状态和长期变量 | 九宫格、线索、语言、困厄/求生统计、可能影响路线的条件提示 | 地点志正文、路线选择、补给操作、事件选择 |
| 日志页 | 复盘最近选择和资源变化 | 时间倒序日志、路线/补给/事件结果、失败原因、重开按钮 | 地图大画布、地点舞台、常驻路线卡、设置调试明细 |

补充边界：

1. 地点志可以在地点页显示摘要，完整长文以后可做弹层；第一版先不必新增长文页。
2. 补给属于地点页，不属于行囊页。行囊只显示已获得或长期状态。
3. 路线选择属于九州图页，不属于地点页。地点页事件处理完后只提示“去九州图选路”。
4. 半途路遇仍归地点页处理，因为玩家心理上是在“路上停下发生了事”，不是在地图上决策。

## 4. 手机端交互顺序

### 4.1 新局进入

1. 打开后默认进入“地点页”。
2. HUD 显示第 1 日、中原驿、三资源、距九州裂隙约 4 段路。
3. 中部显示中原驿行旅舞台。
4. 底部操作区显示地点志摘要、可搜补给、当前地点事件。
5. 底部导航显示“地点”为当前页。

### 4.2 地点补给

1. 玩家在地点页看到补给入口或补给卡。
2. 点击补给卡，例如“清点驿粮”。
3. 资源 HUD 立即更新，补给卡变为“已搜集”或置灰。
4. 最近日志只用一条轻提示反馈，不在地点页展开完整日志。
5. 玩家仍停留地点页，可以继续处理地点事件。

注意：补给在手机端可以放在事件文本上方或下方，但不能把事件选择挤到屏幕外很深的位置。若补给超过 2 个，建议折叠为“补给 2 项”抽屉。

### 4.3 地点事件选择

1. 玩家阅读当前事件文本。
2. 选择按钮紧贴事件文本下方。
3. 点击选择后，资源 HUD 更新，事件面板显示结果。
4. 若地点事件已处理，底部操作区只保留提示：“去九州图选择下一段旅程”。
5. 玩家点击底部导航“九州图”。

### 4.4 路线选择

1. 九州图页中部显示小地图，当前节点最强，可达节点次强，未来/锁定降权。
2. 底部操作区显示路线卡列表。
3. 每张路线卡必须包含：
   - 路线名和目的地。
   - 消耗：车轴、粮草、神志。
   - 风险：低/中/高。
   - 半途路遇：事件名或“无”。
   - 目的地可观察补给：偏车轴 / 偏粮草 / 偏神志 / 地利不明。
4. 点击可达路线卡后，立刻切回地点页或让地点页成为当前状态。
5. HUD 更新日数和资源，中部舞台给出行进反馈。

移动端第一版建议以路线卡点击为主。地图节点可以保留点击，但不要成为唯一操作入口。

### 4.5 半途路遇

1. 如果路线有 `midEvent` 且本轮触发，进入地点页的半途状态。
2. HUD 当前地点仍可显示出发点，但中部/底部必须明确写出“半途：起点 -> 目的地”。
3. 舞台车队处于 `interrupted` 状态，底部操作区显示路遇标题、路遇文本和选择。
4. 补给区只显示原因：“车队停在半途，无法搜集城镇补给。”不要显示可点击补给卡。
5. 玩家选择路遇处理方式后，写入日志，并自动或明确抵达目的地。
6. 抵达后地点页刷新为目的地的地点志、补给、事件。

### 4.6 日志查看

1. 主流程只显示最近一条反馈，例如“补给：粮草 +8”。
2. 玩家点击底部导航“日志”才看到完整 `state.log`。
3. 日志页可显示重开按钮，尤其在失败、结局、困厄状态下。
4. 不要让日志常驻地点页底部。

## 5. 桌面浏览器如何呈现手机游戏预览

桌面端目标不是“适配大屏”，而是“让桌面用户预览手机游戏”。

建议：

1. `body` 用深色背景居中承载手机壳，可保留九州氛围底色，但不要出现大面积后台布局。
2. `.app-shell` 或新增 `.game-phone-shell`：
   - 宽度：`min(430px, 100vw)`。
   - 最小高度：`100svh`。
   - 桌面可加 `max-height: 932px` 与轻微边框/阴影，模拟手机预览。
   - 小屏手机不要外边框，避免浪费宽度。
3. 桌面下不要恢复 `.town-grid` 两栏，也不要恢复 `.map-layout` 左右两栏。
4. 桌面若有剩余空间，可只做背景留白，不能把行囊、日志、地图路线摊到两边。
5. 390x844 和 430x932 是主验收尺寸；桌面 1440 宽只验收“是否居中手机预览”。

## 6. 禁止项

以下设计必须移除、折叠或转移，避免变成网页后台：

1. 禁止 1440px 宽屏主布局作为默认体验。
2. 禁止地点页常驻右侧 `status-panel`，行囊、语言、困厄必须移入“行囊页”。
3. 禁止地点页常驻完整 `log-panel`，完整日志必须移入“日志页”。
4. 禁止音乐、动态层、视频素材状态常驻主玩法区；这些只能进入设置/日志/折叠调试区。
5. 禁止资源以三张大卡片常驻占据首屏；三资源要变 HUD。
6. 禁止地图页在手机端左右两栏展示；路线卡应在小地图下方。
7. 禁止地点页同时承担路线决策；地点事件处理完只引导去九州图。
8. 禁止九州图页处理地点补给或地点事件。
9. 禁止依赖 hover 解释路线、补给、按钮含义。
10. 禁止项目标签、Demo 原型字样出现在游戏主界面第一视觉层。
11. 禁止一屏出现过多面板标题，尤其是“器物 / 音频 / 动态层”这类内部结构说明。
12. 禁止按钮小于 44px 高；核心选择建议 48px 以上。

## 7. 给 C 线程的可执行改造建议

### 7.1 HTML 结构建议

保持现有静态 Demo，不引入 React/Vite/新框架。可以在 `index.html` 做轻量结构重排：

1. 将 `<main class="app-shell">` 视为手机壳，或在其内新增 `.game-phone-shell`。
2. 将现有 `topbar` 和 `resource-strip` 合并成 `.game-hud` 视觉区。DOM 可不完全合并，先用 CSS grid/flex 让它们看起来是一个 HUD。
3. 将 `journey-overview` 的 `.view-tabs` 移到底部，改成 `.game-bottom-nav`。
4. 新增两个 `view-panel`：
   - `inventoryView`：承载当前 `status-panel` 中的行囊、语言、困厄。
   - `logView`：承载当前 `log-panel`。
5. `status-panel` 中的音乐/动态层第一版建议放到日志页底部的“设置”折叠区，或暂时折叠为一个 `<details>`。
6. 地点页 `.town-grid` 中只保留 `event-panel` 作为底部操作区；不要再并排放 `status-panel`。
7. 地图页 `.map-layout` 改为单列：上方 `.kyushu-map`，下方 `.map-route-panel`。

如果 C 线程想最小改动：

- 第一步只做 CSS：隐藏/折叠 `status-panel` 和 `log-panel`，`.app-shell` 改 430px，`.map-layout` 单列。
- 第二步再补四页底部导航。

### 7.2 CSS 建议

重点改这些选择器：

1. `.app-shell`
   - 从 `min(1440px, calc(100vw - 32px))` 改成 `min(430px, 100vw)`。
   - 移动端 `padding: 0` 或极小内边距。
2. `.topbar`
   - 从大标题栏改为紧凑 HUD。
   - `h1` 不再使用 `clamp(28px, 4vw, 48px)`，手机内建议 18-22px。
3. `.resource-strip`
   - 从三列大卡改为三枚横向小状态。
   - `.meter` 可以保留，但高度压到 3-4px。
4. `.journey-overview`
   - 进度与距离保留，页面切换按钮迁到底部。
   - `distance-board` 不要三列大板，可变成一条进度条。
5. `.journey-stage`
   - 手机端高度用 `clamp(220px, 31svh, 300px)` 一类规则。
   - 不要按桌面 `33vw` 计算。
6. `.town-grid`
   - 改为单列。
   - `status-panel` 不再参与地点页布局。
7. `.event-panel`
   - 作为底部操作区，减少阴影和厚重边框。
   - 事件文本优先，地点志和补给降低视觉权重。
8. `.map-layout`
   - 改为单列。
   - `.kyushu-map` 高度 260-340px。
   - `.map-node` 在手机端减少文字，只保留地点名和状态；完整消耗放路线卡。
9. `.route-card`、`.choice-card`、`.supply-card`
   - 保持整卡可点。
   - `min-height` 不小于 56px。
   - 行距收紧，文字不溢出。
10. `.log-panel`
   - 从常驻区改为日志页内容；地点页不显示完整日志。
11. 新增 `.game-bottom-nav`
   - 固定或粘在手机壳底部。
   - 4 等分按钮，当前页高亮。
   - 高度建议 56-64px，并适配 `env(safe-area-inset-bottom)`。

### 7.3 JS 建议

现有 JS 已有 `activeView`、`setActiveView(view)`、`renderRoutes()`、`renderStatus()`、`renderOverview()`，不要重写状态机。建议小改：

1. 扩展 `setActiveView(view)`：
   - 支持 `"town"`、`"map"`、`"inventory"`、`"log"`。
   - 对应隐藏/显示四个 `view-panel`。
   - 更新底部导航 `aria-selected` 和 `.active`。
2. 增加 DOM 引用：
   - `inventoryViewButton`
   - `logViewButton`
   - `inventoryView`
   - `logView`
3. `renderStatus()` 继续负责行囊、语言、困厄、日志渲染，但渲染目标移动到新页面里。
4. `renderOverview()` 保持距离、进度、下一站文案；HUD 可继续复用它写入的 `#distanceLabel`。
5. `move(routeId)` 选择路线后建议明确 `setActiveView("town")`，让手机端立刻回到行旅/路遇反馈。
6. `renderEvent()` 中事件已处理提示里的“切到九州图”可以改成按钮或强提示，例如“打开九州图”，点击调用 `setActiveView("map")`。
7. `renderRoutes()` 不变玩法，只调整输出文案结构也可以。移动端路线卡需要保留现有四行信息。
8. 地图节点点击逻辑可以保留，但移动端不要只靠节点；路线卡仍是主要入口。

### 7.4 实施顺序

建议 C 线程按这个顺序做，便于每一步验收：

1. 手机壳：限制 `.app-shell` 宽度，桌面居中，取消 1440 桌面工作台。
2. HUD：压缩顶部和资源区，让三资源、日数、地点、距离常驻。
3. 单列地点页：隐藏/移出行囊和日志常驻面板，事件区成为主操作。
4. 单列九州图：小地图在上，路线卡在下。
5. 底部四导航：地点、九州图、行囊、日志。
6. 行囊页：迁移九宫格、语言、困厄。
7. 日志页：迁移完整日志和重开入口，设置类内容折叠。
8. 最后做 390x844、430x932 逐条验收。
