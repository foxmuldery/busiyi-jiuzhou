# 《不思异：九州》多线程协作制度

> 主线程：当前 Codex 线程  
> 子线程数量：5  
> 协作原则：所有子线程先提交开工申请，经主线程认证后再进入具体执行；所有交付回到主线程验收后才算完成。

## 1. 总体分工

主线程负责总制作人、产品负责人和最终验收。

主线程职责：

- 维护产品方向和范围。
- 拆解任务并写入任务交接台账。
- 审核子线程开工申请。
- 判断任务是否允许执行。
- 合并和验收子线程产物。
- 更新主产品文档。
- 控制商业风险、版权风险和范围膨胀。

子线程不直接决定产品方向。子线程只能在明确任务范围内工作，不能自行扩大范围、改主设定、改命名、改开发路线。

## 3. Sub-agent 临时专家层

除 5 个长期子线程外，主线程可以按问题临时建立 sub-agent。

sub-agent 的定位：

- 短周期阅读、调研、质询和补充判断。
- 围绕一个明确问题输出结论。
- 不直接修改主产品方向。
- 不直接向用户提交决策问题。
- 不替代长期子线程的正式交付。

适合交给 sub-agent 的任务：

- 阅读并复核长期子线程的交付文档。
- 对核心玩法提出反对意见和改进方案。
- 调研同类产品，并提炼可借鉴点和避坑点。
- 对 Web Demo 的具体功能做难度评估。
- 对一个设计方向做小型对照研究。

不适合交给 sub-agent 的任务：

- 直接决定产品定位。
- 直接改主文档最终结论。
- 直接生成大量正式剧情文本。
- 在没有主线程批准时写代码、安装依赖或启动服务。
- 做法律定论。

sub-agent 输出格式：

```markdown
# Sub-agent 结论

## 1. 阅读范围

## 2. 核心判断

## 3. 支持理由

## 4. 风险与反对意见

## 5. 建议进入主文档的内容

## 6. 需要用户决策的问题
```

主线程负责：

- 给 sub-agent 分配明确问题。
- 阅读 sub-agent 回复。
- 与长期子线程结论交叉验证。
- 去重、裁剪、合并。
- 把值得保留的内容写入主文档或台账。
- 把真正需要用户判断的问题统一提交。

## 3. 交接制度

所有任务必须通过 `不思异九州_任务交接台账.md` 流转。

### 3.1 状态流转

任务状态只能使用以下几种：

```text
待定义 -> 待开工申请 -> 待主线程审批 -> 已批准开工 -> 执行中 -> 待验收 -> 已验收 -> 需返工 -> 暂停
```

### 3.2 标准流程

1. 主线程提出任务，并写入交接台账。
2. 子线程只读取任务，不直接开工。
3. 子线程提交开工申请。
4. 主线程审核开工申请。
5. 主线程批准后，子线程才执行。
6. 子线程提交交付物和自检结果。
7. 主线程验收。
8. 通过后，主线程更新主产品文档或任务状态。

### 3.3 子线程开工申请模板

```markdown
## 开工申请

- 任务 ID：
- 子线程：
- 我理解的目标：
- 我准备做的工作：
- 不会做的工作：
- 预计产物：
- 需要主线程确认的问题：
- 风险点：
- 预计完成标准：
```

### 3.4 子线程交付模板

```markdown
## 子线程交付

- 任务 ID：
- 子线程：
- 已完成内容：

```

```markdown
- 下一步任务：
```
