# BSI-A-013 后段压力与保底复核建议

生成日期：2026-06-20

本轮范围：只读复核 `GitHub资产区/03_WebDemo/prototype/data.js` 与 `balance-sim.js`；未改 `app.js`、`data.js`、`balance-sim.js`。

## 当前判断

结论：下一轮不建议直接增加硬失败风险，建议先保留当前“可稳定通关，但新手会明显告警”的试玩平衡。

理由：

1. 当前模拟结果符合试玩版第一章目标：`balanced` 与 `novice` 都是 100% 到达裂隙结局，没有 hard failure。新手线虽然没有硬失败，但 warning 很高，已经能制造资源紧张感。
2. `balance-sim.js` 的失败保护不是无底线免死：资源降到 0 后会触发 crisis；同类 crisis 2 次、总 rescue 达 3 次、或 badLuck 到 96 都会 stranded。只是当前路线池和补给事件让资源大多在硬失败前被拉回。
3. 后段路线成本已经有实际压力：`red_to_rift` 为 `-12/-9/-14`，`stele_to_rift` 为 `-10/-9/-16`，`dream_to_rift` 为 `-8/-9/-15`。高压集中在最后 1-2 段，不是全程偏软。
4. 当前路线池明显偏“保玩家体验”：`red_marsh`、`broken_stele`、`dream_map_post` 都给了 `sanity_rescue` / `rift_buffer` 方向；`red_to_rift` 虽是直达高压路，但带锁且权重低。这个设计适合试玩入口，避免用户第一次玩就被系统判死。
5. 新手 warningRate 已经偏高。按当前 `balance-sim.js` 的报告判定逻辑，novice warning 超过 78% 会标成“新手压力偏高”。继续加硬失败之前，应先确认用户想要“试玩安全”还是“更像 roguelite 的失败威胁”。

本轮复测补充：用 `bsi-a-013` seed、1000 runs 复核，`balanced` 成功率 100%、warning 9.2%、rescue 0%；`novice` 成功率 100%、warning 81.1%、rescue 0.3%、hard failure 0。与主线程提供数据同向，差异可视为 seed/runs 波动。

## 下一轮可验证调参方案

建议采用“三档小步模拟”，先不直接合入主线数据。

### 方案 A：保留当前主线，只调报告阈值

目标：确认当前是否已经可以进入真人试玩。

验证方式：

- 保持 `data.js` 不动。
- 跑 `balanced`、`novice`、`risky` 各 1000-3000 次。
- 接受标准：`balanced` 成功率 >= 98%，`novice` 成功率 >= 95%，hard failure = 0，novice warning 60%-85%。

建议判断：如果用户目标是“5分钟可玩 Demo”，优先选 A。当前已经接近可交付，只需要真人试玩验证感受。

### 方案 B：只增加后段选择惩罚，不增加硬失败

目标：让用户在后段看到更明确的代价，但仍不轻易死局。

可验证小改方向：

- 将 `red_bones` 的“沿骨行路”从 `车轴 -4，神志 -10` 试调到 `车轴 -5，粮草 -2，神志 -11`。
- 将 `read_name` 的“拓碑带走”从 `神志 -7` 试调到 `粮草 -2，神志 -8`。
- 将 `dream_map_gate` 的“按梦门标记前哨”从 `神志 -6` 试调到 `粮草 -2，神志 -7`。

接受标准：`balanced` 仍 100% 或接近 100%；`novice` rescueRate 可升到 2%-5%，hard failure 仍为 0；warning 不超过 88%。

建议判断：如果真人试玩反馈“后段太顺、没感觉在付代价”，选 B。

### 方案 C：增加极少量失败风险，只给冒险/低神志直冲路线

目标：让连续误判玩家有真实失败可能，但不伤默认玩家。

可验证小改方向：

- 仅调整 `red_to_rift`、`stele_to_rift`、`dream_to_rift` 三条路线之一，不同时全改。
- 优先试 `red_to_rift`：保持锁路条件不变，将成本从 `-12/-9/-14` 试调到 `-13/-10/-16`。
- 或者只让 `red_bones` 的低神志选项“凝视赤水辨路 / 夜里下水”更危险，不动普通选项。

接受标准：`balanced` 成功率 >= 98%，`novice` 成功率 >= 92%，hard failure <= 2%，失败集中在 late_7_plus，不出现 early/mid 突然死。

建议判断：只有用户明确想要“第一章试玩也要有人失败”时再选 C。

## 3 条路线事件补救 / 惩罚建议

### 1. 赤水岸骨：直冲裂隙应更有代价，但保留退路

对应位置：`red_bones` 事件与 `red_marsh` 路线池。

建议：

- 保留“掩骨祭拜”为安全补救：粮草换神志，帮助新手不崩线。
- “沿骨行路”可以略加粮草损耗，让它不只是神志换捷径。
- “凝视赤水辨路”“夜里下水”作为低神志/禁忌试演，可承担更高惩罚，但不建议作为普通玩家必经项。

目的：让赤水成为“可冒险开捷径”的节点，而不是隐性正确答案。

### 2. 断碑读名：把“读名”做成神志风险核心，不做车轴/粮草全面惩罚

对应位置：`read_name` 事件与 `stele_to_rift` / `stele_to_dream`。

建议：

- “拓碑带走”维持中等神志消耗，可附带少量粮草成本，表示停留拓印耗时。
- “毁去碑面”继续作为神志补救，但保留车轴代价。
- “让碑补全你的名字”应保持低神志条件，作为解锁裂隙名的高风险选择，不建议开放给正常神志玩家。

目的：断碑节点负责制造“知道得越多，越接近裂隙，也越伤神志”的叙事压力。

### 3. 梦图前哨：作为最后保底缓冲，不宜改成杀点

对应位置：`dream_map_post`、`dream_map_gate`、`dream_to_rift`。

建议：

- 保留 `dream_map_post` 的 `sanity_rescue` / `rift_buffer` 定位。
- “整束梦图”继续作为最后神志补救，代价是粮草。
- “按梦门标记前哨”可以小幅增加粮草或神志成本，但不建议引入 hard failure。

目的：让最后一段仍有紧张，但玩家抵达梦图前哨时应感觉“还能整理队伍、做最后选择”，而不是突然被系统收走。

## 需要用户拍板的问题

1. 第一章 WebDemo 的目标是“几乎人人能抵达裂隙结局”，还是允许新手约 3%-8% 在后段失败？
2. 当前 novice warningRate 约 80% 是否符合预期？如果觉得提示太吵，应该降 warning，而不是继续加惩罚。
3. 后段压力更想落在哪种资源上：神志为主，粮草为辅，还是三资源一起压？我建议神志为主，粮草为辅，车轴只在水路/骨路偶发承压。
4. `critical rescue` 是否要作为显性体验展示？如果保留 100% 通关，建议把 rescue 做成“惊险补救弹窗”，让玩家感到差点出事，而不是模拟表里隐藏发生。

## 建议下一步

短期：保留当前主线，不马上增加硬失败。先把这一版交给真人试玩，观察玩家是否觉得“紧张但能懂”，还是“只是数值闪红但没有后果”。

若试玩反馈偏安全：优先试方案 B，小幅增加后段事件惩罚，目标是 novice rescueRate 2%-5%、hard failure 0。

若用户明确想要失败威胁：再试方案 C，把失败风险限制在后段直冲/低神志禁忌路线，不动早中期教程路线。
