# D_BSI-D-007 主页面参考融合体验方案

> 子线程：D 视觉体验与交互氛围设计  
> 对应主协调任务：BSI-UX-008  
> 项目：《不思异：九州》Web Demo  
> 日期：2026-06-14  
> 边界：本文件只写体验方案与 C 线程收口清单，不修改 `index.html`、`styles.css`、`app.js`

## 0. 本轮结论

用户最新意见是：主页面仍然要参考“西部横向旅行游戏”和《Out There》。

这两个参考不能混成一种皮相，而要明确分工：

1. 西部横向旅行主画面负责第一眼：车队正在路上、远景在动、云和昼夜变化、半途停下、路标提示下一站。
2. Out There 路线决策负责第二层判断：下一站选择、三资源消耗、风险比较、可观察补给、锁定路线、探索未知。
3. 《不思异：九州》的最终表达必须回到山海九州：辇车、古道、荒泽、废关、异象、神志污染、舆图、路引、地志。

主页面默认应是“横向旅途舞台”，不是信息首页；九州图是“战略路线模式”，不是网页地图后台。

## 1. 两套参考的明确分工

| 参考来源 | 负责什么 | 落到主页面怎么做 | 不负责什么 |
| --- | --- | --- | --- |
| 西部横向旅行主画面 | 路上体感、横向构图、车队行进、远近景层次、昼夜云气、停靠与半途路遇 | `.journey-stage` 成为第一视觉；车队在前景道路上；背景、云、远山、前景车辙有轻微视差；路线选择后回到舞台表现“从左到右走了一段” | 不复制西部题材、马车造型、UI、字体、配色、音效；不变成横版操作关卡 |
| Out There 路线决策 | 出发前的战略比较、资源压力、路线风险、未知节点、锁定条件、目的地收益预期 | `九州图` 模式保留路线节点和路线卡；每条 `.route-card` 必须显示去哪里、消耗什么、风险多高、可能遇到什么、目的地有什么补给 | 不复制星图、太空图标、资源图形、文案结构、商店表达；不宣传成“山海经版 Out There” |

一句话分工：

```text
旅途页像“车队正在路上”。
九州图像“我在舆图上决定下一段路”。
```

## 2. 主页面第一屏信息架构

第一屏玩家不用打开抽屉，也应该能判断：

1. 我在哪里。
2. 我还有几段路到九州裂隙。
3. 车轴、粮草、神志是否危险。
4. 车队此刻是停靠、行进，还是半途路遇。
5. 当前事件要我做什么。
6. 下一步是处理事件、补给，还是打开九州图选路。

### 2.1 必须可见

| 区域 | 必须可见内容 | 现有结构承接 |
| --- | --- | --- |
| 顶部轻 HUD | 当前地点、第几日、三资源数值与危险状态、距九州裂隙、旅途/九州图入口 | `.game-hud`、`.topbar`、`.resource-strip`、`.journey-overview` |
| 旅途舞台 | 天空、远景、路面、车队、路线方向、昼夜/云气/污染状态、半途停下状态 | `.journey-stage`、`.caravan`、`.stage-nodes`、`.sky-gradient`、`.cloud`、`.asset-bg` |
| 底部行动区 | 当前事件标题、正文摘要、主要选择按钮、补给入口、最近一条反馈 | `.event-panel`、`.event-text`、`.choice-list`、`.supply-section`、`.latest-log-peek` |
| 九州图入口 | 明确告诉玩家“打开舆图选下一段路” | `.view-tab[data-view-target="map"]` 或同等入口 |

### 2.2 必须折叠或降权

| 内容 | 处理方式 | 原因 |
| --- | --- | --- |
| 完整行囊九宫格 | 放进 `.side-drawer`，只保留“行囊”入口 | 行囊是复查，不是每秒决策 |
| 完整日志 | 放进 `.side-drawer`，主界面只露一条 `.latest-log-peek` | 日志用于复盘，不应抢主舞台 |
| 设置、音乐、动态层说明 | 放进 `.settings-panel`，主界面不出现开发提示 | “音乐槽位载入中”“视频层未加载素材”会破坏沉浸 |
| 地点志长文 | 第一屏只保留 1-2 行摘要，更多内容以后再展开 | 长文会让底部像网页文章卡 |
| 资源提示小字 | 桌面可保留短词；横屏小尺寸优先隐藏 | 三资源核心是数值和危险，不是解释段落 |
| 地图节点详细说明 | 节点只留地名/状态，详细消耗和风险放路线卡 | 地图承担探索感，路线卡承担决策 |

## 3. 当前原型仍像网页面板的地方与减重方向

当前原型已经有横屏框、舞台、车队、云、昼夜、地图和抽屉基础。仍然偏网页面板的地方主要是信息块太“成组”、边框太多、标题太像后台卡片。

### 3.1 `.game-hud` 仍像三组控制面板

问题：

- `.topbar`、`.resource-strip`、`.journey-overview` 现在像三个并排小面板。
- `.resource-card` 仍是三张资源卡。
- `.view-tabs` 像网页页签。

减重：

1. 把顶部理解为一条 HUD，不是三块卡片。
2. 资源改成短胶囊或小仪表：名称 + 数值 + 细条，危险时才加红色强调。
3. “旅途 / 九州图”可以保留为模式切换，但视觉上应像游戏 HUD 按钮，不像网页 tab。
4. `重开`、`区域`、资源解释小字在小尺寸下继续降权。

### 3.2 `.event-panel` 仍像信息汇总卡

问题：

- `.panel-title`、`.location-lore`、`.supply-section`、`.event-text`、`.choice-list` 同时出现，像把所有后台信息都放进一张大卡。
- `.location-lore` 有独立边框和标题，会抢当前事件。
- 补给区和选择区都像卡片列表，底部行动栏的“游戏感”还不够。

减重：

1. `.event-panel` 定义为底部行动栏，不再强调“面板”。
2. 当前事件标题、正文、选择按钮是主内容。
3. 地点志只留短摘要，默认不做独立卡。
4. 补给改成“整备小组”，只显示可点项和已取状态。
5. 选择按钮可保留触控面积，但边框、阴影、标题层级要轻。

### 3.3 `.map-panel` 仍像网页地图后台

问题：

- `.map-panel` + `.map-layout` 是典型“左地图右列表”的网页结构。
- `.kyushu-map` 的网格感偏科技/后台，山海舆图气质还不足。
- `.map-node` 是一个个信息卡，文字多时会像看流程图。
- `.route-card` 纵向堆叠后像路线列表，而不是战略抉择。

减重：

1. `.map-panel` 应继承游戏框，不再像单独网页页。
2. `.kyushu-map` 改成山海舆图：纸纹、山水纹、墨线、路引、关隘，不强调网格。
3. `.map-node` 更像舆图标记或地名签，节点详情挪给 `.route-card`。
4. `.route-card` 是“决策签”，不是普通列表卡；每张只保留决定前必须看的信息。

### 3.4 `.side-drawer` 基础正确，但内部还可减重

问题：

- 抽屉已经把行囊、日志、设置收起来，这是对的。
- 但抽屉内部继续嵌 `.panel`，会形成“卡片套卡片”的感觉。

减重：

1. 抽屉打开后只作为辅助复查层，不改变当前旅途/地图状态。
2. 抽屉内部减少二级边框，行囊格、日志列表、设置项直接排列。
3. 设置里的技术说明不外露到主界面。

## 4. 旅途舞台如何更像“车队在路上”

`.journey-stage` 是本轮最重要的视觉对象。它不应只是一个背景容器，而要承担“玩家正在路上”的叙事。

### 4.1 舞台

1. 舞台占主画面最大面积，默认比底部行动区更强。
2. 舞台不是纯背景图，上下至少有四层：天空/云气、远山或远地貌、中景道路、前景车辙。
3. 边框要弱，画面本身要强；不要让玩家先看到一个被框起来的图片。
4. 低神志、夜晚、荒泽、废关等状态优先改变舞台气氛，而不是只改文字。

### 4.2 车队

1. 车队必须常驻可见，停靠时在左前景或中左位置。
2. 选择路线后，车队用现有 `.traveling` 状态表现向右推进。
3. 半途路遇时，车队用现有 `.interrupted` 状态停在画面中段，让玩家知道“还没到下一站”。
4. 车队脚下要接住道路和车辙，不要像浮在背景图上。
5. 若使用辇车素材，优先保留剪影和轮廓，不要做成西部马车。

### 4.3 远景、云、日夜

1. 远景应缓慢横移，表现路在走；幅度轻，不要变成跑酷。
2. 云和风沙可以继续慢速移动，作为“路上空气”。
3. 日夜不是装饰，应和日数、地点、神志污染一起让旅途有时间感。
4. 夜晚和低神志状态要克制：看得出异样，但不遮挡车队和路标。

### 4.4 路标

现有 `.stage-nodes` 可以从“文本进度条”改成“路上路标”：

```text
起点 / 当前路况 / 下一站
中原驿 / 故王道有裂纹 / 废关
```

建议：

1. 起点和下一站贴近道路两端。
2. 中间显示路线名或路况，不必总是长标题。
3. 半途路遇时中间改成“半途”或路遇名，下一站保持可见。
4. 小尺寸只显示地名，完整路线信息交给底部或九州图。

### 4.5 半途停下

半途停下是“西部横向旅行主画面”和 Out There 决策之间的连接点：

1. 车队停在路中，不切成新地点页。
2. 舞台可以压暗或加强风沙/异象，表示事件打断旅程。
3. 底部 `.event-panel` 显示“半途：起点 -> 目的地”和路遇事件。
4. 补给不可用或降权，因为还没抵达地点。
5. 处理后再抵达目的地，不能让玩家误以为已经到站。

## 5. 九州图模式：保留 Out There 决策，换成山海舆图表达

九州图模式要保留 Out There 式的战略压力，但视觉语言必须是《不思异：九州》的山海舆图。

### 5.1 必须保留的决策结构

每次出发前，玩家必须能比较：

1. 目的地：下一站是哪一个地点。
2. 消耗：车轴、粮草、神志分别会变多少。
3. 风险：低/中/高，或更具体的路况风险。
4. 半途：是否可能触发路遇。
5. 补给预期：目的地可观察到什么资源倾向。
6. 锁定原因：缺语言、缺器物、缺线索，还是尚未显明。
7. 探索价值：是否通向未知区域或更接近九州裂隙。

这些信息应主要放在 `.route-card`，地图负责让玩家看见“世界和前路”。

### 5.2 山海舆图表达

建议把视觉语言换成：

| Out There 抽象 | 九州表达 |
| --- | --- |
| 星图节点 | 驿、关、泽、墟、山口、祭所、裂隙前沿 |
| 航线 | 古道、水路、山径、旧王道、异族秘径 |
| 燃料/氧气/船体压力 | 车轴、粮草、神志 |
| 未知星系 | 雾中地名、残卷未显、山海缺页 |
| 锁定路线 | 需要语言、器物、地志线索或前置地点 |
| 深空危险 | 异象、断轴、饥荒、失神、裂隙污染 |

### 5.3 地图和路线卡的职责

`.kyushu-map`：

1. 显示当前点、可达点、已走路线、锁定路线、未知远方。
2. 节点少字，突出状态。
3. 使用山水、纸纹、墨线、边境污染表达，不用太空星图或科技网格。

`.route-card`：

1. 显示目的地、路线名和一句路况。
2. 用固定顺序显示三资源消耗。
3. 显示风险和半途路遇。
4. 显示目的地补给预期。
5. 锁定时显示原因，不让玩家误以为可点。

### 5.4 交互闭环

推荐流程：

```text
旅途舞台停靠
-> 底部提示“打开九州图选择下一段路”
-> 九州图中点选路线或路线卡
-> 地图高亮该路线，路线卡显示消耗/风险/补给
-> 确认前进
-> 回到旅途舞台，车队向右推进
-> 半途路遇或抵达下一站
```

这样保留 Out There 的“出发前压力”，但玩家看到的是山海路途，而不是星际航行。

| `.side-drawer` | 保留为行囊/日志/设置入口，不改变主流程状态 | 打开关闭后仍回到原旅途或九州图状态 |
| `.drawer-panel` | 内部减少卡片套卡片，宽度继续控制在 42% 以内 | 抽屉是辅助层，不抢主舞台 |
| `.status-panel` | 行囊九宫格只在抽屉里展示 | 行囊不常驻挤压舞台 |
| `.log-panel` | 完整日志只在抽屉里滚动 | 主页面不变成日志页 |
| `.settings-panel` | 音乐、动态层、视频说明收纳在设置里 | 主页面不出现开发说明 |

## 7. 第一屏验收清单

C 线程收口后，建议按以下问题验收：

1. 打开后 2 秒内，玩家是否先看到“车队正在神秘九州路上”？
2. 不打开抽屉，是否能看见当前位置、日数、三资源、距九州裂隙？
3. 舞台是否比底部行动区更强？
4. 当前事件和主要选择是否清楚？
5. 补给是否可见但不抢事件？
6. “九州图”是否明显是选下一段路的入口？
7. 半途路遇时，画面是否清楚表达“还没到下一站”？
8. 九州图中，每条可走路线是否能比较消耗、风险、补给和锁定条件？
9. 行囊、日志、设置是否已经退到辅助抽屉？
10. 页面是否避免了“网页 dashboard / 卡片墙 / 后台地图”的第一印象？

## 8. 竞品表达边界

本项目可以借鉴竞品的抽象体验结构，但不能复制表达。

禁止复制：

1. 不复制 Out There 的星图 UI、太空题材、图标、资源视觉、商店表达、文本结构、路线命名。
2. 不复制西部横向旅行游戏的具体 UI、美术、马车造型、镜头构图、配色、字体、音效、事件表达。
3. 不使用竞品宣传话术，不写“山海经版 Out There”或类似绑定式文案。
4. 不把《不思异：九州》做成西部片气质。
5. 不引入竞品的实时驾驶、碰撞、跳跃、横版关卡玩法。

允许借鉴：

1. “旅途画面先建立路上体感”的结构。

4. “未知远方逐步显明”的探索压力。

## 9. 需要用户决策的问题

本轮没有必须阻塞 C 线程收口的用户决策。建议先按以下默认执行：

1. 主页面默认偏“西部横向旅行主画面”的路上体感。
2. 九州图默认偏 Out There 式路线/资源/风险决策，但视觉改成山海舆图。
3. 行囊、日志、设置继续用 `.side-drawer`。
4. 桌面按 16:9 游戏框收口，手机按横屏小尺寸收口。

后续可以集中让用户判断的非阻塞问题：

1. 横屏比例最终更偏 16:9，还是更贴近 19.5:9 手机横屏。
2. 九州图是完全替换舞台，还是带一点旅途舞台底色的覆盖模式。
3. 路线卡视觉更像“木牌/路引”，还是“残卷/舆图签”。
4. 车队行进反馈要偏写实慢动，还是更明显的假动态推进。
