# D_BSI-D-006 横屏手游界面方案

> 子线程：D 视觉体验与交互氛围设计  
> 对应主协调任务：BSI-UX-007  
> 项目：《不思异：九州》Web Demo  
> 日期：2026-06-14  
> 边界：本文件只做体验方案与验收标准，不修改 `index.html`、`styles.css`、`app.js`

## 0. 修正结论

D_BSI-D-005 的“竖屏手机壳”方向作废。新的核心判断是：本项目应是横屏手游界面，第一视觉必须是横向山海旅途舞台。玩家打开后应先看到车队、远景、路面、日夜、异象和横向行进感，而不是竖屏卡牌列表或网页工作台。

第一版推荐结构：

```text
横屏游戏画幅
┌──────────────────────────────────────────────┐
│ 顶部轻 HUD：地点 / 日数 / 车轴 / 粮草 / 神志 / 距终点 │
├──────────────────────────────────────────────┤
│                                              │
│            横向山海旅途舞台 / 九州图模式             │
│                                              │
├────────────────────────────────┬─────────────┤
│ 底部主行动区：事件 / 补给 / 路线卡 / 选择按钮       │ 右侧轻入口  │
│                                │ 行囊 日志 设置 │
└────────────────────────────────┴─────────────┘
```

桌面浏览器不再预览竖屏手机，而是预览一个横屏游戏画幅。移动端默认按横屏 844x390 / 932x430 验收。

## 1. 横屏手游信息架构

### 1.1 主层级

横屏界面只保留三层主信息：

1. 顶部 HUD：轻量常驻状态。
2. 中央舞台：游戏主视觉，地点页显示旅途舞台，九州图模式显示战略地图。
3. 底部行动区：当前可做什么，包括事件选择、补给、路线选择。

辅助信息不常驻挤压舞台：

- 行囊：右侧抽屉或弹层。
- 日志：最近一条反馈常驻，完整日志折叠。
- 设置：小入口，放音乐、动态层、重开等低频操作。

### 1.2 横屏优先的阅读顺序

玩家每次看屏幕，应该按这个顺序理解局面：

```text
我在哪里 / 还剩多少路
-> 车队现在是停靠、行进，还是半途停下
-> 当前事件或下一段路线是什么
-> 我点哪个按钮继续
```

因此：

1. `.journey-stage` 是第一视觉，应占画幅高度的约 52%-62%。
2. 事件、补给、路线卡贴在舞台下方，像游戏行动栏，不像网页卡片墙。
3. 行囊、日志、音乐、动态层不应和主事件并列常驻。
4. 九州图可以切换模式，但仍在横屏游戏框内，不做独立网页仪表盘。

### 1.3 基准画幅

建议 C 线程以这些尺寸为基准：

- 桌面 16:9：1366x768、1280x720。
- 手机横屏：844x390、932x430。
- 桌面超宽：只居中显示横屏游戏画幅，不铺满成后台。

桌面预览建议：

- `.app-shell` 或等价容器宽度可用 `min(100vw - 32px, 1366px)`。
- 内部保持 `aspect-ratio: 16 / 9` 或接近手机横屏比例。
- 高度不足时允许内部内容纵向压缩或行动区滚动，但不能让网页本身变成多面板工作台。

## 2. 地点页布局

地点页负责“车队停靠、正在行进、半途路遇”三种状态。它不是信息主页，而是一幕横向旅途。

### 2.1 地点页默认布局

```text
顶部 HUD
  当前地点 / 区域 / 第 X 日 / 车轴 / 粮草 / 神志 / 距九州裂隙

中央横向舞台
  背景长图 / 前景 / 车队 / 路面 / 日夜 / 神志污染
  from -> route -> to 节点标签轻量叠加

底部行动区
  左：地点志摘要 + 当前事件文本
  中：事件选择按钮
  右：补给入口 + 行囊/日志/设置小入口
```

落地到现有结构：

- `.journey-stage` 放大，成为地点页主视觉。
- `.event-panel` 从普通网页面板改成底部行动栏。
- `.supply-section` 保留在地点页，但视觉上降为“整备/补给”小组。
- `.status-panel` 不再常驻占用右栏，改成行囊抽屉/弹层来源。
- `.log-panel` 不再作为地点页完整列表，只保留最近一条反馈入口。

### 2.2 停靠状态

停靠状态下：

1. HUD 显示当前地点、日数、三资源、终点距离。
2. 舞台显示当前地形和车队停靠位置。
3. 底部行动区显示地点志摘要、补给、地点事件。
4. 补给按钮可点，已搜集后保留“已取/已搜集”状态。
5. 地点事件处理后，行动区提示“打开九州图选择下一段路”。

### 2.3 行进状态

选择路线后：

1. 现有 `move(routeId)` 已会 `setActiveView("town")`，应继续保留。
2. 舞台播放或表现车队从左向右移动。
3. HUD 立即更新日数和资源。
4. 若没有半途路遇，抵达后刷新为目的地停靠状态。
5. 若有半途路遇，进入半途状态。

### 2.4 半途路遇状态

半途状态必须让玩家知道“还没抵达目的地”：

1. 舞台车队停在中段，使用现有 `.interrupted` 逻辑。
2. HUD 或行动区写明“半途：起点 -> 目的地”。
3. 底部行动区显示路遇标题、路线名、事件文本、选择按钮。
4. 补给按钮不可用，只保留解释：“车队停在半途，无法搜集城镇补给。”
5. 路遇处理后，自动或明确抵达目的地。

## 3. 九州图模式布局

九州图不是网页页签里的地图页，而是横屏游戏框内的“战略地图模式”。

### 3.1 九州图模式结构

```text
顶部 HUD
  保持地点、日数、三资源、终点距离

中央地图舞台
  左/中：九州路线节点图
  当前点最强，可达路线次强，锁定/未来降权

底部/右侧路线决策区
  可选路线卡：目的地 / 消耗 / 风险 / 半途路遇 / 目的地可观察补给
```

落地到现有结构：

- `#mapView` 可以继续作为模式容器，但视觉上应和地点页共享同一横屏游戏框。
- `.kyushu-map` 占主视觉，不要像网页插图。
- `.map-route-panel` 变成底部路线卡栏或右下行动栏。
- `.map-layout` 可以保留为横屏内部布局，但不能变成大网页两栏；推荐“地图 65%-72%，路线区 28%-35%”。

### 3.2 路线选择要求

每条路线卡必须在点击前回答：

1. 去哪里。
2. 消耗多少车轴、粮草、神志。
3. 风险等级。
4. 半途是否有路遇。
5. 目的地可观察补给偏向。
6. 若锁定，为什么锁定。

点击路线后：

1. 回到地点页/旅途舞台。
2. 车队进入行进反馈。
3. 资源、日数、日志同步变化。

### 3.3 地图触控

横屏手机上节点可以点，但路线卡仍是主要确认入口：

- 点节点可以选中或直接前进，取决于 C 线程现有逻辑。
- 如果保留直接前进，节点必须只对可达路线可点。
- 锁定节点不能像可点按钮。
- 小尺寸横屏下，节点文字可以简化，完整信息放路线卡。

## 4. 行囊、日志、设置

### 4.1 行囊弹层

行囊不是常驻页，也不是右侧固定后台栏。推荐两种实现：

1. 右侧抽屉：从画面右侧滑出，宽度约 280-340px；844x390 下宽度不超过画幅 42%。
2. 居中弹层：覆盖舞台和行动区中间，背景轻暗，保留关闭按钮。

内容：

- 行囊九宫格。
- 语言：羽民语 / 巫咸古辞。
- 困厄：厄运、求生、败局、断轴、饥荒、失神。
- 与路线条件有关的线索可突出，但不要在弹层里处理路线选择。

验收重点：

- 打开行囊时不改变当前地点/路线状态。
- 关闭后回到原模式。
- 844x390 下九宫格文字不互相遮挡。

### 4.2 日志折叠

日志分两层：

1. 常驻最近反馈：行动区或右侧入口显示最近一条，例如“补给：粮草 +8”。
2. 完整日志：点击“日志”后展开抽屉或弹层，显示 `state.log`。

要求：

- 完整日志不常驻占据底部高度。
- 日志能解释资源变化、路线选择、路遇、补给、失败/结局。
- 小尺寸横屏下完整日志内部滚动，不能撑破游戏画幅。

### 4.3 设置入口

设置入口建议放右上角或右侧轻入口区，用一个短按钮或图标表示。

设置内包含：

- 播放/关闭音乐。
- 启用/关闭轻量动态。
- 视频层/素材状态说明。
- 重开按钮，或至少在失败/结局状态显示重开。

禁止把“音乐槽位载入中”“视频层未加载素材”等开发说明常驻在主界面。

## 5. 横屏中的模式切换

第一版建议保留两个主模式：

```text
地点/旅途模式
九州图模式
```

切换入口：

- 放在顶部 HUD 右侧或底部行动区右侧，命名为“九州图 / 返回旅途”。
- 不做 D-005 的竖屏四底栏。
- 行囊、日志、设置是弹层入口，不算主模式。

流程闭环：

```text
地点页停靠
-> 补给/地点事件
-> 打开九州图
-> 比较路线
-> 选择路线
-> 回到横向旅途舞台
-> 半途路遇或抵达
-> 处理路遇/地点事件
```

## 6. D_BSI-D-005 作废与保留

### 6.1 明确作废

以下 D-005 建议不再执行：

1. “竖屏手机壳”为主体验。
2. 桌面浏览器预览 390-430px 竖屏手机。
3. 390x844 / 430x932 作为主验收尺寸。
4. 底部固定四导航：地点 / 九州图 / 行囊 / 日志。
5. 行囊页、日志页作为与地点页/九州图平级的四主页面。
6. `.app-shell { width: min(430px, 100vw) }` 这类竖屏窄壳建议。
7. 地点舞台高度按竖屏 220-300px 规划。
8. 九州图移动端改成“小地图上方 + 路线列表下方”的竖屏单列结构。

### 6.2 可以保留

以下 D-005 判断仍然有效，但要横屏化：

1. 当前网页感来自信息同时摊开、卡片太多、行囊/日志/设置常驻。
2. 顶部 HUD 要轻，三资源、日数、当前位置、终点距离常驻。
3. 行囊九宫格不要常驻挤压主舞台。
4. 完整日志不要常驻挤压主界面。
5. 音乐、动态层、视频素材说明要进设置或折叠。
6. 地点页负责当前事件、补给、半途路遇；九州图负责路线选择。
7. 路线卡必须显示目的地、消耗、风险、半途路遇、目的地可观察补给。
8. 不依赖 hover，按钮和卡片要适合触控。
9. 不引入新框架，优先基于现有 HTML/CSS/JS 做最小改造。
10. 选择路线后回到旅途舞台，这点与现有 `move()` 行为一致，应保留。

## 7. 给 C 线程的落地建议

### 7.1 最小结构方向

基于当前文件：

- `index.html`
  - 保留 `.app-shell` 作为横屏游戏框。
  - 保留 `.topbar` + `.resource-strip`，视觉合并成 HUD。
  - 保留 `#townView` / `#mapView`，但把它们定义为“旅途模式 / 九州图模式”。
  - `.status-panel` 改成行囊/设置抽屉内容来源。
  - `.log-panel` 改成日志抽屉内容来源。

- `styles.css`
  - `.app-shell` 从宽网页改成横屏画幅容器。
  - `.journey-stage` 放大，占主要高度。
  - `.control-grid.town-grid` 改成横屏底部行动区。
  - `.map-layout` 改成游戏内地图模式，不要网页后台感。
  - `.panel` 阴影和边框减重，避免卡片墙。

- `app.js`
  - 现有 `setActiveView("town" / "map")` 可继续作为主模式切换。
  - 现有 `renderStage()`、`renderRoutes()`、`renderMap()`、`renderSupplies()`、`renderEvent()` 可保留。
  - 如需弹层，只新增轻量打开/关闭状态，不改玩法数据。

### 7.2 推荐布局比例

1366x768 / 1280x720：

- HUD：48-64px。
- 舞台/地图：约 430-500px。
- 底部行动区：160-210px。
- 右侧轻入口区：80-180px，可只放图标/短按钮。

844x390：

- HUD：38-46px。
- 舞台/地图：190-225px。
- 底部行动区：110-140px。
- 行囊/日志/设置入口必须很轻，必要时只保留短按钮。

932x430：

- HUD：42-52px。
- 舞台/地图：220-255px。
- 底部行动区：120-150px。
- 可以允许路线卡横向滚动，但不能横向撑破页面。

### 7.3 视觉减重

1. 减少大面积面板阴影，面板应像游戏 HUD 贴片。
2. `.project-label` 中的 `Web Demo Prototype` 不应出现在主视觉。
3. 面板标题减少，保留当前事件、补给、路线这类玩家任务名。
4. 资源不做三张大卡，改为 HUD 短条或小胶囊。
5. 地点志只显示摘要，长文本以后可做“详情”折叠。
6. 地图节点减少文字，决策信息放路线卡。

## 9. 需要用户决策的问题

本轮没有阻塞 C 线程做技术预案的用户决策。建议先按以下默认执行：

1. 横屏优先，桌面为 16:9 横屏游戏预览。
2. 手机按 844x390 / 932x430 横屏验收。
3. 九州图做同一游戏框内的战略地图模式。
4. 行囊用右侧抽屉或居中弹层，先由 C 线程评估哪种改动更小。
5. 日志默认最近一条，完整日志折叠。
