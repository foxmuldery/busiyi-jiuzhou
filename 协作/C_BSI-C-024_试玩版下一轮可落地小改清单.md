## 结论

当前 Web Demo 已具备第一章试玩纵切片：横向旅途舞台、三资源、补给、九州图选路、固定/随机半途路遇、资源飘字、告急浮层和基础音频钩子都已经跑通。

下一轮最值得做的不是继续堆机制，而是把 5 分钟试玩路径变顺：让玩家更快知道“先处理地点事件 / 再打开九州图 / 选路 / 半途处理 / 抵达 / 补给”，同时补上真实浏览器验证和本地存档韧性。

本清单按 P0/P1/P2 排序：

- P0：会影响本地试玩稳定性、卡死或验收可信度的小修。
- P1：明显改善“看不懂、点不顺、反馈不明显”的小修。
- P2：有价值但需要等待 A/B/D/F 输入，或容易扩范围，暂缓。

## 验证方法

### 固定入口

建议本轮统一使用：

```text
http://127.0.0.1:4177/03_WebDemo/prototype/
```

若端口被占用，可临时换端口，但交付时必须写明真实入口。

### 3-5 分钟核心路径

1. 打开页面，确认没有控制台 error。
2. 新开局，确认三资源、旅途舞台、当前遭遇、补给区可见。
3. 点击“播放音乐”，确认状态显示音乐启用；关闭后不再播放。
4. 处理起点事件，确认出现资源变化、飘字、舞台浮层和日志。
5. 打开九州图，选择一条路线，确认路线消耗、预感标签和半途路遇提示。
6. 处理半途路遇，确认处理后才抵达下一地点。
7. 抵达后使用一次补给，确认补给只能在本次抵达使用一次。
8. 用控制台触发 `window.BSI_PROTOTYPE.forceCrisis("sanity")`，确认濒死补救可读、可点、可返回继续。
9. 用控制台触发硬失败测试，确认“旅途断绝”页有重开入口。

### 三档横屏检查

- 844x390：模拟横屏手机最小档，重点看按钮是否挤压、路线卡是否可滚动。
- 932x430：主力手机横屏档，重点看资源告急、浮层和选择区。
- 1280x720：桌面验收档，重点看舞台、地图和日志抽屉。

### 自动/半自动检查

- `node --check prototype/app.js`
- `node --check prototype/data.js`
- `window.BSI_PROTOTYPE.validate()` 期望 `ok: true`
- `window.BSI_PROTOTYPE.getRouteEventPreview(routeId)` 能返回固定路遇、随机候选和预感标签
- `window.BSI_PROTOTYPE.getMapSummary()` 能返回路线预览和 `randomRouteEventPreview`
