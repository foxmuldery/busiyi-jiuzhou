# A_BSI-A-006 资源补给与死局保护规则

> 子线程：A，核心规则与平衡  
> 任务：BSI-A-006  
> 范围：只写规则文档，不修改 WebDemo 原型代码  
> 依据：当前网页版首版方向、`A_BSI-A-002`、`A_BSI-A-004`、`A_BSI-A-005`、当前 `03_WebDemo/prototype/data.js`

## 1. 本文目标

本文补齐 Web Demo 首版的资源、补给、死局保护口径。目标不是把游戏变轻，而是保证玩家输得明白：该紧张时紧张，该补救时能看见补救，不因为迷雾抽卡的坏运气直接卡死。

首版核心句：

```text
车轴管能不能走，粮草管能不能撑，神志管能不能看清路。
资源归零先入濒死补救，不直接死亡；补救失败或反复崩线才判失败。
```

## 2. 三资源总规则

### 2.1 初始值与阈值

Web Demo 首版沿用当前数据。

| 资源 | 上限 | 初始值 | 低值预警 | 危急线 | 归零处理 | 失败主题 |
|---|---:|---:|---:|---:|---|---|
| 车轴 | 100 | 80 | 30 | 12 | 进入断轴濒死补救 | 车停路坏，不能再行 |
| 粮草 | 100 | 90 | 35 | 15 | 进入粮尽濒死补救 | 队伍离散，不能再撑 |
| 神志 | 100 | 85 | 45 | 20 | 进入神志崩线补救 | 路与字散，不能判断 |

结算后统一限制在 0 到 100。若某次结算会低于 0，先压到 0，再进入对应濒死补救。

### 2.2 消耗节奏

首版每次推进通常包含三层消耗：

1. 路线消耗：选择前可见，承担主要压力。
2. 半途路遇：首次多为确定触发，之后轻随机。
3. 地点事件或补给：抵达后由玩家选择，通常带收益和代价。

推荐节奏：

| 阶段 | 资源压力 | 规则目的 |
|---|---|---|
| 前 1-2 次移动 | 低到中 | 让玩家理解车轴、粮草、神志分工 |
| 中段 3-6 次移动 | 中为主，少量高 | 迫使玩家比较路线和补给 |
| 赤水/断碑段 | 神志与锁定压力上升 | 让路线解锁和低神志幻路成立 |
| 裂隙前 | 高风险但应有最后整理 | 让结局选择像“压线抵达” |

低风险路线不应连续无意义，主要承担回撤、补救、教学；高风险路线不应成为唯一出口。

### 2.3 失败条件

资源归零不是立刻失败。首版采用“濒死补救优先”的结构。

| 情况 | 处理 |
|---|---|
| 任一资源 <= 0，且本资源本局濒死次数未超过上限 | 进入对应 `crisisEvents`，暂停普通路线和普通路遇 |
| 濒死补救选择后，该资源仍 <= 0 | 判定硬失败 |
| 同一资源第 3 次进入濒死 | 判定硬失败，除非已经进入结局事件 |
| 三资源同时低于危急线 | 不抽普通风险，强制给喘息/交换/危机选择 |
| 当前地点无可走路线且无补救事件 | 判定为不公平死局，规则上必须提前规避 |
| 玩家主动连续选择高风险、禁忌或低神志危险路径后失败 | 可判定为公平失败 |

建议首版上限：

| 项 | 值 |
|---|---:|
| 同一资源允许濒死补救次数 | 2 |
| 全局允许濒死补救次数 | 4 |
| 濒死后保护步数 | 2 |
| 保护期内同资源高损耗路遇 | 禁止 |

## 3. 地点类型与资源偏向

地点的补给不要像现代商店。它应该像远行队伍在山海旧地里“拆、换、采、祭、记、避”。

| 地点类型 | 主要补什么 | 常牺牲什么 | 风险气质 | 首版数值建议 |
|---|---|---|---|---|
| 粮仓 / 驿仓 | 粮草 | 少量车轴或神志 | 粮可食，但来历不安 | 粮草 +8 到 +16，车轴 -0 到 -3，神志 -0 到 -3 |
| 驿站 | 车轴、粮草 | 粮草、时间感 | 人间秩序最后残影 | 车轴 +6 到 +10，粮草 +6 到 +8，代价 -0 到 -3 |
| 祭所 | 神志、路线线索 | 粮草、坏运 | 以粟问路，以名定心 | 神志 +5 到 +10，粮草 -3 到 -6，可给 flag |
| 墟市 | 粮草、神志、语言线索 | 车轴、粮草、承诺 | 交易可救急，但价格偏斜 | 粮草 +10 到 +15 或神志 +6 到 +10，另一资源 -3 到 -8 |
| 禁地 | 线索、捷径、锁定解除 | 神志、车轴 | 路不是给清醒者走的 | flag/language，神志 -8 到 -12，车轴 -0 到 -5 |
| 山隘 | 车轴风险、少量修补 | 粮草、车轴 | 石路窄，能垫车也会伤车 | 车轴 +4 到 +8 或 -5 到 -10，粮草 -2 到 -5 |

## 4. 补给动作规则

### 4.1 每城次数

Web Demo 首版建议采用：

```text
每个补给 id 每局只能使用一次；
每次抵达城市，最多执行 1 次补给动作；
若以后允许回城复搜，复搜只能给低收益并触发风险。
```

这样既兼容现有 `supplies[].id` 和 `usedSupplies`，也避免玩家在一个城市把所有补给吃空后压力过低。

C 线程实现可以先用简单规则：

| 规则 | 首版建议 |
|---|---|
| 同一 `supply.id` | 每局一次 |
| 同一城市同一次抵达 | 最多选择 1 个补给 |
| 已使用补给 | 地图和路线卡标注“已空”或降低诱因 |
| 回到旧城 | 可显示未用补给，但建议只允许 1 次额外复搜，且加风险 |

如果实现复杂度要压低，也可以先退一步：允许每个 `supply.id` 一次，但路线卡和保底计算不得把已用补给重复计入。

### 4.2 成功 / 失败

首版补给动作不建议做纯随机失败。玩家已经支付资源代价，就应得到可预期结果。

| 类型 | 成功规则 | 失败或代价 |
|---|---|---|
| 普通补给 | 必定成功，按 `effect` 结算 | 支付写明的代价 |
| 风险补给 | 必定获得主收益 | 额外支付神志、车轴或坏运 |
| 禁地补给 | 可获得线索或路线解锁 | 神志损耗更高，可能触发低神志文本 |
| 危机补给 | 必定让最低资源脱离 0 | 代价更重，可增加坏运或旗标 |

首版不要出现“花了粮草祭拜，结果什么也没有”的随机空结果。可以让代价不舒服，但不能让补给动作变成暗坑。

### 4.3 低神志扭曲

神志低可以扭曲感受，但不能让玩家失去基本判断。

| 神志区间 | 补给提示 | 效果规则 |
|---|---|---|
| 61-100 | 明确显示主收益和代价 | 正常结算 |
| 31-60 | 文案可变得不安，仍显示主资源方向 | 正常结算 |
| 1-30 | 可隐藏小代价或显示范围，但必须显示“主要补什么 / 主要损什么” | 禁地、泽国补给可额外神志 -2 或坏运 +5 |
| 0 | 不显示普通补给 | 进入神志濒死补救 |

禁止首版出现：低神志时补给文案完全失真，玩家看不出会扣哪类资源，然后被直接扣死。

### 4.4 时间与风险

Web Demo 首版可以不做复杂时间系统，但补给仍应有“耽搁感”。

建议：

1. 普通补给不额外消耗天数，只结算资源。
2. 危机补给或回城复搜可让 `day + 1`。
3. 若连续 2 次补给或回撤，`badLuckMeter + 5`，表示路上的东西追近。
4. 高收益补给应有代价，不用额外再抽随机惩罚。

## 5. 迷雾路线抽卡保底

迷雾抽卡的保底不是送玩家好路，而是避免“无路可走”。

### 5.1 显现前检查

每次城市显影、路线抽卡前，先做 5 个检查：

| 检查 | 通过标准 | 不通过时 |
|---|---|---|
| 可推进 | 至少 1 条路线能到新地点或关键回撤点 | 强制显出低收益回撤或困境补救 |
| 可承受 | 至少 1 条路线不会让任一资源立即 <= 0 | 降低一条路线消耗或显出补救 |
| 可补救 | 当前最低资源有补给方向可追 | 雾中提示对应地点，如驿、仓、碑、市 |
| 不连压 | 上一条高风险后，本次不全是高风险 | 至少加入 1 条低/中风险 |
| 不误导 | 已搜集补给不算保底 | 改找新补给或危机事件 |

### 5.2 资源危险时的抽卡倾向

| 当前状态 | 抽卡倾向 |
|---|---|
| 车轴 < 30 | 提高驿站、废关、旧道、山隘修补的显现权重 |
| 粮草 < 35 | 提高粮仓、泽国、墟市交易的显现权重 |
| 神志 < 45 | 提高祭所、断碑、界标、静神交易的显现权重 |
| 任一资源 < 危急线 | 下一次显现必须包含补救或低成本出口 |
| 任一资源 = 0 | 不抽普通迷雾路线，进入危机补救 |

### 5.3 仍保留压力的办法

保底路线不能变成免费安全通道。

| 保底方式 | 可以给 | 必须保留的压力 |
|---|---|---|
| 低成本回撤 | 少扣资源，回到已见地 | 不推进主线，补给不刷新 |
| 补救地点显现 | 指向修车、采粮、安神地点 | 路线仍有小消耗或事件 |
| 危机补救 | 把资源拉离 0 | 另一资源扣损、坏运增加、旗标后患 |
| 线索补发 | 给锁定路线缺口 | 通常要付神志或粮草 |
| 高风险替代 | 可绕过锁定 | 明确标注危险，不作为唯一解 |

## 7. 给 C 线程的最小字段建议

不要求本轮改代码；这里只给后续实现口径。

| 字段 | 类型 | 用途 |
|---|---|---|
| `usedSupplies` | 已有 object | 标记每个补给 id 是否用过 |
| `usedSupplyLocations` | 可选 object | 若要限制每城每次只补一次，可记录地点补给状态 |
| `supplyActionTakenThisVisit` | 可选 boolean | 当前抵达本城后是否已补给 |
| `crisisCooldown` | number/object | 濒死后保护步数 |
| `crisisCounts` | object | 三资源各自濒死次数，可复用 `failureStats` |
| `consecutiveBadEvents` | number | 连续坏事件计数 |
| `stepsWithoutPositive` | number | 连续无正向收益计数 |
| `badLuckMeter` | 已有 number | 坏运保底权重 |

最小实现可以只做：

```text
usedSupplies + badLuckMeter + failureStats + crisisCooldown
```

其余字段可先从日志或事件结果派生。

| 抽卡状态 | 当前可见路线中至少有一条车轴消耗 <= 5，或当前城可修车 |
| 玩家行为 | 优先取修车补给 |
| 预期 | 低于危急线时不再只给高车轴消耗路线；车轴归零先触发断轴补救，不直接失败 |

### 8.5 极端坏运气局

## 9. A 线程结论

Web Demo 首版建议采用“看得懂、救得回、仍然紧”的资源规则：

1. 三资源沿用 80 / 90 / 85 初始值，归零先进入濒死补救。
2. 地点补给每个补给 id 每局一次，每次抵达最多做 1 次补给动作。
3. 低神志只污染文案和小代价，不隐藏主资源方向，不制造误杀。
4. 迷雾抽卡必须保证至少一条可推进、可承受或可补救路径。
5. 危机补救要让玩家至少再活 2 步；做不到就说明路线池还要继续修正。

这份规则可直接交给 C 线程作为首版数据与保底逻辑的实现口径。
