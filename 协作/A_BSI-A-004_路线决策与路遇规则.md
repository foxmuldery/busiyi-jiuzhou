# A_BSI-A-004 路线决策与路遇规则

> 子线程：A，玩法系统与平衡  
> 任务：BSI-A-004，九州图路线决策与路遇规则  
> 范围：只写规则文档，不修改 WebDemo 原型代码  
> 依据：`主协调_BSI-UX-004_九州图与路遇交接单.md`，以及当前 `03_WebDemo/prototype/data.js`

## 1. 设计目标

九州图不是普通关卡选择页，而是玩家每一步生存判断的主界面。第一版要验证四件事：

1. 玩家能在出发前判断“这条路会花什么、可能遇到什么、目的地大概有什么”。
2. 路线选择有 Out There 式压力：每次移动都消耗资源，且风险和收益不完全对称。
3. 路遇能打断直达流程，但第一版必须可控、可复现，不能靠纯随机压死玩家。
4. 车轴、粮草、神志三资源在路线、补给、路遇、地点事件里各有明确分工。

当前 WebDemo 数据已经具备可用骨架：

| 项 | 当前状态 | A 线程判断 |
|---|---|---|
| 地点 | 8 个地点，均有 `map.x/y/step`、`terrain`、`supplies` | 足够做首版九州图 |
| 路线 | 10 条路线，含低/中/高风险、消耗、提示、部分 `midEvent` | 可直接作为第一版测试集 |
| 路遇 | 3 个：`wheel_omen`、`roadside_shrine`、`black_cloud` | 适合做确定触发和半随机池 |
| 锁定路线 | `red_to_rift` 使用 `requireAny` | 可作为第一版锁定条件样例 |
| 资源初始 | 车轴 80，粮草 90，神志 85 | 与 A-001/A-002 完整线一致 |

## 2. Out There 式路线决策信息模型

路线决策应由五层信息组成：当前位置、候选路线、目的地观察、路遇预期、锁定与后果。

### 2.1 当前节点信息

当前节点负责告诉玩家“我在哪里，以及还能承担什么风险”。

| 信息 | UI 表达 | 规则用途 |
|---|---|---|
| 地点名 / 区域 | 如“中原驿 / 人间边境” | 建立九州段落感 |
| 地貌 | `road / market / water / rift` | 决定路线风险、路遇池、补给倾向 |
| 西行进度 | `map.step` 或距离条 | 告诉玩家离九州裂隙还有多远 |
| 当前资源 | 车轴、粮草、神志 | 判断是否能走、是否应补给 |
| 已搜集补给 | `usedSupplies` | 防止玩家误以为能重复刷资源 |
| 当前旗标 | 如 `followed_bones`、`rift_name` | 决定锁定路线是否可见或可走 |

### 2.2 候选路线信息

每条路线是一次“有代价的承诺”。玩家点击路线前，应至少能看到：

| 字段 | 数据来源 | 是否必须显示 | 说明 |
|---|---|---|---|
| 路线名 | `route.name` | 必须 | 例如“循旧王道西行” |
| 目的地名 | `route.to -> location.name` | 必须 | 让玩家知道不是盲跳 |
| 地貌 | `route.terrain` | 必须 | 可用图标或短标签 |
| 风险等级 | `route.risk` | 必须 | 低 / 中 / 高 |
| 路线消耗 | `route.cost` | 必须 | 三资源明确数值 |
| 路线提示 | `route.hint` | 必须 | 解释风险来源，不写成纯数值 |
| 目的地可观察资源 | `to.supplies[].hint` 汇总 | 必须 | 只显示“可观察倾向”，不等于已经获得 |
| 路遇提示 | `route.midEvent` 或规则推断 | 必须 | 第一版显示“半途可能停下”或具体征兆 |
| 锁定条件 | `requireAny` / `lockedHint` | 必须 | 不可走时说明原因 |

### 2.3 目的地可观察资源

目的地资源不是完整剧透，而是路线选择前的侦察信息。建议按“资源倾向 + 代价倾向”显示。

| 目的地 | 当前补给倾向 | 推荐观察文案口径 |
|---|---|---|
| 中原驿 | 车轴、粮草 | 有驿钉与余粮，可作出发补整 |
| 故王道 | 车轴，少量粮草代价 | 旧道石和贡绳能稳车，但路声不安 |
| 废关 | 高车轴，神志代价 | 可拆关木铁钉，旧令会压神志 |
| 雷泽浅畔 | 高粮草，车轴或神志代价 | 泽中可采食，泥水伤车或扰神 |
| 黑齿市 | 高粮草、神志，交易代价 | 可交易补给，但价格未必只在资源上 |
| 赤水外滩 | 少量粮草，高神志代价 | 岸草可食，赤水污染明显 |
| 巫咸断碑 | 神志恢复，粮草/车轴代价 | 碑文可定方正名，但需要消耗 |
| 九州裂隙 | 神志恢复，粮草代价 | 残图可整理，接近结局判断 |

UI 上不要直接写“这里有 +10 粮草可拿”作为唯一信息。可以显示：

```text
可观察补给：粮草偏多 / 可能损神志
已搜集：否
```

当目的地补给已被搜集后，路线卡仍可显示地点倾向，但必须标注“此地补给已空”。

## 3. 路线卡展示规则

路线卡建议分为六个固定区块，保证玩家能快速比较。

### 3.1 标准路线卡

```text
循旧王道西行 -> 故王道
风险：低    地貌：旧道
消耗：车轴 -4 / 粮草 -4 / 神志 0
半途：轮声征兆，可能停车处理
目的地：可修车；可能消耗粮草或神志
提示：直道尚平，适合作为教程段
```

### 3.2 锁定路线卡

锁定路线必须保留可见性，让玩家知道存在目标和缺口。

```text
夜近裂隙 -> 九州裂隙
风险：高    地貌：裂隙
消耗：车轴 -12 / 粮草 -9 / 神志 -14
状态：未满足
条件：需沿骨行路、读出裂隙名，或神志低至能看见幻路
```

第一版不建议把锁定路线完全隐藏。隐藏会削弱“离九州裂隙还差什么”的战略感。

### 3.3 低神志时的信息污染边界

神志低可以污染文案，但不能破坏基本决策。

| 神志区间 | 路线卡显示 |
|---|---|
| 61-100 | 显示完整消耗、风险、目的地观察 |
| 31-60 | 副风险文案变模糊，但主消耗仍明确 |
| 1-30 | 路遇提示和目的地文案可异常化，但风险等级、至少一种主消耗必须保留 |
| 0 | 不显示普通路线选择，进入神志濒死补救 |

禁止第一版出现“所有路线提示同时失真，且其中一条会致死”的情况。

## 4. 路遇触发规则第一版

第一版路遇采用“确定触发为主，轻随机为辅，保底兜底”的结构。这样 C 线程容易实现，主协调也容易验收。

### 4.1 路遇类型

| 类型 | 规则 | 例子 | 第一版用途 |
|---|---|---|---|
| 确定触发 | 路线配置了 `midEvent`，且该路线事件未触发过 | `central_to_road -> wheel_omen` | 保证玩家第一次体验半途停车 |
| 随机触发 | 无 `midEvent` 或已触发过时，按风险/地貌抽取 | `road_to_pass` 可抽低风险路遇 | 增加重玩变化 |
| 保底触发 | 资源危急、连续坏事件、无正向收益过久时插入 | 补给/喘息/低代价线索 | 防止随机死局 |
| 禁止触发 | 濒死后保护期、结局前强制事件、当前无可承受选项 | 不抽高损耗路遇 | 保证补救有效 |

### 4.2 确定触发

当前 `data.js` 中已有 `midEvent` 的路线，建议第一版全部按“首次必触发”处理：

| 路线 | 路遇 | 触发建议 |
|---|---|---|
| `central_to_road` | `wheel_omen` | 首次必触发，作为路遇教学 |
| `central_to_pass` | `roadside_shrine` | 首次必触发，展示献粟/绕行选择 |
| `road_to_marsh` | `black_cloud` | 首次必触发，展示泽地危险 |
| `pass_to_market` | `roadside_shrine` | 若同事件已触发过，可改为 40% 复现或跳过 |
| `marsh_to_red` | `black_cloud` | 高风险路线首次必触发 |
| `market_to_red` | `black_cloud` | 高风险路线首次必触发 |

同一个 `route.id` 的 `midEvent` 每局只确定触发一次。若未来允许回头路，第二次走同路线不再必触发。

同一个 `routeEvent.id` 在一局内最多出现 2 次。第二次出现时应当使用不同文本层级或不同结果权重，避免重复感。

### 4.3 随机触发

无 `midEvent` 的路线，或 `midEvent` 已被该路线触发过后，可走轻随机。

| 风险 | 路遇触发率 | 可抽事件 |
|---|---:|---|
| 低 | 15% | 低损耗、线索、轻补给、轻神志 |
| 中 | 35% | 当前地貌匹配事件、交易、风险/补给二选一 |
| 高 | 60% | 地貌风险、神志污染、关键线索、高收益高代价 |

第一版随机池可以先不做大池，只在 3 个现有路遇中按地貌匹配：

| 地貌 | 优先路遇 | 说明 |
|---|---|---|
| `road` | `wheel_omen` | 车轴与道路征兆 |
| `market` | `roadside_shrine` | 交易、祭祀、绕路 |
| `water` | `black_cloud` | 泽地、水气、雷云 |
| `rift` | 暂不随机，优先地点事件 | 裂隙段避免半途随机冲淡结局 |

### 4.4 保底触发

保底不是送资源，而是恢复“还能做选择”的空间。

按优先级从高到低处理：

| 优先级 | 触发条件 | 动作 |
|---:|---|---|
| 1 | 任一资源 <= 0 | 不触发普通路遇，进入对应 `crisisEvents` |
| 2 | 所有可见路线都会导致某资源 <= 0 | 插入补救路遇或显示低成本回撤 |
| 3 | 任一资源 < 15 | 下一次路遇必须是补给/喘息/低代价线索 |
| 4 | 连续 2 个坏事件 | 下一次路遇不得为高损耗风险 |
| 5 | 连续 3 步无正向收益 | 下一次事件必须有 `positiveTag` 效果 |
| 6 | 刚完成濒死补救后的 2 步 | 禁止抽同资源高损耗事件 |

坏事件第一版定义：

```text
没有获得资源、旗标、语言、路线线索或结局进展，
且总资源净损耗 >= 8，
或任一资源单项净损耗 >= 6。
```

正向收益包括：资源净增、获得 flag、获得 language、解锁路线、获得明确目的地提示、完成补救。

### 4.5 避免死局

每次显示九州图前，系统应做一次“能否继续走”的检查。

| 检查项 | 通过标准 | 不通过时 |
|---|---|---|
| 有可见路线 | 当前地点至少 1 条可见路线 | 生成回撤/困境事件 |
| 有可承受路线 | 至少 1 条路线不会立即让资源 <= 0，或有补救 | 插入对应资源补救 |
| 锁定路线有替代 | 关键终点至少 2 种解锁来源或 1 条绕行线 | 显示缺口，不直接卡死 |
| 补给不被误判 | 已搜集地点不再作为可获得补给计算 | 标注“已搜集” |
| 高风险后有出口 | 高风险路线抵达后至少有 1 条低/中风险出口或地点补救 | 调整路线或插入地点补救 |

当前 `red_to_rift` 的 `requireAny` 是好的锁定样例：`followed_bones`、`rift_name`、`sanityMax: 34` 三种入口分别对应地点事件、碑文事件、低神志幻路。建议后续所有主线锁定都采用“至少两条清醒路径 + 一条危险认知路径”的结构。

## 5. 三资源数值边界建议

本节按当前 WebDemo 完整线给边界，不采用 `A_首版可运行最小数值口径.md` 的宽松临时线。当前 `data.js` 初始值为车轴 80、粮草 90、神志 85，可以继续使用。

### 5.1 全局阈值

| 资源 | 上限 | 初始 | 低值预警 | 危急线 | 归零处理 |
|---|---:|---:|---:|---:|---|
| 车轴 | 100 | 80 | 30 | 12 | 进入断轴补救 |
| 粮草 | 100 | 90 | 35 | 15 | 进入粮草补救 |
| 神志 | 100 | 85 | 45 | 20 | 进入神志补救 |

资源效果结算后建议统一 clamp 到 0-100。若单次结果会低于 0，先进入濒死补救，不要直接失败。

### 5.2 路线消耗边界

| 风险 | 车轴 | 粮草 | 神志 | 当前数据是否匹配 |
|---|---:|---:|---:|---|
| 低 | -3 到 -5 | -3 到 -5 | 0 到 -2 | 匹配：如 `central_to_road`、`road_to_pass` |
| 中 | -4 到 -9 | -5 到 -8 | -2 到 -7 | 基本匹配：`market_to_stele` 神志 -7 是上沿 |
| 高 | -7 到 -12 | -8 到 -11 | -8 到 -14 | 匹配：`red_to_rift` 神志 -14 是上限 |

建议第一版不要让单条路线的任一资源消耗超过当前 `red_to_rift` 的强度：车轴 -12、粮草 -11、神志 -14。超过这个数值应当只出现在玩家主动选择的地点事件或结局代价里。

### 5.3 补给边界

地点补给目前范围健康：多数是单项 +4 到 +12，代价 -2 到 -5。建议保持以下边界。

| 补给类型 | 推荐收益 | 推荐代价 | 说明 |
|---|---:|---:|---|
| 小修车 | 车轴 +4 到 +7 | 粮草 -2 或神志 -2 | 普通地点补给 |
| 大修车 | 车轴 +8 到 +12 | 神志 -3 到 -5，或粮草 -4 | 废关、驿站等强地点 |
| 小粮草 | 粮草 +6 到 +8 | 车轴 -2 到 -3 或神志 -2 | 常规采集 |
| 大粮草 | 粮草 +10 到 +15 | 车轴 -3 到 -5 或神志 -3 到 -5 | 泽地、交易 |
| 小安神 | 神志 +4 到 +6 | 粮草 -2 或车轴 -2 | 碑文、记录、祭祀 |
| 大安神 | 神志 +8 到 +12 | 粮草 -4 到 -6 | 市集药粉、裂隙前整理 |

地点补给只可搜集一次，因此单个地点可以给到 +12 或 +15；但若未来出现可重复休整，重复项每次收益应降到 +4 到 +8，并带冷却或递增代价。

### 5.4 路遇边界

当前三个路遇都比较温和，适合第一版。建议保持：

| 路遇类型 | 单项收益上限 | 单项损耗上限 | 说明 |
|---|---:|---:|---|
| 教学路遇 | +5 | -5 | 如 `wheel_omen`，让玩家理解停车/赶路 |
| 中风险路遇 | +6 | -6 | 如 `roadside_shrine`，用粮换神志或绕路伤轴 |
| 高风险路遇 | +6 | -8 | 如 `black_cloud`，高风险路线可稍重 |
| 关键线索路遇 | +0 到 +4 | -8 到 -12 | 必须给 flag/language/路线解锁 |
| 保底路遇 | +10 到 +18 | -4 到 -6 | 只在危急时出现，救最低资源 |

非主动路遇不要让任一资源单项损耗超过 -8。若玩家明确选择“强行、献祭、夜行、凝视、接受幻路”，可放宽到 -12。

### 5.5 地点事件边界

地点事件比路遇更像剧情选择，可以承担更重代价，但必须有明显提示。

| 事件选择类型 | 推荐资源范围 | 示例 |
|---|---|---|
| 普通安全选择 | 总净损耗 0 到 -6 | 整点车具、验土再走 |
| 普通风险选择 | 总净损耗 -6 到 -12 | 沿深辙快行、拓碑带走 |
| 高风险换线索 | 单项 -8 到 -12，必须给 flag/language | 听雷辨路、沿骨行路、让碑补全名字 |
| 结局选择 | 可高代价或直接 ending | 按梦中地图前进 |
| 试演/调试选择 | 必须标注，不进正式平衡 | 当前“神志 -75”只能作为纵切片试演 |

当前 `red_bones` 中“凝视赤水辨路 / 神志 -75”不应进入正式首版平衡。若保留在 WebDemo，应明确作为测试项或特殊演示项，不能让普通玩家误点后认为系统不公平。

## 6. 当前 WebDemo 路线平衡备注

### 6.1 推荐主路压力

以当前路线估算，一条较清醒主路为：

```text
中原驿 -> 故王道 -> 雷泽浅畔 -> 赤水外滩 -> 巫咸断碑 -> 九州裂隙
```

纯路线消耗约为：

| 资源 | 总消耗 |
|---|---:|
| 车轴 | -38 |
| 粮草 | -32 |
| 神志 | -29 |

从初始 80/90/85 出发，理论可承受，并且中途需要玩家使用 1-2 次补给或选择较温和的路遇。这符合“能通、会紧张、不随机死局”的首版目标。

另一条更危险主路：

```text
中原驿 -> 废关 -> 黑齿市 -> 赤水外滩 -> 九州裂隙
```

纯路线消耗约为：

| 资源 | 总消耗 |
|---|---:|
| 车轴 | -29 |
| 粮草 | -32 |
| 神志 | -33 |

这条路车轴压力较小，但粮草和神志压力更高，且 `red_to_rift` 有锁定条件，适合做“交易/禁忌换捷径”的分支。

### 6.2 需要 C 线程注意但 A 不改代码

1. 路线消耗应在确认移动时结算，路遇结果在半途结算，地点补给/地点事件在抵达后结算。
2. `routeEventResults` 应记录至少 `route.id` 和 `event.id` 两层，避免同一事件在不同路线上的触发状态混乱。
3. `pendingRoute` 应只用于半途路遇未处理时，防止玩家绕过路遇直接抵达。
4. `usedSupplies` 必须影响路线卡的目的地资源观察，避免玩家把已空地点当成补给点。
5. `requireAny` 的 `sanityMax` 需要明确是“当前神志 <= 值”，路线卡文案要避免误读。

## 7. 给主协调的验收要点

主协调验收 A/C/D 线程整合时，建议按以下清单看：

| 验收项 | 通过标准 |
|---|---|
| 路线卡可比较 | 玩家能同时看到消耗、风险、目的地补给倾向、路遇提示、锁定条件 |
| 起点教学成立 | 从中原驿走任一路线，玩家能理解路线消耗和半途路遇 |
| 半途停车成立 | 触发路遇后必须先处理，再抵达目的地 |
| 锁定路线不糊 | `red_to_rift` 未满足时仍可见，并显示明确缺口 |
| 补给一次性清楚 | 已搜集地点不再被路线卡暗示为可重复补给 |
| 保底不破坏难度 | 资源危急时给选择空间，但不直接免费满资源 |
| 不出现随机死局 | 任一资源归零前至少有补救、回撤或明确自愿高风险选择 |
| 高风险有回声 | 高风险路线后不能只接高风险路线或立即无解 |

## 8. 需要用户决策的问题

1. 路遇第一次是否“有 `midEvent` 的路线必触发”？
   - A 线程建议：是。第一版需要稳定验收半途停车流程。

2. 路遇是否允许正收益？
   - A 线程建议：允许，但正收益必须带代价或信息收益定位。路遇不应全是惩罚，否则玩家会把半途停车理解为系统刁难。

3. 锁定路线是隐藏还是灰显？
   - A 线程建议：灰显。九州图的战略感来自知道“那里有路，但我还差条件”。

4. 低神志路线是否可以显示错误数值？
   - A 线程建议：第一版不要。可以污染文案，但不要让错误数值成为主要失败原因。

5. 当前 `red_bones` 的“神志 -75”是否保留？
   - A 线程建议：只作为纵切片试演/调试选项保留；正式 WebDemo 应改成更小代价或加二次确认。

6. 是否允许回头路？
   - A 线程建议：第一版可以允许局部回撤，但回撤路线要低收益、低风险，主要用于防死局，不做刷补给。

## 9. A 线程结论

当前 WebDemo 数据已经足够支撑第一版 Out There 式九州图。A 线程建议本轮不要再扩路线数量，先把路线卡信息、首次必触发路遇、锁定路线灰显、补给一次性观察、资源危急保底这五件事做稳。

第一版核心口径：

```text
看得懂代价 -> 选得出路线 -> 半途可能停 -> 处理路遇 -> 抵达地点 -> 一次性补给/地点事件 -> 回到九州图
```

只要这个闭环成立，后续再扩事件池、权重随机和更复杂的九州图叙事。
