# D_BSI-D-008 羊皮纸九州图体验方案

> 子线程：D 视觉体验与交互氛围设计  
> 对应主协调任务：BSI-UX-009  
> 项目：《不思异：九州》Web Demo  
> 日期：2026-06-15  
> 边界：本文件只写体验方案与 C 线程落地清单，不修改 `index.html`、`styles.css`、`app.js`

## 0. 本轮结论

用户最新意见可以收成一句话：

```text
战略选择参考 Out There 的大界面压力，但视觉必须是羊皮纸山海舆图；到新城市后，雾散、城显、路显。
```

这次不要把九州图做成“完整路线图”。它应该像一张正在被车队走出来的旧舆图：已走之处有车辙，当前城市有朱印，近处可走路线被描亮，远处只露出雾中影，锁定路线像被封住的断墨。

核心体验目标：

1. 玩家每次打开九州图，都觉得自己在做一次大界面战略选择。
2. 玩家只能看清当前已知区域和下一轮可走前路。
3. 到达新城市后，地图有明确的“显影”反馈。
4. 路线随机和抽卡感来自“后续路线此刻才显现”，不是把全地图提前摊开。
5. 借鉴 Out There 的决策结构，不复制星图、太空、图标和 UI 表达。

## 1. 羊皮纸 / 山海舆图的大界面视觉结构

九州图模式应是一个完整的战略界面，不是旅途页里的小地图。现有 `mapView`、`.map-panel`、`.map-layout`、`.kyushu-map`、`.map-route-panel` 已经能承接，不需要推翻结构。

### 1.1 主画面层级

建议视觉层级从底到顶：

| 层级 | 现有结构 | 表现 |
| --- | --- | --- |
| 羊皮纸底 | `.kyushu-map` | 旧纸、帛书、轻微污渍、边缘暗角，整体偏暖纸色而不是深色科技底 |
| 山海底纹 | `.map-grid` | 从网格改成舆图经纬、折痕、山川淡线、河泽墨痕；保留“坐标感”，但不显得像后台网格 |
| 路线层 | `.map-routes-layer` / `.map-route-line` | 墨线、朱砂线、旧车辙、断线、雾中虚线 |
| 节点层 | `.map-node-layer` / `.map-node` | 地名签、朱印、关隘印、残卷边角；少字，避免流程图卡片感 |
| 雾层 | 可用 `.kyushu-map::before/::after` 或新增轻量层 | 半透明纸雾、墨晕、水渍、残缺遮罩，雾区不能完全黑掉 |
| 决策签 | `.map-route-panel` / `.route-card` | 像路引签、残卷条、驿牒；承担消耗、风险、补给、锁定原因 |

第一版重点不是画复杂山水，而是把气质换对：

- 背景从“深色网格地图”转成“暖旧纸 + 淡墨地势”。
- 路线从“发光连线”转成“墨线 / 朱砂 / 车辙”。
- 节点从“信息卡”转成“舆图标记”。
- 右侧路线卡从“网页列表”转成“出发前抽出的几张路引签”。

### 1.2 构图比例

桌面和常规横屏：

1. `.kyushu-map` 占主面积，至少 65%-70% 宽度。
2. `.map-route-panel` 在右侧，像放在地图旁边的路引签，不要比地图更抢。
3. `.map-panel > .panel-title` 保留短标题即可，避免“地图说明页”感。
4. 地图内部不放长说明，详情交给 `.route-card`。

小横屏：

1. 现有 CSS 已把 `.map-layout` 变成上地图、下路线横条，这是正确方向。
2. 地图仍是主视觉，路线卡横向滑动。
3. 节点只留地名或符号，隐藏消耗和补给小字。

## 2. 节点状态视觉表现

第一版建议保留 5 个玩家可理解状态，再兼容现有代码里的 `pending / future`。

| 玩家语义 | 现有类名承接 | 视觉表现 | 交互 |
| --- | --- | --- | --- |
| 当前节点 | `.map-node.current` | 朱砂大印 + 清晰地名 + 车队小标或小旗；纸面周围有淡金圈 | 不一定可点，是锚点 |
| 已见节点 | `.map-node.visited` | 清晰墨线地名签，颜色比当前低；可有浅车辙或小印 | 可复看，但不作为当前出发动作 |
| 可达节点 | `.map-node.reachable` | 半亮地名签，边缘有金墨或朱砂描边；像“刚被描出来” | 可点，点后高亮对应 `.route-card` 或直接出发 |
| 雾中节点 | `.map-node.future`，后续可增 `.hinted` | 地名不完整或只露轮廓；模糊墨点、残缺山影、问号不要太现代 | 不可点，最多提示“雾中有城影” |
| 锁定节点 | `.map-node.locked` | 断墨边、红墨封记、斜向封条；地名可见但压暗 | 不可走，点击/路线卡显示条件 |
| 抵达中 | `.map-node.pending` | 目的地边缘有墨晕扩散，像正在显影 | 不可重复点，提示先处理半途路遇 |

### 2.1 当前节点

当前城市必须一眼能找到。建议：

1. 节点主体用朱印或关防印，不用发光圆点。
2. 地名清晰，字号略大。
3. 节点下方可有短词：`车队所在`。
4. 周围一圈淡金或淡朱砂晕，表示视野中心。
5. 小横屏下只保留地名 + 朱印，不显示说明。

### 2.2 已见节点

已见节点不是可选项，应该安静：

1. 墨色清楚但不高亮。
2. 可以有浅车辙或小勾印表示已录入地点志。
3. 不要和可达节点同等亮度，否则玩家会误点。
4. 现有代码已输出 `.visited`，CSS 里建议补一个明确样式。

### 2.3 可达节点

可达节点是玩家的注意力目标：

1. 节点边缘用金墨或浅朱砂描亮。
2. 鼠标/触摸聚焦时，对应路线线条和路线卡同步高亮。
3. 不要把所有可达节点做成按钮卡；更像地图上的地名签。
4. 可达节点只来自当前城市的后续路线，不能让远方所有点都像可点。

### 2.4 雾中节点

雾中节点负责神秘感和抽卡感：

1. 不完整显示地名，优先用“泽中影”“西北有城”“残卷缺字”这类短提示。
2. 节点边缘被纸雾遮住，透明度低，边框不完整。
3. 不使用现代问号图标，不使用星球轮廓。
4. 如果技术上第一版仍需保留所有节点 DOM，至少让 `.map-node.future` 视觉上退到雾后。

### 2.5 锁定节点

锁定不是雾中，玩家应知道“有这条路，但现在走不了”：

1. 地名可读，但压暗。
2. 用断墨、红封、裂纹表达条件不足。
3. 锁定原因放路线卡，不把长条件塞在节点上。
4. 点击锁定路线时只给反馈，不执行移动。

## 3. 路线状态视觉表现

路线比节点更能表达“地图正在被走出来”。建议 4 个基础状态加一个抵达中状态。

| 玩家语义 | 现有类名承接 | 视觉表现 |
| --- | --- | --- |
| 已走路线 | `.map-route-line.traveled` | 淡墨车辙 / 旧折痕，低饱和，不发光 |
| 可走路线 | `.map-route-line.reachable` | 朱砂或金墨实线，略粗，像新描出的路引 |
| 雾中路线 | `.map-route-line.unknown` | 断续虚线、墨晕残影、被纸雾遮断 |
| 锁定路线 | `.map-route-line.locked` | 断裂红墨、封条感、局部裂开 |
| 抵达中路线 | `.map-route-line.pending` | 墨线正在晕开或轻微流动，提示车队在半途 |

### 3.1 已走路线

已走路线用来让玩家形成旅程记忆：

1. 不要继续高亮，避免误以为还能选择。
2. 可以像旧车辙、浅墨、纸面压痕。
3. 和已见节点共同构成“我来过”的区域。

### 3.2 可走路线

可走路线是出发选择：

1. 从当前城市向外延伸，颜色最明确。
2. 视觉上可以像朱砂批注，表示“此刻抽出的可行路引”。
3. 鼠标/触摸路线卡时，对应线条加粗或稍亮。
4. 路线方向不必用箭头，可以用线尾淡出、路签朝向或节点高亮表达。

### 3.3 雾中路线

雾中路线不能透露完整路径：

1. 只露一小段或若隐若现的断线。
2. 不显示完整 `from -> to` 的强关系。
3. 可提示方向和气质，例如“西北有旧道影”“泽边墨迹未干”。
4. 视觉上要服务随机性：玩家知道远方有可能，但不知道下一轮会抽到什么。

### 3.4 锁定路线

锁定路线的重点是“诱惑 + 条件”：

1. 路线存在感比雾中强，但比可走弱。
2. 红封或断墨让玩家知道不是普通未知。
3. `.route-card.locked` 必须显示简短条件，例如“需白羽语片”“神志低于 35”“需沿骨行路”。
4. 锁定高收益路线可以保留地名和风险，形成目标感。

## 4. 到达新城市后的“雾散、城显、路显”反馈

到达新城市不要只是数值变了、节点变色。需要一个短而清楚的三段反馈，建议总时长 1.2-1.8 秒，足够有仪式感但不拖慢节奏。

### 4.1 反馈顺序

```text
抵达触发
-> 雾散：当前城市周围纸雾退开
-> 城显：城市标记和地名写出，朱印落下
-> 路显：从新城市出发的可走路线被描亮，远方新增雾中影
-> 路线卡刷新：抽出的路引签进入右侧/底部路线栏
```

### 4.2 雾散

表现：

1. 以新城市为中心，纸雾向外淡开。
2. 可用墨晕、水渍、纸纤维边缘，不用科技扫描圈。
3. 雾散范围只覆盖当前城市周边和本轮可见前路，不揭完整地图。
4. 神志低时，雾散可以带一点血色或错字，但不能影响可读性。

C 线程落点：

- `.kyushu-map` 可用伪元素做全局纸雾。
- 给刚抵达节点临时加类，如 `.is-revealing` 或 `.just-revealed`。
- 不一定第一版做复杂遮罩，先做“局部淡雾 + 节点/路线进入动画”即可。

### 4.3 城显

表现：

1. 城市节点从模糊墨点变成可读地名签。
2. 地名像被写出，不像 UI 弹窗出现。
3. 当前节点落朱印，已见节点退成淡墨。
4. 地点志可同步出现一条短反馈：`新见闻：废关已录入地点志。`

C 线程落点：

- `.map-node.pending` 到 `.map-node.current` 的过渡要明显。
- `.map-node.visited` 要有独立低亮样式，承接旧当前点。
- 小横屏不做长文字动画，只做节点亮起和地名出现。

### 4.4 路显

表现：

1. 从新当前节点向外，1-3 条可走路线被朱砂/金墨描出。
2. 锁定路线如果出现，像被红封压住，不和可走路线混淆。
3. 更远处只新增雾中节点或短路线影，不显示完整目的地关系。
4. 右侧 `.route-card` 像抽出的路引签依次进入，形成抽卡反馈。

C 线程落点：

- `.map-route-line.reachable` 可用线条描入动画。
- `.map-route-line.unknown` 保持断续低透明。
- `renderRoutes()` 刷新后，`.route-card` 可以做轻微上移/淡入，但不要弹得像手游抽卡。

## 5. 如何借鉴 Out There 的大界面战略选择感

可以借鉴的是“决策压力”，不是视觉表达。

### 5.1 要借鉴的结构

1. 大界面：玩家切到九州图后，当前操作只围绕“下一段路怎么走”。
2. 资源压力：每条路线都展示车轴、粮草、神志消耗。
3. 风险比较：路线风险和半途路遇是选择前必须看见的信息。
4. 未知探索：远方不完全显示，下一轮路线到达后才展开。
5. 诱惑路线：有些高价值路线被条件锁住，形成短期目标。
6. 回合节奏：选路、上路、半途或抵达、再开图选路。

### 5.2 不能复制的表达

1. 不做星图、星球、航线、宇宙背景。
2. 不复制太空资源图标、舰船 UI、圆形星系节点。
3. 不用科幻扫描、雷达、HUD 线框表达迷雾。
4. 不照搬文案结构，不写“山海经版 Out There”。
5. 不把地图做成现代导航或 RTS 大地图。

### 5.3 九州自己的等价表达

| Out There 的功能 | 《不思异：九州》的表达 |
| --- | --- |
| 星系选择 | 驿、关、泽、墟、祭所、裂隙前沿 |
| 航线 | 古道、水路、山径、王道、幻路、秘径 |
| 飞船资源压力 | 车轴、粮草、神志 |
| 未知星域 | 雾中地名、残卷缺页、墨晕山影 |
| 锁定跳跃 | 语言、器物、地志线索、神志阈值 |
| 深空危险 | 异象、断轴、饥荒、失神、裂隙污染 |

最终玩家感受到的应该是：

```text
我在旧舆图上决定车队下一段命运。
```

而不是：

```text
我在古风皮肤的星图上点星球。
```

## 6. 给 C 线程的落地清单

以下按现有结构收口，优先 CSS 和少量状态类，不要求本轮重写玩法。

| 结构 / 类名 | 建议动作 | 验收信号 |
| --- | --- | --- |
| `.map-panel` | 弱化网页面板感，标题压短；九州图像游戏内战略界面 | 第一眼不是后台地图页 |
| `.map-layout` | 桌面保持地图主、路线辅；小横屏继续上地图下路线横条 | 地图承担探索，路线卡承担决策 |
| `.kyushu-map` | 改成羊皮纸底：暖纸色、暗角、污渍、轻山水纹 | 不像太空星图或科技网格 |
| `.map-grid` | 从方格改成舆图经纬/折痕/淡墨山川；可保留很弱坐标感 | 有地图结构，但不显后台 |
| `.map-routes-layer` | 保持路线独立层，便于做雾中/可走/锁定状态 | 路线状态清楚 |
| `.map-node-layer` | 保持节点独立层，节点文字减少 | 地图不变成流程图 |
| `.map-node` | 从信息卡改成地名签/舆图标记；默认少字 | 节点不像小面板 |
| `.map-node.current` | 朱印、车队标、最清晰地名 | 当前城市 1 秒内可找到 |
| `.map-node.visited` | 新增明确样式：淡墨、已见、低亮 | 已见和可达不混淆 |
| `.map-node.reachable` | 金墨/朱砂描边，允许点击 | 玩家知道这是下一站候选 |
| `.map-node.future` | 作为第一版雾中节点：模糊、残缺、低透明 | 远方有神秘感，但不可误点 |
| `.map-node.locked` | 红封、断墨、压暗 | 玩家知道有路但条件不足 |
| `.map-node.pending` | 墨晕显影、抵达中 | 半途状态不误判成已到达 |
| `.map-route-line.traveled` | 淡墨车辙或纸痕 | 已走路线形成旅程记忆 |
| `.map-route-line.reachable` | 朱砂/金墨实线，可做描入动画 | 可走路线最明确 |
| `.map-route-line.unknown` | 雾中断线、低透明、局部露出 | 不提前泄露全地图 |
| `.map-route-line.locked` | 红色断裂线或封印线 | 条件不足一眼可知 |
| `.map-route-line.pending` | 轻微墨晕流动或高亮 | 玩家知道车队在这段路上 |
| `.map-route-panel` | 定义为路引签栏，不是普通侧栏 | 路线卡是战略选择工具 |
| `.route-list` | 桌面竖排可比较；小横屏横向滚动 | 多路线可读、不挤压地图 |
| `.route-card` | 固定信息顺序：路线名 / 去向 / 消耗 / 风险与半途 / 目的地补给 / 提示 | 每张卡能独立支持决策 |
| `.route-card.locked` | 显示锁定原因，视觉像封住的路引 | 锁定不是坏按钮，而是目标提示 |

### 6.1 现有状态映射建议

现有 `renderMap()` 已输出：

- `.map-route-line.traveled`
- `.map-route-line.reachable`
- `.map-route-line.pending`
- `.map-route-line.locked`
- `.map-route-line.unknown`
- `.map-node.current`
- `.map-node.pending`
- `.map-node.reachable`
- `.map-node.locked`
- `.map-node.visited`
- `.map-node.future`

第一版可以先用这些状态做完整视觉，不急着新增大量 DOM。若 C 线程要做更精细的迷雾，再考虑新增：

- `.map-node.hinted`：方向已露但城市未明。
- `.map-route-line.hinted`：只露方向，不露完整终点。
- `.is-revealing`：抵达新城市后的临时动画类。
- `.just-revealed`：本轮新显现节点/路线的短暂强调类。

### 6.2 迷雾与随机性的实现边界

D 线程建议第一版的体验边界：

1. 不必一次做真正程序生成世界。
2. 先让“当前后续路线显现、远方雾中影出现”成立。
3. 不要在视觉上展示全地图完整路线。
4. 即使 DOM 中存在全部节点，CSS/状态也要让未揭示部分退进雾里。
5. 路线卡只列当前城市可抽出的候选路线，不能列全局路线。

### 6.3 给 C 线程的实现优先级

根据 `BSI-UX-010` 的持续推进要求，D 线程不等待 A/B/C/E/F 完成，先给 C 线程一份可直接排期的 UI 优先级：

| 优先级 | 内容 | 最小完成标准 |
| --- | --- | --- |

## 7. 小横屏下如何保持可读

小横屏的目标不是展示更多，而是保住三件事：

```text
当前在哪里 / 哪几条路可走 / 每条路的关键代价是什么
```

### 7.1 地图

1. `.kyushu-map` 保持在上方，占主要高度。
2. `.map-node` 小横屏只显示地名或 2-4 字短名。
3. 隐藏 `.map-node span`、`.map-node small` 是正确方向，详情交给路线卡。
4. 当前节点和可达节点必须比雾中节点更清楚，不能全部缩成同样的小块。
5. 雾层透明度要降低，不能把小屏地名糊掉。
6. 节点触控热区保持至少 38-44px；视觉可以小，按钮不要太小。

### 7.2 路线卡

1. 现有小横屏 `.route-list` 横向滚动可保留。
2. `.route-card` 宽度建议控制在 240-270px，避免一屏只露半张卡。
3. 每张卡小屏只保留：路线名、去向、三资源消耗、风险、锁定原因。
4. 目的地补给和长提示可以压成一行，必要时省略。
5. 锁定卡不要占太多解释空间，显示短条件即可。

### 7.3 字和状态

1. 不用更小的密集长文换信息量；小屏优先删字。
2. 当前 / 可达 / 雾中 / 锁定的差异主要靠色、线型、透明度和封记，不靠说明文字。
3. 地名超过 4-5 字时，小屏可用短名，例如“黑齿市”“九州隙”。
4. 不在地图上放图例。图例会抢空间，状态应通过视觉自明。

### 7.4 动效

1. 雾散、城显、路显在小横屏缩短到 0.8-1.2 秒。
2. 不做大面积遮罩动画，避免挡住可点击路线。
3. 对 `prefers-reduced-motion` 和现有 `.disabled-motion` 保持尊重，动效可被关闭。

## 8. 本轮验收清单

C 线程或主线程收口时，可以按这些问题验收：

1. 九州图第一眼是否像羊皮纸山海舆图，而不是星图、网页地图或后台流程图？
2. 当前城市是否 1 秒内可定位？
3. 玩家是否只看到已知区域、当前可达路线和少量雾中暗示？
4. 远方是否没有被提前完整剧透？
5. 可走路线是否比已走、雾中、锁定路线更明确？
6. `.route-card` 是否仍能比较消耗、风险、半途、补给和锁定原因？
7. 到达新城市后是否有“雾散、城显、路显”的短反馈？
8. 小横屏是否仍能看清当前节点、可走节点和路线卡核心信息？
9. 是否避免复制 Out There 的星图、太空图标、科幻扫描和文案表达？

## 9. 需要用户决策的问题

本轮没有必须阻塞 C 线程的用户决策。建议默认按以下方向执行：

1. 九州图采用“羊皮纸山海舆图”大界面。
2. 迷雾揭示采用“当前 / 已见 / 可达 / 雾中 / 锁定 / 抵达中”状态。
3. 到达新城市后使用“雾散、城显、路显”三段短反馈。
4. 路线卡作为抽出的路引签，继续承担 Out There 式资源/风险决策。
5. 小横屏优先保证可读，少字、强状态、路线卡横向滚动。

后续可以让用户挑风格的非阻塞问题：

1. 路线卡更像“驿牒路引签”，还是更像“残卷批注”。
2. 当前节点的标记更偏“朱砂官印”，还是“车队小旗”。
3. 雾区更偏“纸雾水渍”，还是“墨晕污染”。
