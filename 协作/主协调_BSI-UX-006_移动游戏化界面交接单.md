## 一、用户新增意见

当前 Web Demo 的感觉太重，更像网页应用或桌面工作台，不像手机游戏。

这条意见优先级高。它影响的不是皮肤，而是玩家第一眼对产品类型的判断：

- 玩家应觉得自己打开的是一款竖屏手机游戏，而不是一个网页原型。
- 地图、地点、行囊、日志都应像游戏页面，而不是横向并排的信息面板。
- 当前已有机制可以保留，但信息呈现必须明显减重。

## 二、问题诊断

当前“网页感”主要来自：

- 宽屏桌面布局一次展示太多区块。
- 顶部资源、页面切换、横向舞台、事件、补给、行囊、日志同时出现，像调试台。
- 行囊、日志、音乐、动态层等低优先级信息占据常驻空间。
- 地图页和地点页都像信息页，缺少手机游戏常见的主画面、底部操作区、弹层/分页结构。
- 桌面端没有手机预览边界，导致界面自然铺满浏览器。

## 三、目标体验

第一版移动游戏化界面采用“竖屏手机壳”思路：

```text
顶部轻 HUD：日数 / 当前位置 / 车轴 / 粮草 / 神志 / 距九州裂隙
中部主画面：地点舞台或九州路线图
底部操作区：事件选择、补给、路线卡、返回地图等主要动作
底部导航：地点 / 九州图 / 行囊 / 日志
```

桌面浏览器可以作为手机预览器使用：主体最大宽度建议约 390-430px，居中显示；不要铺成桌面仪表盘。

## 四、默认产品决定

除非用户另行拍板，子线程按以下默认执行：

- 第一版以竖屏手机为主，桌面端只是居中预览，不做复杂桌面版。
- 保留当前静态 Web Demo 技术形态，不迁移 React/Vite，不引入重型 UI 库。
- 不改变核心玩法数据和路线逻辑，只调整页面结构、信息优先级、可触达性和视觉重量。
- 音乐、动态层、调试类状态收进设置或折叠区，不常驻抢占主界面。
- 行囊九宫格从右侧面板改为“行囊页”或底部弹层，不作为桌面侧栏。
- 日志默认只露最近一条，完整日志在“日志页”或抽屉里看。

## 六、C 线程任务

负责：在 D 线程方案验收后，修改当前 Web Demo 的静态原型。

负责文件：

- `GitHub资产区/03_WebDemo/prototype/index.html`
- `GitHub资产区/03_WebDemo/prototype/app.js`
- `GitHub资产区/03_WebDemo/prototype/styles.css`

原则：

- 以 CSS 和少量结构调整为主。
- 不删除已有机制。
- 不改路线、事件、补给、死局防护等玩法数据，除非是为了适配渲染字段。
- 不引入复杂依赖。
- 不覆盖其他线程未验收成果。

建议实现方向：

1. 建立 `.game-phone-shell` 或等价结构，限制主体宽度并模拟手机游戏画幅。
2. 将资源状态改成轻 HUD，保持车轴、粮草、神志始终可见。
3. 将主行动区放到底部，选择按钮必须适合拇指点击。
4. 将地点页与九州图页做成单页切换，不同时展示大量侧栏。
5. 将行囊九宫格移动到独立页、抽屉或底部弹层。
6. 将日志折叠，只保留最近关键反馈。
7. 在 390x844 和 430x932 两个竖屏尺寸下自检。

## 七、主协调验收标准

必须同时满足：

- 打开后第一眼像手机游戏，不像网页后台或桌面信息面板。
- 390px 宽度下无横向滚动，文本不互相遮挡。
- 玩家始终能看到三资源和离终点还有多远。
- 路线选择仍能看到目的地、消耗、风险、可观察补给。
- 地点页仍能完成：读地点志、补给、事件选择、回九州图。
- 半途路遇仍能正常中断、选择、抵达。
- 行囊九宫格可访问，但不挤压主事件区。
- 日志可访问，但不把主画面变成记录列表。
- `node --check app.js` 与 `node --check data.js` 通过。
