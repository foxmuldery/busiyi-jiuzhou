## 1. 本轮验收结论

当前 `data.js` 已有 8 个地点、11 条路线、3 类危机补救，能够作为第一章闭环的最低验收基线。C 线程若扩到 10-12 个地点，新增地点不应增加主线硬门槛，而应优先补两类空位：

1. 粮草缓冲点：中段或赤水前，提供 `粮草 +8 到 +14`，代价为 `车轴 -2 到 -5` 或 `神志 -2 到 -5`。
2. 神志缓冲点：赤水/断碑前后，提供 `神志 +5 到 +10`，代价为 `粮草 -3 到 -6`。

放给用户试玩前，最低标准是：

```text
至少 1 条主线能从中原驿走到九州裂隙；
任一非结局地点都有可走、可补救或可回撤出口；
三资源归零先触发濒死补救，不直接硬失败；
迷雾显影不会把玩家推入“所有路线都会立刻归零”的局面。
```

| 资源 | 上限 | 初始值 | 低值预警 | 危急线 | 归零处理 |
|---|---:|---:|---:|---:|---|

| 粮草 | 100 | 90 | 35 | 15 | 进入粮尽补救 |
| 神志 | 100 | 85 | 45 | 20 | 进入神志崩线补救 |

补救与保底阈值：

| 项 | 验收值 |
|---|---:|
| 同一资源允许濒死补救次数 | 2 |
| 全局允许濒死补救次数 | 4 |
| 濒死后保护步数 | 2 |
| `badLuckMeter` 强制补救阈值 | 75 |
| 连续坏事件警戒 | 2 |

### 3.2 推荐可通关路线

保守主线：

```text
中原驿 -> 故王道 -> 雷泽浅畔 -> 赤水外滩 -> 巫咸断碑 -> 九州裂隙
```

只计路线成本，总消耗为：

| 路线组 | 车轴 | 粮草 | 神志 | 说明 |
|---|---:|---:|---:|---|
| 保守主线，不计补给/路遇 | -38 | -32 | -29 | 初始资源可承受，抵达裂隙前不应濒死 |
| 保守主线，按每 1-2 城取一次补给 | 通常净损耗 <=25 | 通常净损耗 <=25 | 通常净损耗 <=25 | 应能压线但稳定抵达裂隙 |

支线验收路线：

```text
中原驿 -> 废关 -> 黑齿市 -> 巫咸断碑 -> 九州裂隙
```

只计路线成本为 `车轴 -24 / 粮草 -26 / 神志 -28`。这条路线更短，但神志压力集中在废关、黑齿市、断碑，验收重点是低神志时仍能看懂风险。

高压捷径：

```text
中原驿 -> 故王道 -> 雷泽浅畔 -> 赤水外滩 -> 九州裂隙
```

`red_to_rift` 必须满足 `followed_bones`、`rift_name` 或 `sanity <= 34` 之一。它是高压禁地路线，不应作为唯一出口；赤水必须同时保留 `red_to_stele` 作为清醒回撤解释线。

## 5. 迷雾显影保底的可检查条件

每次城市显影或路线刷新后，C 线程可以把下面条件做成验证函数，也可以由手动验收逐条检查。

| 检查项 | 可检查条件 | 不通过时的修正 |
|---|---|---|
| 引用完整 | `revealPlan.routes/fogRoutes` 中每个 id 都存在于 `routes`；`fogLocations` 每个 id 都存在于 `locations` | 修正 id 或移出显影计划 |
| 起点不泄全图 | 开局只完整显示 `central_post`、两条起点路线和少量雾中远影 | 隐藏非一跳地点和非当前可走路线 |
| 当前可推进 | 非结局地点刷新后，至少 1 条从当前地点出发的路线可见 | 强制显出低/中风险出口或回撤 |
| 当前可承受 | 至少 1 条可走路线不会让任一资源结算后 <= 0 | 降低一条路线消耗，或先触发危机补救 |
| 危急可补救 | 任一资源低于危急线时，下一次显影必须包含对应补给、危机事件或低成本出口 | 插入补救地点、补救事件或低消耗回撤 |
| 补给不重复计算 | `usedSupplies[id] = true` 的补给不能作为保底来源 | 改找未用补给或进入危机补救 |
| 高压不连锁 | 玩家刚走高风险路线后，下一地点不能只显示高风险离开路线 | 至少加入 1 条低/中风险出口 |
| 锁定可理解 | `red_to_rift` 在赤水外滩必须可灰显，且显示 `lockedHint` | 不可完全隐藏主线锁定 |
| 主线不单点随机 | 通向裂隙至少有断碑线、赤水事件线、低神志危险线中的 2 类可达解释 | 增加线索补发或替代解锁 |
| 低神志不误杀 | 神志 1-30 时可污染文案，但路线风险等级和主消耗方向仍可判断 | 保留风险等级、资源图标或主消耗文字 |
| 半途不偷跑 | `pendingRoute` 未处理完时，不打开目标地点补给和地点事件 | 等路遇完成后再抵达显影 |
| 终点不误判 | `kyushu_rift.routes` 为空时不算无路死局 | 进入结局事件或终点整理 |

显影保底的硬断言可以压成一句：

```text
每次刷新后，玩家必须至少拥有一种有效行动：可承受路线、可用补给、危机补救、结局选择或明确回撤。
```

## 7. 主线程试玩数值验收标准

### 7.1 必须全部通过

| 门槛 | 标准 |
|---|---|
| 语法 | `node --check app.js`、`node --check data.js` 通过 |
| 数据 | 原型数据验证通过；所有路线、显影、事件、补给引用有效 |
| 地点数 | 当前至少 8 个地点可玩；若宣称扩展第一章，则至少 10 个地点进入 `data.js` |
| 路线闭环 | 至少 1 条清醒主线能从中原驿到九州裂隙 |
| 锁定解释 | 赤水到裂隙的锁定路线可见、可理解、有替代路径 |
| 死局率 | 手动验收 0 死局；若有模拟器，软死局率必须为 0% |
| 不公平失败 | 手动验收不出现无选择硬死；若有模拟器，不公平失败率 <=1%，最高不得超过 3% |
| 危机补救 | 三资源归零都先触发对应补救；补救后至少能存活 2 步 |
| 低神志 | 神志低时不隐藏主要资源方向，不靠误导造成死亡 |
| 补给限制 | 已使用补给不刷新，不被保底重复计算 |

### 7.2 建议模拟指标

若 C 线程写了自动模拟或验证函数，主线程可按下面指标判断是否进入试玩。

| 指标 | 可试玩标准 |
|---|---:|
| 总通关率 | 25% 到 40% |
| 保守玩家通关率 | 45% 到 65% |
| 冒险玩家通关率 | 20% 到 40% |
| 随机玩家通关率 | 5% 到 18% |
| 软死局率 | 0% |
| 不公平失败率 | <=1%，最高 3% |
| 平均失败步数 | >=8 |
| 濒死后 3 步存活率 | >=65% |
| 单一资源失败占比 | 不超过 50% |

### 7.3 A 线程建议放行口径

可以放给用户试玩的口径：

```text
能放：手动 8 条最低保底测试通过，至少一条主线可抵达裂隙，赤水锁定可解释，三资源危机都先补救，无直接死局。
谨慎放：能走到裂隙但补给状态或 badLuck 保底还未自动化，需要主线程标注“第一章数值仍在调试”。
不能放：任一非结局地点无出口；所有可见路线都会立刻归零；赤水锁定没有替代路；资源归零直接硬失败；低神志让玩家看不出主风险。
```

最关键的试玩门槛不是通关率，而是公平性：

```text
玩家可以输，但不能因为迷雾刷新、路线锁定或补给重复计算错误而无选择地死。
```
