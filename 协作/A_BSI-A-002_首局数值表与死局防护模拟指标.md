# A_BSI-A-002 首局数值表与死局防护模拟指标

> 子线程：A，核心规则与数值平衡设计师  
> 任务：BSI-A-002，首局数值表与死局防护模拟指标  
> 范围：服务 Web Demo 第一局和 C 线程最小模拟器，不写代码，不写正式剧情文本

## 1. 首局目标

Web Demo 第一局用于验证“能玩、能输、能复盘、不会随机死局”。本表不是正式版平衡，而是 C 线程可以直接转成配置和模拟器断言的第一组数值。

| 目标 | 首局建议 |
|---|---:|
| 地点数量 | 8 到 12 个 |
| 可通关主路长度 | 7 到 9 次移动 |
| 完整探索长度 | 12 到 18 次移动 |
| 事件次数 | 12 到 24 次 |
| 首局预期通关率 | 25% 到 40% |
| 非自愿死局率 | 0% 为目标，最高不得超过 1% |
| 平均失败步数 | 第 8 步以后 |
| 平均通关步数 | 第 12 到 20 步 |

第一局必须保证至少一条“保底主路”理论可通关：玩家选择低/中风险混合路线，接受 1 到 2 次补给或休整，不主动连续选择高风险选项，就应有机会抵达一个基础结局。

## 2. 初始资源与阈值

### 2.1 初始资源

| 资源 | 初始值 | 上限 | 低值预警 | 危急线 | 濒死触发 |
|---|---:|---:|---:|---:|---:|
| 车轴 | 80 | 100 | 30 | 12 | 0 |
| 粮 | 90 | 100 | 35 | 15 | 0 |
| 神志 | 85 | 100 | 45 | 20 | 0 |

首局不建议低于这些初始值。初始资源太低会让玩家以为玩法是惩罚生存，而不是路线判断。

### 2.2 资源结算顺序

每一步按固定顺序结算，避免 C 线程实现时出现边界差异：

1. 生成或读取当前位置的可选路线。
2. 检查路线是否造成移动前死局。
3. 玩家或模拟器选择路线。
4. 扣除路线消耗。
5. 若资源归零，先进入濒死状态，不抽普通事件。
6. 若未归零，按路线等级和地点标签抽事件。
7. 执行事件结果。
8. 更新 `badLuckMeter`、连续坏事件、连续无正向收益步数。
9. 再次检查濒死、死局、结局。

## 3. 路线消耗表

### 3.1 首局固定路线档位

C 线程第一版可以先使用固定值，不必在每次移动里随机浮动。

| 路线等级 | 车轴 | 粮 | 神志 | 推荐占比 | 说明 |
|---|---:|---:|---:|---:|---|
| 低风险 | -4 | -4 | 0 | 40% 到 50% | 保证玩家有喘息和回撤路径 |
| 中风险 | -7 | -6 | -3 | 35% 到 45% | 首局主要压力来源 |
| 高风险 | -12 | -9 | -10 | 10% 到 20% | 高收益或捷径，不作为唯一主路 |

首局地图中，任意节点的可选路线建议 2 到 3 条：

| 节点类型 | 路线配置 |
|---|---|
| 起点后 1 到 2 步 | 至少 1 低 + 1 中 |
| 中段普通节点 | 至少 1 中，另配低或高 |
| 高风险奖励节点 | 至少 1 高 + 1 可回撤低/中 |
| 结局前节点 | 至少 1 中或低，不得只剩高风险 |
| 补给节点 | 至少 1 条离开路线车轴需求低于 8 |

### 3.2 路线消耗上限

首局每次移动后的路线扣除不得超过：

| 情况 | 单步净损耗上限 |
|---|---:|
| 玩家选择低风险 | 任一资源不超过 5 |
| 玩家选择中风险 | 任一资源不超过 9 |
| 玩家选择高风险 | 车轴不超过 15，粮不超过 11，神志不超过 16 |
| 非主动事件追加损耗 | 任一资源不超过 12 |
| 濒死补救后 2 步内 | 任一资源不超过 8 |

“主动”指玩家或模拟器选择了明确标记为高风险、高代价、强行、夜行、禁地、献祭一类选项。

## 4. 事件池与权重

### 4.1 事件池分类

| 事件池 | 类型 ID | 首局作用 | 典型净效果 |
|---|---|---|---|
| 风险事件 | `risk` | 扣资源并给选择压力 | 总净损耗 6 到 16 |
| 补给事件 | `supply` | 恢复粮或车轴 | 一项 +10 到 +24，另一项可小扣 |
| 线索事件 | `clue` | 给地图、语言、路线提示 | 小损耗或无损耗 |
| 器物事件 | `item` | 给一次性或被动修正 | 获得器物，或交换资源 |
| 神志事件 | `sanity` | 体现认知污染和隐藏收益 | 神志 -6 到 -16，可能给线索 |
| 喘息事件 | `rest` | 降压、休整、复盘 | 神志 +5 到 +15，或小补给 |

### 4.2 首局基础权重

| 路线等级 | risk | supply | clue | item | sanity | rest |
|---|---:|---:|---:|---:|---:|---:|
| 低风险 | 18 | 27 | 20 | 10 | 5 | 20 |
| 中风险 | 34 | 18 | 18 | 10 | 12 | 8 |
| 高风险 | 46 | 10 | 12 | 8 | 20 | 4 |

首局比 A-001 通用表略降低低风险路线的风险权重，目的是让玩家在前 5 分钟能理解系统，不被连续惩罚劝退。

### 4.3 事件结果范围

| 事件类型 | 车轴范围 | 粮范围 | 神志范围 | 正向收益标记 |
|---|---:|---:|---:|---|
| risk | -12 到 0 | -12 到 0 | -12 到 0 | 通常无，若给线索则有 |
| supply | -4 到 +12 | +10 到 +24 | -4 到 +4 | 有 |
| clue | -4 到 0 | -4 到 0 | -6 到 0 | 有 |
| item | -6 到 +8 | -6 到 +8 | -6 到 +4 | 有 |
| sanity | -2 到 0 | -4 到 0 | -16 到 -6 | 有隐藏路线/真名/线索时才算有 |
| rest | 0 到 +6 | -6 到 +8 | +5 到 +15 | 有 |

## 5. 补给、修复与神志风险

### 5.1 补给和修复最低频率

首局生成或手工配置地图时必须满足：

| 检查项 | 最低要求 |
|---|---|
| 每 4 次移动内 | 至少出现 1 个补给、修复、休整或交易机会 |
| 前 3 次移动内 | 至少出现 1 个低成本正向事件 |
| 进入结局前 2 个节点 | 至少出现 1 个补救或回撤机会 |
| 任一资源进入危急线后 | 下一事件或当前节点选项必须包含补救 |
| 高风险路线之后 | 下一个节点不得只接高风险路线 |

### 5.2 补给/修复事件建议数值

| 补救类型 | 触发条件 | 恢复 | 代价 |
|---|---|---:|---|
| 小采集 | 任意粮低于 50 的非神域节点 | 粮 +12 | 神志 -2 或车轴 -2 |
| 大采集 | 粮低于 25 | 粮 +22 | 车轴 -5，badLuckMeter -15 |
| 小修车 | 车轴低于 50 的安全或废关节点 | 车轴 +12 | 粮 -4 |
| 大修车 | 车轴低于 20 | 车轴 +24 | 粮 -8 或失去普通器物 |
| 记录休整 | 神志低于 45 | 神志 +10 | 粮 -5 |
| 强制喘息 | 任一资源危急且连续坏事件 >= 2 | 最低资源 +15 | 另一资源 -4 |

### 5.3 降神志风险控制

神志是特色资源，但首局不能让玩家被低神志信息不可靠直接坑死。

| 神志区间 | 路线提示 | 事件提示 | 规则限制 |
|---|---|---|---|
| 61 到 100 | 明确显示 | 明确显示主要代价 | 正常 |
| 31 到 60 | 副风险可模糊 | 可隐藏小代价 | 不隐藏主要资源方向 |
| 1 到 30 | 文案可污染 | 选项仍保留风险等级 | 不允许所有路线提示同时失真 |
| 0 | 进入崩解濒死 | 不抽普通事件 | 必须给濒死选择 |

首局禁止把“错误路线提示”作为主要失败原因。可用文本氛围表现污染，但系统层面必须保留基本可判断性。

## 6. badLuckMeter 与保底触发

### 6.1 首局参数

| 参数 | 值 |
|---|---:|
| 初始 badLuckMeter | 0 |
| 上限 | 100 |
| 连续坏事件警戒 | 2 |
| 连续无正向收益警戒 | 3 |
| 强制补救阈值 | 75 |
| 濒死后保护步数 | 2 |

### 6.2 增减规则

| 事件 | badLuckMeter 变化 |
|---|---:|
| risk 且总净损耗 6 到 11 | +8 |
| risk 且总净损耗 12 以上 | +15 |
| sanity 且神志损耗 10 以上 | +12 |
| 连续第 2 个坏事件 | +10 |
| 连续第 3 个坏事件 | +25 |
| 任一资源进入危急线 | +15 |
| 触发濒死 | +25 |
| 抽到 supply/rest | -20 |
| 获得 clue/item | -12 |
| 完成濒死补救 | -25 |

坏事件定义：事件结果没有 `positiveTag`，且任一资源净损耗大于 5，或总净损耗大于 8。

### 6.3 保底触发顺序

当多个保底同时满足时，按以下顺序处理：

1. 当前节点无可走路线：生成回撤或困境补救。
2. 任一资源归零：进入对应濒死状态。
3. 任一资源危急且下一步必归零：插入补救事件或低成本路线。
4. badLuckMeter >= 75：下一事件强制从 `supply/rest/clue` 中抽取。
5. 连续坏事件 >= 2：禁止再抽高损耗 `risk`。
6. 连续无正向收益 >= 3：下一事件必须带 `positiveTag`。
7. 高风险路线后：下一节点至少提供一条低/中风险离开路线。

## 7. 死局组合清单与生成规避

### 7.1 必须判定为死局或不公平的组合

| 组合 ID | 随机组合 | 为什么不公平 | 生成时规避 |
|---|---|---|---|
| A-LOCK-01 | 当前车轴低于所有出口路线需求，且无修车/回撤/徒步选项 | 玩家无法继续操作 | 至少生成 1 条车轴需求 <= 当前车轴的回撤路线，或插入断轴补救 |
| A-LOCK-02 | 粮低于下一次最低移动消耗，且当前节点没有补粮事件 | 必然饥荒且无选择 | 插入采集/交易/短休事件，或降低一条路线粮耗 |
| A-LOCK-03 | 神志低于 10，所有出口都扣神志，且无休整/记录选项 | 必然崩解且无选择 | 插入记录休整，或生成不扣神志的退路 |
| A-LOCK-04 | 连续 3 次 risk，且第三次仍为高损耗 | 坏运气压死玩家 | 第三次强制改为 supply/rest/clue |
| A-LOCK-05 | 高风险路线后只连接高风险路线 | 玩家被高风险链条锁死 | 高风险节点必须至少有一条低/中风险离开路线 |
| A-LOCK-06 | 关键线索只在单一随机事件中出现，错过后无替代 | 随机阻断主线 | 关键线索至少有 2 个来源，或可由地点抵达自动补发 |
| A-LOCK-07 | 濒死补救后立即抽到同资源高损耗事件 | 补救没有实际意义 | 濒死后 2 步保护，不抽同资源高损耗 |
| A-LOCK-08 | 低神志下所有路线提示都失真，且至少一条为致死路线 | 信息不足导致误杀 | 至少保留路线等级或主消耗方向 |
| A-LOCK-09 | 三资源同时低于预警线，事件池仍按高风险权重抽取 | 系统无视危机 | 强制进入喘息/交换/补救池 |
| A-LOCK-10 | 起点后前 3 步连续无正向收益 | 首局学习阶段过挫败 | 前 3 步至少 1 个 positiveTag |

### 7.2 生成前检查

地图或事件生成完成后，C 线程模拟前应先做静态检查：

| 检查 | 通过条件 |
|---|---|
| 主路可达 | 从起点到基础结局存在一条总车轴消耗 <= 65、总粮消耗 <= 75、总神志消耗 <= 45 的路线 |
| 补给间隔 | 任意连续 4 步主路内至少 1 个补给/休整/修复机会 |
| 高风险隔离 | 不存在连续 3 条强制高风险路线 |
| 出口检查 | 每个非结局节点至少 1 条离开路线 |
| 关键线索冗余 | 每个必需线索至少 2 个获取来源或 1 个保底补发条件 |
| 濒死出口 | 三种资源归零后都有至少 2 个可显示选项 |

### 7.3 运行中检查

每次玩家或模拟器行动前执行：

```text
if noAvailableRoutes:
  forceRescue("no_route")
if minResourceWillReachZeroOnAllRoutes:
  insertRescueForLowestResource()
if badLuckMeter >= 75:
  forcePositiveEvent()
if consecutiveBadEvents >= 2:
  blockHighLossRisk()
if deathSaveTriggeredRecently:
  applyTwoStepProtection()
```

这段是接口说明，不是要求 C 线程照抄代码。

## 8. 给 C 线程的最小模拟器接口建议

### 8.1 输入

模拟器第一版只需要接收 5 类输入。

| 输入 | 类型 | 说明 |
|---|---|---|
| `seed` | number/string | 随机种子，保证复现 |
| `runs` | number | 建议默认 1000 |
| `playerModels` | array | `conservative`、`explorer`、`greedy`、`returner`、`random` |
| `balanceConfig` | object | 初始资源、路线消耗、事件权重、badLuckMeter 参数 |
| `mapConfig` | object | 节点、路线、事件池、结局条件、关键线索 |

### 8.2 输出

每局输出：

| 字段 | 类型 | 说明 |
|---|---|---|
| `runId` | number | 局编号 |
| `seed` | string | 实际种子 |
| `playerModel` | string | 玩家模型 |
| `result` | string | `win`、`fail`、`special`、`return` |
| `steps` | number | 总步数 |
| `failureReason` | string | 失败主因 |
| `fairness` | string | `fair`、`questionable`、`unfair` |
| `softlockFlag` | boolean | 是否死局 |
| `resourcesFinal` | object | 结束三资源 |
| `resourcesMin` | object | 本局最低三资源 |
| `badLuckPeak` | number | badLuckMeter 峰值 |
| `deathSaveCount` | number | 濒死补救次数 |
| `forcedRescueCount` | number | 强制保底次数 |
| `eventCounts` | object | 各事件池次数 |
| `routeCounts` | object | 低/中/高路线次数 |

汇总输出：

| 字段 | 说明 |
|---|---|
| `winRateByModel` | 各玩家模型通关率 |
| `failReasonDistribution` | 失败原因占比 |
| `unfairRate` | 不公平失败占比 |
| `softlockRate` | 死局率 |
| `avgStepsByResult` | 成功/失败平均步数 |
| `deathSaveSurvivalRate` | 触发濒死后仍存活至少 3 步的比例 |
| `badLuckPeakP95` | 95 分位 badLuckMeter 峰值 |
| `resourceFailureDominance` | 单一资源失败是否过度集中 |

### 8.3 什么算不公平

模拟器应把以下情况标记为 `unfair`：

1. `softlockFlag = true`。
2. 玩家没有主动选高风险，却在连续 3 步内被同一资源扣到归零。
3. 濒死补救后 2 步内，因同一资源再次归零失败。
4. 失败前连续 3 个事件没有正向收益，且 badLuckMeter 未触发补救。
5. 所有可见路线都会导致资源归零，且没有补救或回撤。
6. 关键线索被随机错过后没有替代来源，导致主路不可达。
7. 低神志提示完全失真，模拟器记录不到可判断风险等级。

标记为 `questionable`：

1. 玩家主动选择高风险，但路线预览不够明确。
2. 玩家连续跳过补给后失败。
3. 玩家因贪心选择损失关键器物后失败。
4. 玩家进入神志特殊路线后失败。

标记为 `fair`：

1. 玩家多次主动选择高风险。
2. 玩家忽略明确补给或休整机会。
3. 玩家资源低时仍继续深入。
4. 失败前至少有 1 次明确补救机会。

## 9. 首局建议调参阈值

| 指标 | 目标 | 调整方向 |
|---|---:|---|
| 总通关率 | 25% 到 40% | 低于 25% 降低中/高风险消耗；高于 45% 提高高风险代价 |
| 保守玩家通关率 | 45% 到 65% | 低于 40% 增加补给密度 |
| 冒险玩家通关率 | 20% 到 40% | 高于 45% 削弱高风险收益 |
| 随机玩家通关率 | 5% 到 18% | 高于 20% 说明选择压力不足 |
| 不公平失败率 | 0% 到 3% | 高于 3% 必须先修保底 |
| 死局率 | 0% 到 1% | 高于 1% 禁止进入体验测试 |
| 断轴失败占比 | 20% 到 40% | 超过 50% 降低车轴消耗或增加修车 |
| 饥荒失败占比 | 20% 到 40% | 超过 50% 增加补粮或降低粮耗 |
| 崩解失败占比 | 15% 到 35% | 超过 45% 降低神志事件密度 |
| 平均失败步数 | >= 8 | 低于 8 说明前期太狠 |
| 濒死后 3 步存活率 | >= 65% | 低于 65% 说明补救是假补救 |

## 10. 待确认问题

1. 首局是否先以“基础结局可达”为目标，而不是要求多个结局都可达？建议默认：是。
2. 神志归零的“接受幻路”是否在 Web Demo 第一版就做成特殊路线？建议默认：保留接口，先做简化结果。
3. 首局是否允许“舍弃随从口粮/队伍稳定下降”这类抽象残酷选项？建议默认：可以抽象，不写细节。
4. C 线程模拟器是否先用固定事件类型和数值范围，不接正式文本？建议默认：是。
5. 死局率是否作为体验测试前的硬门槛？建议默认：是，超过 1% 不进入用户试玩。
